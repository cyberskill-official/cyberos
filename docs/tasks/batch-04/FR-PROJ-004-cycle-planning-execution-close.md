# Feature Request

> Turn Your Will Into Real.

## Summary

Ship the cycle-management surfaces: **cycle creation** with start/end dates and a **sprint goal**; **capacity planning** (per-Member estimate-points budget for the cycle, derived from prior cycles' completion + manual override); **cycle execution** dashboards (progress bar, burndown, blocker surface, at-risk indicators); **cycle close** with an authored **cycle review** (draft generated by CUO/COO from the cycle's data; human-edited and approved); and **carryover** (incomplete issues are bulk-moved to the next cycle or unscheduled with audit). The cycle is the team's heartbeat (Linear's "Cycle" is the same primitive); without proper close + carryover, velocity numbers are noise. PRD §17.5 demo S0-5 exit includes "sprint review opens with auto-prepared agenda from PROJ + BRAIN"; this FR delivers it.

## Problem

Today the team's cycle review is a 30-minute synchronous meeting where the founder pulls numbers from the project tracker by hand, summarises by memory, and writes a Slack post afterwards. Three fragility points:

- **Capacity is invisible.** "Did we plan to do 80 points and finish 50?" is answerable only post-hoc; in-cycle the team has no view of "are we on track".
- **Cycle close is manual.** Carrying forward incomplete work + writing the review takes 45 minutes; if it's skipped, the next cycle starts blind.
- **The review's content is built by hand.** The founder's narrative ("we shipped Acme's onboarding flow but Beta's auth integration slipped because of the SSO regression") is reconstructed each Friday from memory + chat scrollback. CUO can produce the draft in 90 seconds with citations; the founder edits and ships.

The PRD §9.5.3 names this explicitly: "Status-update generation. End of cycle: CUO drafts the cycle review for the Account Manager to edit and send."

## Proposed Solution

The shape of the answer is a `cyberos-proj-cycle` service that owns the planning + close lifecycle, the capacity model, the cycle dashboard surfaces in the frontend (the actual rendering ships in FR-PROJ-005), the CUO-driven review-draft pipeline (the persona work lives in FR-PROJ-006; this FR specifies the surface), and the carryover semantics integrated with the sync engine (FR-PROJ-002).

**Cycle creation.**

```graphql
mutation projCreateCycle(
  projectId: ID!
  number: Int                       # auto-incremented if omitted
  startsAt: DateTime!
  endsAt: DateTime!
  goalMd: String                    # optional one-paragraph goal
)
```

Constraints enforced server-side:
- `endsAt > startsAt`.
- Two-cycle overlap allowed (a cycle can be planned while the prior is still active) but no triple overlap per project.
- Cycle length defaults to project's `default_cycle_length_days` (FR-PROJ-001) — typically 14 for CyberSkill's two-week sprints.

The cycle goal is plain Markdown ≤ 800 chars; surfaced in the cycle header, the daily-flow card (FR-GENIE-003), and the cycle-close review.

**Capacity planning.**

```sql
CREATE TABLE proj.cycle_capacity (
  tenant_id UUID NOT NULL,
  cycle_id UUID NOT NULL REFERENCES proj.cycle(id) ON DELETE CASCADE,
  member_id UUID NOT NULL,
  capacity_points INT NOT NULL,                 -- the estimate-points budget for this Member in this cycle
  derived_from_avg_last_n_cycles REAL,          -- the rolling average that fed the suggestion
  manual_override_at TIMESTAMPTZ,
  manual_override_by UUID,
  PRIMARY KEY (tenant_id, cycle_id, member_id)
);
```

Capacity suggestion algorithm:
1. Compute completion-points-per-Member-per-cycle for the last 6 cycles in the project.
2. Apply a 0.85 confidence factor (Linear's heuristic; we adopt with telemetry).
3. Subtract known PTO (read from FR-TIME's leave records when TIME ships in batch-05; stubbed today).
4. Surface as a chip in the cycle-planning view; user accepts or adjusts.

Total cycle capacity is the sum across active Members; the cycle header shows `committed: 120 / capacity: 154` as a calibration signal.

**Sprint-goal authoring.**

When the user opens the cycle-planning view, CUO/COO pre-drafts a goal from:
- The Engagement's contract scope (FR-PROJ-007).
- The current state of the previous cycle's open work.
- BRAIN-derived context ("Acme has the Q3 launch in 6 weeks").

The draft is suggested in a card; the user accepts, edits, or writes from scratch. The goal is BRAIN-ingested as a fact (`subject: cycle:<id>, predicate: has_goal, object: <text>`).

**Cycle execution dashboards.**

The cycle dashboard (in FR-PROJ-005) shows:

- **Progress bar.** Completion-points / committed-points; colour shifts amber after day 60% with > 50% incomplete; red after day 80% with > 30% incomplete.
- **Burndown chart.** Daily snapshot of remaining points; the ideal-burndown line is the diagonal from full-commit on day 1 to zero on `endsAt`.
- **At-risk surface.** Issues that are: in `blocked` state; not started after day 50%; assigned to a Member at WIP limit; missing a non-trivial estimate. Each at-risk issue shows the reason + a CUO-suggested next step.
- **Per-Member capacity bar.** Each Member's committed vs. completed; over-committed Members surface red.
- **Blocker graph.** A small graph of `blocked_by_issue_ids` relationships; a blocking issue with high downstream count surfaces with high priority.

Snapshots for the burndown live in `proj.cycle_daily_snapshot{cycle_id, date, committed_points, remaining_points, completed_points, blocked_points}`; written by a daily 00:00 ICT job.

**Cycle close.**

The close mutation:

```graphql
mutation projCloseCycle(
  id: ID!
  reviewMd: String!                # the human-approved review
  carryoverIssueIds: [ID!]!        # issues to move to the next cycle
  cancelIssueIds: [ID!] = []       # issues to cancel
  unscheduleIssueIds: [ID!] = []   # issues to remove from cycle (parking)
)
```

Server-side:
1. Validate every `carryoverIssueIds` issue is in this cycle and not in `done`/`cancelled`/`duplicate`.
2. Move carryovers to the *next* cycle (must exist; created if not — auto-creates with default length).
3. Cancel the explicit cancels; unschedule the explicit unschedules.
4. Set the cycle's `status: completed`, `completed_at: now()`, `cycle_review_md: <text>`.
5. Compute the cycle's final stats (committed / completed / carried / cancelled / velocity-points-per-Member) and store in `proj.cycle_close_summary`.
6. Audit row with the close payload.

A cycle cannot close while issues remain in `in_progress` or `in_review` without an explicit decision (carry / cancel / unschedule) — the UI walks the user through every such issue.

**Cycle review draft pipeline.**

The cycle-close UI shows a "Draft review" button. Clicking it triggers the CUO/COO `cyberos.proj.draft_cycle_review(cycle_id)` MCP tool (full implementation in FR-PROJ-006), which returns:
- A 6-10 sentence narrative covering: cycle goal recap; what shipped; what slipped + why; a calibration note (committed vs. delivered); next-cycle recommendations.
- Citations: cycle-data + key issues + key BRAIN facts.
- The persona-version stamp.

The draft populates the `reviewMd` field; the user edits + closes the cycle. Acceptance metrics flow into the persona-quality dashboard (FR-GENIE-002).

**Cycle review consumption.**

The cycle review is:
- Saved on the cycle record (`proj.cycle.cycle_review_md`).
- Ingested into BRAIN as a fact (`subject: cycle:<id>, predicate: has_review, object: <summary>`) so future "what did we ship in 2026-Q3?" queries can cite cycle reviews.
- Surfaced as a Notify card to the Engagement's external contacts (CRM-linked, FR-PROJ-007 + batch-05) — *if* the Engagement's `client_visibility_default` is `client_visible_summary` and the Member confirms the publish.
- Posted to the Engagement's CHAT channel (when CHAT is in use) by default — Notify-confirm before post.

**MCP tool surface (extends FR-PROJ-001's read set).** Mutations in FR-PROJ-008; this FR adds:

- `cyberos.proj.list_capacity(cycle_id)` (read).
- `cyberos.proj.get_cycle_progress(cycle_id)` (read; returns the dashboard payload).
- `cyberos.proj.list_at_risk_issues(cycle_id)` (read).
- `cyberos.proj.list_carryover_candidates(cycle_id)` (read; returns issues that would carry by default).

## Alternatives Considered

- **Skip capacity planning; the team self-regulates.** Rejected: PRD §9.5.3 explicitly names "estimate calibration" as a CUO/CPO-skill feature; capacity is the substrate it runs on.
- **Auto-close cycles at `endsAt` with no review.** Rejected: the review is the most-valuable artefact of the cycle; auto-close without one defeats the point.
- **Allow cycle close with `in_progress` issues left unhandled.** Rejected: leads to silent carryover where work is forgotten.
- **Use Linear's velocity formula directly.** We adopt the broad pattern (rolling average + confidence factor) but our N is 6 cycles + 0.85 factor; tunable per project.
- **Skip burndown chart in P1; ship velocity only.** Rejected: burndown is the daily-driver dashboard; velocity is the cycle-close summary; both are needed.

## Success Metrics

- **Primary metric.** P1 sprint demo passes: (1) the founder creates a cycle with start/end + goal; (2) capacity suggestion populates with the 6-cycle rolling average; (3) burndown chart updates daily; (4) at-risk surface flags a synthetic blocked-issue; (5) cycle close runs the review-draft pipeline; the founder edits + commits; carryover bulk-moves 4 issues to the next cycle.
- **Founder-cognitive-load metric.** Cycle-review prep time ≤ 5 minutes (down from ~45 minutes baseline) for at least 4 consecutive cycle closes.
- **Latency NFR.** Cycle dashboard p95 ≤ 600 ms over a 200-issue cycle.

## AI Risk Assessment

Cycle-goal drafting + review drafting + at-risk explanations are AI surfaces. EU AI Act risk class: `limited`.

### Data Sources

Per-tenant data only: cycle data, issue data, BRAIN context. No third-party data. The CUO draft runs through the AI Gateway with persona-stamping.

### Human Oversight

- Goal draft: user accepts / edits / writes from scratch.
- Review draft: user edits and explicitly approves before close.
- At-risk surface: informational; the user decides what to do with each flagged issue.
- Carryover: every issue the user explicitly handles; no silent carryover.

### Failure Modes

- **Capacity suggestion wildly wrong.** Mitigation: manual override; the suggestion shows its sources; cycle-close calibration metric flags drift.
- **Review hallucinates a fact.** Mitigation: citation correctness via the persona regression suite (FR-GENIE-002); the founder edits before commit.
- **Cycle close blocked by an undecided issue.** Mitigation: the UI walks every undecided issue; bulk decisions supported.
- **Snapshot job missed.** Mitigation: hourly self-check; backfill missed days from the canonical mutation log.
