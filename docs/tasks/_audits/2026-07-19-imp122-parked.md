# IMP-122 (rules_sha recompute) parked - 2026-07-19

Decision: TASK-IMP-122 is parked (status on_hold) as an enhancement, consistent with the
2026-07-19 decision to accept finding #9 (rules_sha stored-not-recomputed) as by-design.
Recorded here with the verification that settled it.

## What was verified at HEAD (fresh same-version install into scratch)

- The current drift check is stored-to-stored. A payload built at HEAD and installed fresh
  into a scratch repo produces `verdict=up_to_date`: payload stored `rules_sha` ==
  install stored `rules_sha` (`fb7682e7...` both). The comparison does NOT false-mismatch
  for a correct install. The current mechanism works for its stated purpose - detecting
  that two channels were built from different rule content - and its only real limit is
  that, being stored-to-stored, it cannot detect an in-place edit after install.
- IMP-122's cone evidence is nonetheless correct. A RECOMPUTE of the current cone
  (`cuo plugin mcp cli memory`) over each side's live tree false-mismatches even on that
  fresh matched install (payload `fb7682e7...` vs install `f8f65a1c...`), because `cli/` is
  payload-only and `memory/store` is install-only, both inside the cone. Any future move to
  recompute MUST fix the cone first, exactly as IMP-122 argues.

## Why parked rather than pursued or dropped

Pursued: IMP-122 changes the design from stored-to-stored to recompute, to gain
tamper-detection. That is an enhancement, not a fix for a broken current behavior, and #9
was accepted by-design (not adding tamper-detection now). So it is not pursued at this time.

Dropped: not dropped, because its central technical finding - the cone is wrong for a
recompute (cli payload-only, memory/store install-only) - is real, verified, and would be
needed by anyone who later wants recompute-based tamper-detection. Dropping it would throw
that away. It is also a heavy task (20h, p1, 5 rewrites, last audited 8/10 FAIL with a
recorded stop-signal), so re-authoring it should be a deliberate decision, not incidental.

Parked (on_hold): the task record and its analysis are preserved; it re-enters the queue
only if and when the operator decides tamper-detection is wanted. The escape hatch is the
same one named in the #9 note: recompute-at-verify over a corrected, symmetric cone.

No code change was made. This is a recorded decision.
