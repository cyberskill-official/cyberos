# TASK-IMP-124: authors do not check what they originate

## Summary

Seven audits across two tasks and three authors found one recurring defect: authors verify what they INHERIT and ship falsehoods they ORIGINATE. Round 4 of TASK-IMP-122 re-measured every inherited claim correctly and then asserted `1525` as the cone's file count in the guardrail AC, where the true count is 1534 and 1525 is the count of the cone the spec explicitly rejects. TASK-IMP-121's rewrite corrected all six inherited mis-citations and then shipped an impossibility proof that is false in four lines of awk. Add TRACE-007, a judgment-family rule requiring every ORIGINATED claim to carry a re-runnable derivation, and restructure the AI Authorship Disclosure so it is a claim the audit tests rather than an attestation it credits.

## Problem

An inherited claim carries a provenance. It came from somewhere, that somewhere might be wrong, so the author looks. An originated claim arrives already believed: the author IS the provenance, so nothing prompts a check, and nothing in the workflow flags it as needing one. The gates are all downstream of that. The result, measured across seven independent audits:

**1. A false number, asserted normatively, under a sentence certifying it was not.** TASK-IMP-122 rewrite 4 (`63705483`) closed both CRITICALs, broke a four-round flat line at 8/10, and had every inherited citation re-measured and correct. Its AC 10 (`:280`) asserts `102dc507` "over 1525 files per side". Derived at HEAD `15894b1e` over `dist/cyberos`:

| cone | command (files, then `build.sh:353`'s `_rsha()` pipeline) | count | digest |
|---|---|---|---|
| §1.4 REJECTS | `find cuo plugin mcp lib docs-tools -type f` | **1525** | `86cafee8` |
| §1.4 MANDATES | the same, plus `memory` minus `memory/store/`, plus the six root scripts | **1534** | `102dc507` |

1525 + 3 + 6 = 1534. The spec pairs the mandated cone's digest with the rejected cone's count, in the guardrail AC, and a test written faithfully to that AC asserts 1525 and **FAILS on a correct implementation**. Thirty-eight lines above it, the AI Authorship Disclosure (`:242`) reads: "Every numeric and line-number claim was re-measured against source at HEAD by this author during this rewrite." The claim was true of everything round 4 inherited and false of the one number it invented. **The one clause meant to prevent the failure certified it instead** - and it could, because it is self-certifying: the author attests to their own diligence and nothing tests the attestation.

**2. A false impossibility proof, load-bearing, disproved by a counter-example nobody was asked for.** TASK-IMP-121's rewrite (`7309cd80`) dissolved four CRITICALs, fixed all six inherited mis-citations, and reproduced every inherited number exactly. Then, at `15894b1e:.../TASK-IMP-121/spec.md:168`: an operator hook with no trailing newline "cannot be restored byte-exact ... The information is destroyed at append time, so no uninstall-side rule can invert it." Measured by this author, replaying `install.sh:851-853` verbatim against a 6-byte `exit 0`:

```
line-oriented strip (the spec's rule):   6B -> 7B   DIFFERS
byte-oriented rule (4 lines of awk):     6B -> 6B   BYTE-EXACT
same rule, newline-terminated control:   7B -> 7B   BYTE-EXACT
same rule, three-newline control:        9B -> 9B   BYTE-EXACT
```

The information is not destroyed. `install.sh:851-853` always prepends exactly one `\n` before the marker, so the byte preceding the marker is always ours, and the operator's terminator survives as the presence or absence of a blank line above it. The error is a **level confusion**: the author reasoned in LINES, found no blank line to consume in the no-newline case, and generalised "my line-oriented rule cannot" to "no rule can". Their own evidence - "16 -> 17 bytes with the candidate strip" (`15894b1e`, `:49` and `:168`) - supports the narrow claim and was published as the universal one. That false proof narrowed §1.6, which excuses a live §1.8 violation and scopes AC 6/7 so the suite can never catch it.

**And it is why §1.2 requires a citation to name a revision.** While this task was being drafted, TASK-IMP-121's spec was rewritten in the working tree. A bare `spec.md:168` now returns an unrelated AC; the impossibility has moved to `:178`, where that rewrite RETRACTS it - independently reaching the same four measurements printed above. The anti-example is unaffected, because it is pinned to the commit where the claim shipped. A citation into a live document is a claim with a half-life, and this one expired inside a single session.

**3. A dead path in the author's own frontmatter, that no gate reads.** Found by this author while sizing this task. `TASK-IMP-118/spec.md:22` FORMERLY declared, in `modified_files`, a path that never existed: `git show d4b1419e:docs/tasks/improvement/TASK-IMP-118-tests-must-test-their-clause/spec.md | sed -n '22p'` returns 'tools/install/docs-tools/templates/task-audit/RUBRIC.md', while `ls tools/install/docs-tools/templates/` returns "No such file or directory" and the five real copies live under `modules/skill/` and the build outputs. **The earlier instance, and the shape of the cure.** `docs/tasks/_audits/2026-07-18-phase-corpus-measurement.md` exists because three specs shipped four false counts from greps of one subdirectory reported as corpus-wide facts. That file did one thing differently: `:8` carries "Reproduce: `node docs/tasks/_audits/measure-phase.mjs "$(pwd)"`". Re-running it at HEAD prints **550** where the file's table (`:14`) says **551** - a real drift, found in one command, explained in one more (`TASK-IMP-123`'s spec was deleted at `f8899d64`). The number that carried its derivation was checkable by a stranger in seconds. `1525` carried none and took an independent audit.

**Why every gate held.** task-lint is clean on all of it - four rounds running (`TASK-IMP-122/audit.md:12`). FM/SEC/COND/TRACE are structural and have no view into whether a number is true. COND-004 (`RUBRIC.md:62`) requires the disclosure to EXIST with three labelled bullets; it constrains nothing the Scope bullet may assert, so "every claim was re-measured" satisfies it exactly as well as the truth would. The model audit reads a document and can only re-derive what it can see a derivation for. And the human gates were shown a green suite and a truthful-looking summary. No check failed. No check was pointed at this.

## Proposed Solution

Two halves of one mechanism, both in `audit_rubric@2.0`.

**(a) TRACE-007 - every originated claim carries its derivation.** Judgment family, beside TRACE-006. For every claim the spec ORIGINATES in a governed class, the audit states the claim, states its derivation, re-runs it, and FAILS when the derivation is absent, does not re-run, does not reproduce the claimed value, or **supports a narrower scope than the claim asserts**. Three classes, three derivation forms: a NUMBER needs a command, a script invocation, or an arithmetic composition of other derived values; a CITATION needs `file:line` plus the text found there; a UNIVERSAL NEGATIVE needs a record of the witness search, and the audit must attempt one counter-example before accepting it. The point is not to make the author trustworthy. It is to stop the reader having to trust: a claim that carries the command that produced it can be re-run by a stranger, and the phase-corpus file proves a stranger will.

**(b) The disclosure becomes a claim, not a certificate.** COND-004 gains a shape: the `Scope:` bullet MUST partition its verification claims into **re-derived and CONFIRMED**, **re-derived and CORRECTED**, and **measured and ADDED**, MUST state whether the audit's own figures were treated as claims to verify, and MUST NOT make an unscoped universal attestation. "Every claim was re-measured" is unfalsifiable, which is why it survived being false. The partition is falsifiable: a number in the CONFIRMED set that no derivation reproduces is a finding, and a governed originated claim in none of the three sets is a finding. The audit tests the partition against the document and never credits it.

This is not invented here. TASK-IMP-122 rewrite 5 (`spec.md:279-304`) wrote exactly this disclosure voluntarily, one round after the failure, and it is the working prototype: three labelled partitions, the corrections named (`1525 -> 1534`), and an explicit statement that the round-4 audit's own figures "were treated as claims to verify, not facts to copy".

**Kinship, stated in the rubric.** TRACE-006 fails a test that asserts less than its clause's VERB. TRACE-007 fails a derivation that supports less than its claim's SCOPE. They are one defect at two levels - the author is the wrong reader of their own work - and 121's §3 is the crossover case: a measurement of the author's own candidate rule, standing in for a claim about all rules, is a weaker assertion wearing a stronger clause's words.

## Alternatives Considered

- **(c) The author of a rewrite must not be the author of the draft.** REJECTED, and rejected by the evidence offered for it. The score history is real - TASK-IMP-122 went 4/6/6/6 under one author and 8 on the first round with a fresh one (`audit.md:6`; `7309cd80`) - and author rotation IS already the working method for a different defect (patching only what the audit names). It does not touch this one. Both fresh-author rounds shipped an originated falsehood: round 4 the false 1525, the 121 rewrite the false impossibility. Rotation changes who INHERITS; it does not make anyone check what they originate. Worse, it is the mechanism by which this failure became visible: an author who re-measures everything inherited is left with exactly one unchecked category, and it is theirs. Rotation without TRACE-007 converts inherited-claim failures into originated-claim failures and calls it progress. Kept as practice, refused as this task's answer.
- **Make the audit re-derive every number.** Rejected: the audit cannot re-derive what it cannot see a derivation for. An auditor facing `1525` with no command must either invent the measurement (which is what caught it - once, expensively, by a subagent who happened to build all four cones) or take it. TRACE-007 asks the author to pay a cost they have already paid: the command is in their scrollback at origination time and gone by review time. §1.9 puts the instruction at that moment for exactly this reason.
- **Put derivation-checking in task-lint.** Rejected, and the reasons are two. Executing a derivation out of a spec body runs arbitrary shell from a document the rubric itself treats as capable of carrying injected content (SAFE-001..004 exist because spec bodies quote untrusted material). And a lint that appeared to verify derivations would restore the false assurance this task removes - the TASK-IMP-118 §1.5 argument, unchanged.
- **Follow TASK-IMP-118 §1.5 exactly and add NOTHING mechanical.** Rejected on a measured difference, not a preference. 118's argument is that a structural check shaped like TRACE-006 would PASS 108 §1.7's original assertion, because the string genuinely is in the file. Apply 118's own test here: a check that the disclosure's Scope bullet carries the three partition labels FAILS rewrite 4's disclosure (it carries none) and PASSES rewrite 5's (it carries all three). It discriminates the motivating case. So the shape enters the machine floor and nothing else does - §1.8 bounds it explicitly, and its finding message must name itself as shape-only so a green lint is never read as a verified disclosure.
- **Require a derivation for EVERY claim, inherited included.** Rejected: it prices the check at the volume that guarantees it is skipped, and it targets the category authors already check. Inherited claims are covered by TRACE-007 through the disclosure's own partition (an inherited number the author reproduced sits in CONFIRMED and must survive re-derivation), which is where the evidence says the cost belongs.
- **Extend TRACE-003 to `modified_files` and every other asserted path.** Out of scope, named in §Scope. It is real (Problem 3), it is structural rather than judgment, and it is a different rule with a different family. Filing it as its own task is TASK-IMP-117's precedent, not this task's job.
- **Effort 13 hours.** Breakdown in Scope; it sums to the field.

## 3. Edge cases

- A claim with NO derivation possible: TRACE-007 does not fire on a claim marked as a gap. TASK-IMP-121's `15894b1e:spec.md:168` is the anti-example precisely because it was NOT marked as a gap - it was asserted as a proof. "I could not measure this" is a derivation of a different kind and discharges the rule; "no rule can" is a universal negative and does not.
- A citation into a document that moves BETWEEN authoring and audit: this happened to this spec, mid-draft (Problem 2). `TASK-IMP-121/spec.md:168` was accurate when written and returned an unrelated AC before the draft was finished, because a rewrite retracted the claim. NOT a TRACE-007 failure against the author - it is why §1.2's CITATION class requires a revision for a document in flight. A `file:line` into a live document is a claim with a half-life; a `sha:file:line` is not.
- A number ORIGINATED by the audit, not the spec: covered, and this is the sharpest row. The round-4 audit supplied `1534` and rewrite 5 re-derived it rather than copying it (`spec.md:292`). An audit's figure is an inherited claim to the author and an originated one to the auditor. TRACE-007 governs the spec; §1.6 forces the author to say which side they treated it as.
- A derivation that re-runs and reproduces a DIFFERENT value than when written: this is the phase-corpus case (551 -> 550, `f8899d64` deleted a spec). NOT a TRACE-007 failure - the derivation did its job by surfacing the drift. It is a STALE-001 disposition, and TRACE-007's text must route it there rather than fail an honest number that the tree moved under.
- A derivation that re-runs on the author's machine and not the auditor's: fails TRACE-007 ("does not re-run"). This is intended and is the same standard TRACE-003 sets for test paths. A command only its author can run is prose.
- Two authors, one claim - author B reproduces author A's number and asserts it: it is INHERITED for B and B's disclosure must place it in `CONFIRMED`. If B did not re-derive it, it belongs in no partition and B must say so. Round 4 got this right on everything it inherited; the rule exists for the other column.
- A clause with two verbs (§1.8: MUST check only shape, MUST NOT execute): both are compared under TRACE-006 and AC 8 asserts both halves separately. This is TASK-IMP-118 §3 row 5, and this task must not fail it.
- SECURITY: a spec body carrying a derivation crafted to be executed. §1.8 forbids any execution from a body in the machine floor, and the auditor re-runs derivations under the same human-gated conditions as any other command they run. SAFE-001..004 already treat spec bodies as able to carry injected content; TRACE-007 must not become the reason a tool starts trusting them.
- SECURITY: a derivation that reproduces the value and lies about what it measures (`echo 1525`). TRACE-007 fails it under "supports a narrower scope than the claim asserts" - `echo` derives a literal, never a count. The rule is not "a command is present"; that reading is exactly the shape-check trap §1.8 bounds, and it is why §1.1 is judgment-family.
- DEGRADATION: TRACE-007 applied to a task audited before it existed - reported, never auto-failed. 183 `done` specs (derived: `measure-phase.mjs`, `done_total = 183`) could not have carried derivations. TASK-IMP-118 §3 row 11's disposition, unchanged.
- DEGRADATION: the auditor cannot run a derivation (no network, missing fixture) - reported as unverified, never credited as confirmed. Unreadable is not satisfied; TASK-IMP-118 §3 row 10's disposition.
- NULL/EMPTY: a spec that originates no governed claim (a pure policy change) - TRACE-007 is silent and COND-004's partitions may be empty, but the labels must still be present and each empty set must say so. An absent partition and an empty one are different claims.
- BOUNDS: a derivation STRONGER than the claim (a full corpus scan behind a single count) - passes. Stronger is never a finding, per TASK-IMP-118 §3 row 3.
- The disproof witness in `source_pages` was measured under GNU Awk 5.1.0 and its portability is untested. Immaterial: disproving "no rule can invert it" needs one witness, not a portable one. Named here so it is not mistaken for a recommendation to TASK-IMP-121, whose fix is its own to choose.
- `15894b1e` records "three authors" and this author cannot verify author identity from git - every commit in this range carries one committer. The audit COUNT (seven) is derived and holds; the author count is inherited and is used nowhere normative.
