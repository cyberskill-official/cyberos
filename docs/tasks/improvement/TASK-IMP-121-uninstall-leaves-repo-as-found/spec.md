# TASK-IMP-121: Uninstall must leave the repo as it found it

## Summary

Uninstall deletes the machine and leaves things pointing at it. A byte-level harness on a fresh git repo found five dangling skill symlinks and two MCP registrations naming a server that is gone. Two further defects are byte-level: stripping our managed block from an operator's pre-commit hook leaves one newline behind every cycle, and doing the same to a pre-existing `.gitignore` leaves one behind once. Nothing here destroys operator content - the harness recorded zero casualties across its six cases - but the repo is left broken rather than merely untidy, and two files an operator owns do not come back byte-identical.

## Problem

Measured 2026-07-18. `uninstall.sh:151` runs `rm -rf "$CY"`; `:155-158` then recreates `.cyberos/memory/` and moves the stashed BRAIN back. **`.cyberos/` survives by default** - what is gone in both modes is `plugin/`, `mcp/`, `cuo/`, `lib/`, `docs-tools/` and the root scripts. Every pointer install writes targets `.cyberos/plugin/skills/...` or `.cyberos/mcp/cyberos-mcp.mjs`, and both live in the part that is removed under `CYBEROS_UNINSTALL_KEEP_BRAIN=1` and `=0` alike. So the question is never "does `.cyberos/` still exist" - it is "does this pointer name a path we deleted", and that is answered by the pointer's own text.

**1. Five dangling skill symlinks.** Install writes six channels: `.agents/skills/`, `.claude/skills/`, `.grok/skills/`, `.commandcode/skills/`, `.codex/skills/`, `.opencode/skill/` (`:623-627`, `:631-632`, `:650-679`; note OpenCode's singular `skill`). Uninstall's loop (`:94`) looks at two. `.agents/skills/ship-tasks` is removed by `:98-99`; `.claude/skills/{task-author,task-audit}` by `:113-114`. `.claude/skills/ship-tasks` is exempted by name at `:112` (`[ "$_sc" != "ship-tasks" ]`), and `.grok`, `.commandcode`, `.codex`, `.opencode` are never looked at - `grep -ci` returns **0** for each of them in `uninstall.sh`. Five survivors, all `-> ../../.cyberos/plugin/skills/ship-tasks`, all confirmed dangling.

**2. Two dead MCP registrations.** `.mcp.json` and `.cursor/mcp.json` both register `node .cyberos/mcp/cyberos-mcp.mjs` (`:685-688`). Uninstall never unregisters - `grep -ci mcp` returns 0. Every agent reading those configs launches a missing server on every startup, in a repo the operator just cleaned.

**3. The hook gains a newline every cycle.** Install appends its block with `cat >> "$hk" <<'HOOK'` whose heredoc's **first line is blank** (`:851-853`), so the file gains `\n# >>> cyberos-status-hook v2`. `uninstall.sh:68`'s sed range starts **at** the marker, so the blank above it survives. Proven by the harness at 29 -> 30 -> 31 -> 32 bytes over three cycles, final tail `exit 0\n\n\n\n`; reproduced this round at 7 -> 8 -> 9 -> 10 on a 7-byte fixture. Unbounded: nothing ever trims it.

**4. The same defect in `.gitignore`, which the first draft missed.** `install.sh:747` writes `[ -s "$gi.cyberos.tmp" ] && printf '\n'` before the block, and `uninstall.sh:80`'s sed range also starts at the marker. Measured by replaying both verbatim: a pre-existing `.gitignore` goes **14 -> 15 bytes** and stays at 15 across three cycles - `install.sh:745`'s awk trims the trailing blank run on re-install, so this leak is one-time rather than accumulating. It is still a mutation of an operator's file, and it is why the fix belongs to the strip, not to the hook.

The `[ -s ]` guard at `:747` also explains the separate `.gitignore` symptom the harness saw: where install **created** the file (`:706`), the tmp is empty, no separator is written, and the strip leaves a 0-byte file.

**Against section 6.** `uninstall.sh:162` reads `# 6. skill symlinks into .cyberos (dangling) - leave dirs; operator cleans`. The dangling is **known and deliberate**, and `:111`'s comment records the same posture for `.claude/skills/ship-tasks`. This is a decision to overturn, not a gap to fill. It is overturned because its premise does not survive contact with `:113-114`. The restraint section 6 encodes is the right instinct about *operator files* - and a symlink whose target text names our own machine is not an operator file. `:113-114` already trusts exactly that proof to remove `task-author` and `task-audit`, in the same run that leaves `ship-tasks` dangling. Section 6 is not restraint applied consistently; it is one channel's entry escaping a test its two siblings pass. Leaving a pointer to a file we deleted protects nobody.

**What ownership can and cannot be proven from.** The discipline `:98-99` and `:113-114` use is: read the artefact, decide from what it says. It generalises to every artefact here - a marker-delimited block, an MCP command naming our path, a `.cyberos-owned` file - and it stops exactly at the container. Whether install *created* `.gitignore` or `.mcp.json` is not recorded anywhere: `install.sh`'s `wrote`, `AGENT_FILES` and `SKILL_DIRS` (`:479`, `:580`, `:621`, `:686`) are shell variables that die with the process, and the only on-disk marker, `.cyberos-owned`, is written solely into `.agents/skills` copy-fallback dirs (`:672-675`). An operator who ran `touch .gitignore` before install is byte-indistinguishable from install's `: > "$gi"`. So no clause below conditions on it.

**Two proofs, not one, and they do not generalise alike.** The direct form (`:613`) writes a target naming `.cyberos/plugin/skills/<entry>`: that text names the machine we deleted, so it proves ownership for *any* entry under *any* channel. The chained form (`:662`) writes `.agents/skills/<cmd> -> ../../.claude/skills/<cmd>`, which names no `.cyberos/` path at all - it is ours only *by construction*, and only for the three names `:649`'s `SHARED_CMDS` lists. Generalising the chained pattern to every entry in the directory would keep the pattern and destroy the justification: an operator's `.agents/skills/my-notes -> ../../.claude/skills/my-notes` matches it exactly and is not ours. The two forms are therefore separated below, and only the direct one is name-blind.

## Proposed Solution

Extend the readlink ownership test to every channel install writes, and apply the same read-the-artefact discipline to the MCP registrations and to both managed blocks. Four proofs, each evaluated at uninstall time from the artefact itself:

| artefact | proof it is ours | generalises? |
|---|---|---|
| skill symlink, direct form | `readlink` target text naming `.cyberos/plugin/skills/<entry>` (`:98`) | yes - the text names our deleted machine |
| skill symlink, chained form | ours by construction (`install.sh:662`) | no - only the three `:649` names, only under `.agents/skills/` |
| `.agents/skills` copy dir | the `.cyberos-owned` file it carries (`:101`) | yes - the marker is ours |
| `.gitignore` / hook block | the `# >>>` / `# <<<` marker pair delimiting it | yes - the markers are ours |
| MCP registration | the command naming `.cyberos/mcp/cyberos-mcp.mjs` | yes - the command names our deleted machine |

**The predicate is textual, never resolution.** `readlink` returns a string; a target naming our machine proves ownership whether or not it still resolves. This is what makes the rule identical under `CYBEROS_UNINSTALL_KEEP_BRAIN=1` (where `.cyberos/` survives with BRAIN inside) and `=0` (where it does not), and it is why an operator's own broken symlink pointing elsewhere is untouched: unresolvable is not the same as unowned, in either direction.

**What is provably ours is removed; the container is not.** The block comes out of `.gitignore`, the file stays. The registration comes out of `.mcp.json`, the file stays with an empty registry. The symlink goes, the channel dir stays. Removing a container would require knowing install created it, which nothing records - so the line is drawn where the proof runs out. An emptied container is inert; a pointer to a deleted file is not. This rule is applied to all six channel dirs, which costs `:118-119`'s `rmdir` of `.agents/skills`: that rmdir deletes a container for emptiness, which is the one move the rule forbids, and it is a live casualty for an operator whose empty `.agents/skills` predates install.

**Ownership and capability are different questions, and `.mcp.json` needs both.** What proves the *registration* is ours is its command text, exactly as `readlink` output proves a symlink. What byte-identity to `install.sh:685`'s form decides is something else entirely: whether `uninstall.sh` can rewrite the file **without a JSON parser it does not have**. So byte-identity is a capability gate, not an ownership test - which is why the prior draft's own objection ("byte-identity proves the content is ours; it cannot prove the file is") is correct and does not bite: nothing below uses byte-identity to prove the file is ours, and the file is never deleted either way.

**Strip the separator with the block, by bytes and not by lines.** Install always appends exactly one `\n` immediately before the `# >>>` marker (`:851-853`, `:913-915`: the heredoc's first line is blank), so the byte immediately before the marker is always ours. The strip must delete **that byte**, not that line. Measured: the line-oriented rule restores a newline-terminated hook exactly but adds a byte to a hook with no trailing newline (6B -> 7B); the byte-oriented rule is byte-exact for both, and for a hook ending in three newlines, and for a hook with content above and no trailing newline.

## Alternatives Considered

- **Have install write a receipt of what it created, so uninstall can know rather than guess.** The operator's instinct, floated for this task and not decided; adopted in spirit for TASK-IMP-122, where the underivable fact (what `install.sh` vendors) is load-bearing and has no safe fallback. It is rejected here because the fact it would record is not load-bearing and the fallback is safe. For every artefact install's own text already marks, the only clause a receipt changes is "delete the container": everything else is already decided from the artefact. And a receipt never decides *that* alone - an operator who adds `node_modules/` outside our markers owns that file whatever the receipt says, so the rule is always "receipt says created **and** the file is now empty", and the emptiness test does the work in every case. There is exactly one clause a receipt would genuinely decide: §3's `CYBEROS_COPY_SKILLS=1` gap, where an unmarked copy under `.claude/skills/` is provable from nothing on disk today. That is a real exception to the sentence above, and it is not enough - because a `.cyberos-owned` marker decides the same case, install already writes one at `:672-675`, and a marker is local to the artefact where a receipt is a new cross-script contract. Both are `install.sh` changes and both are out of scope here. Against that, a receipt costs a format, staleness semantics across re-installs, `install.sh` in `modified_files` - and a new casualty vector: a receipt that wrongly claims creation deletes an operator's file, which is the one outcome §1.9 forbids and which no mechanism can do today. Note also what a receipt does **not** rescue: it says nothing about how to remove one key from JSON an operator has since edited (§1.5), and nothing about `.gitignore`'s pre-install shape (§3), which install destroyed before uninstall ever ran. Trading a guardrail for tidiness is the wrong direction.
- **Delete an emptied container when it is byte-identical to install's exact output.** Rejected: that is the TASK-IMP-118 defect in miniature. Install's `.mcp.json` is a fixed 5-line, 105-byte string (measured) and `install.sh:689`'s own summary points operators at `.cyberos/mcp/README.md` to hand-register (`:683`'s comment says the same), so an operator-authored file can be byte-identical to ours. Byte-identity proves the *content* is ours; it cannot prove the *file* is. §1.4 uses it only as a capability gate, never as a licence to delete.
- **Trim trailing blank lines after the strip instead of consuming the separator.** Rejected: an operator hook whose pre-install content legitimately ends in three newlines would come back with one. Measured - the correct rule removes exactly one *byte* immediately above the marker, and that is byte-exact for the three-newline shape (9B -> 9B) as well as the no-newline shape (6B -> 6B).
- **Keep the prior draft's line-oriented strip and record the no-trailing-newline hook as impossible.** Rejected because the impossibility is false, and measurement says so: the byte-oriented rule inverts that shape (6B -> 6B). The error was a level confusion - the rule was reasoned about in line-space, where the no-newline case has no blank line to consume, and "my line rule cannot" was generalised to "no rule can". Install's `cat >>` is injective on hook content, so an inverse must exist; only the line-oriented expression of it fails.
- **Scope §1.6's container rule to five channel dirs and leave `:118-119` alone.** Rejected: it makes the clause satisfiable without making the behaviour defensible. `:118` deletes an operator's pre-existing empty `.agents/skills` today, which §1.9 forbids regardless of what §1.6 says, so scoping the container rule around it would leave the casualty and merely stop naming it. Either the container rule is principled or it is not; `.agents/skills` is a container by exactly the argument that keeps `.claude/skills`.
- **Remove the whole agent surface on uninstall.** Rejected: `.cursorrules`, `GEMINI.md` and the rules pointers may predate install or carry operator edits; unprovable ownership is not a licence to delete. TASK-IMP-094's recorded reasoning, and it still holds.
- **Leave the dangling links and let TASK-IMP-106's summary name them.** Rejected: 106 §1.5 forbids changing behaviour, so it can only describe the breakage honestly. A truthful report of a broken repo is not a fixed repo.
- **Effort 8 -> 9 hours.** Breakdown: channel loop rewrite (direct form, all six, name-blind) 1h; chained/direct split 0.5h; MCP partition 1.5h; byte-exact strip 1.5h; `:118-119` removal 0.5h; `t22` rename + re-point 0.5h; suite 3.5h.

## 3. Edge cases

- **RETRACTED: "an operator hook with NO trailing newline cannot be restored byte-exact."** The prior draft asserted this as an impossibility proof and it is FALSE. Install's `cat >>` appends exactly `\n` + block (`:851-853`: the heredoc's first line is blank), so the byte immediately before the marker is ALWAYS ours, for every shape. Deleting **that byte** rather than **that line** inverts the append. Measured this round, install replayed verbatim then each strip: line-oriented 6B -> 7B DIFFERS / byte-oriented 6B -> 6B BYTE-EXACT; newline-terminated 7B -> 7B both; three-newline 9B -> 9B both; no-newline-with-content-above 24B -> 25B line / 24B -> 24B byte. The information is not destroyed at append time - it survives as the presence or absence of the blank line above the marker, and install's append is injective on hook content. The prior author's error was a level confusion: the rule was reasoned about in line-space, where the no-newline case has no blank line to consume, and "my line rule cannot" became "no rule can". §1.7 therefore binds on ALL pre-install hook content.
- **The impossibility is real, but it is `.gitignore`'s and its cause is on the install side.** `install.sh:733-746`'s awk does not append to `.gitignore`, it REWRITES it: `:744` buffers each line and `print`s it back with a terminator, and `:745` trims the trailing blank run. Measured with that awk applied ALONE (no block, no separator): `*.log\n` (6B) -> 6B unchanged; `*.log\n\n\n` (8B) -> 6B; `*.log` (5B) -> 6B. Three distinct pre-install files collapse to one post-install state, so no uninstall-side rule can tell them apart - this is the pigeonhole the prior draft claimed for the hook, sitting on the other file. §1.7 is scoped to the newline-terminated, no-trailing-blank-run shape for `.gitignore` because that is the measured domain where install is injective. Closing the rest means changing `install.sh:744-745`, which is out of scope here; recorded as an uncovered gap, not handed to a clause that cannot honour it.
- **Limit of the byte-oriented rule as expressed in awk, found by testing it rather than assuming it.** Where an operator has added content BELOW our block and that content lacks a trailing newline, an awk END clause re-adds one: measured want 17B, got 18B. Where the below-content is newline-terminated it is byte-exact (18B -> 18B). Install always appends at EOF, so this shape only arises from an operator edit made between install and uninstall, and the residue lands on that later content, not on the pre-install bytes §1.7 and §1.9 speak about. Named because the rule is this author's own and its limits are the first thing that should be checked, not the last.
- `CYBEROS_UNINSTALL_KEEP_BRAIN=0` vs the default: in the default path `.cyberos/` SURVIVES (`:151` removes it, `:155-158` recreates it with BRAIN inside); under `=0` it does not. §1.1's predicate is textual, so it is the same rule in both, and both are covered by AC 9. This is the premise the first draft inverted; the fix is not to pick the other mode but to stop conditioning on the mode at all.
- A symlink into `.cyberos/` whose target was ALREADY absent before uninstall (a prior partial install): still provably ours by target text, still removed. Unresolvable is not unowned.
- **The chained form is bounded by name, and that is not a name exemption.** `.agents/skills/<cmd> -> ../../.claude/skills/<cmd>` names no `.cyberos/` path, so it is ours by construction (install writes exactly that shape at `:662`, for exactly the three names at `:649`) rather than by target text. §1.1's "no entry may be exempt by name" is therefore about the DIRECT form only - it is what removes `:112`'s guard. Widening the chained rule to every entry under `.agents/skills/` would keep the pattern and lose the justification: an operator's `.agents/skills/my-notes -> ../../.claude/skills/my-notes` matches it exactly. Bounding it to `:649`'s three names under `.agents/skills/` is precisely today's shipped behaviour at `:94`+`:98` - genuinely carried forward unchanged, and now stated as a scope limit rather than left as an accident of the loop.
- `CYBEROS_COPY_SKILLS=1`: the five native channels' copy fallback (`:613`'s `|| cp -R`) writes NO `.cyberos-owned` marker - only `.agents/skills` gets one (`:672-675`). So a copied `ship-tasks` under `.claude/skills/` is not provably ours and §1.3 keeps and reports it. It is untidy, not broken: a copy is self-contained and dangles at nothing. Closing it means marking the copy at install time, which is an `install.sh` change and out of scope here. This is the one case a receipt would genuinely decide - see Alternatives, where a marker is preferred and both are deferred.
- **`.mcp.json` an operator hand-registered BEFORE install, byte-identical to ours.** `install.sh:687` is create-if-absent, so install leaves it untouched; `:689`'s summary points operators at `.cyberos/mcp/README.md` to hand-register, so this is a supported path. §1.4 fires and removes the registration, mutating a path present before install. That is permitted by §1.9's carve-out and by nothing else: the registration's command names the machine we deleted, so the CONTENT is provably ours even though the FILE is not, and the file itself survives (§1.6). AC 9 pins the boundary - the file must still exist afterwards. An operator who wants the registration kept should keep the server.
- `.mcp.json` containing both our registration and an operator's: §1.5, not §1.4 - the file is not byte-identical to install's form, so it is left alone and the dead registration is reported. Surgically removing one key from arbitrary JSON needs a parser `uninstall.sh` does not have and `node` it must not require; reporting is what it can honestly do, and TASK-IMP-106's kept-list is the idiom for saying it.
- **An operator's pre-existing EMPTY `.agents/skills` (or `.agents`).** `uninstall.sh:118-119` rmdirs both when they empty, with no ownership check - rmdir's failure on a non-empty dir is the only guard, and it does not fire when the dir is an operator's and happens to be empty. That is a §1.9 casualty today, not a tension: a path present before install goes missing. §1.6 removes the rmdir rather than carving the clause around it. The cost is accepted and named: where install CREATED `.agents/skills`, an empty dir is now left behind, exactly as `.claude/skills` and the other four already are. An emptied container is inert.
- An operator hook whose pre-install content already ended in multiple newlines: restore is byte-exact to THAT content - §1.7 consumes exactly one byte, never normalises. Measured byte-exact for a hook ending in three newlines (9B -> 9B). Note this does NOT hold for `.gitignore`, where `:745` already trimmed those newlines at install time - see the second bullet.
- `CYBEROS_GLOBAL_SKILLS=1` copies `ship-tasks` into `$HOME/.claude/skills` and three siblings (`:634-637`); uninstall never touches them. Outside the repo is outside this task's contract ("leave the repo as it found it"); named so it is not mistaken for coverage.
- Uninstall run twice: the second run finds nothing provably ours and removes nothing; §1.8's byte-identity still holds.
- §1.9 compares post-uninstall bytes to PRE-INSTALL bytes, which is only meaningful for a file the operator did not edit between install and uninstall. An operator who edits their hook after install owns that edit and uninstall must not revert it; AC 9's fixtures do not edit between the two, and that is an assumption of the measurement, not a proven property of the code.
- Security-class: ownership is decided by `readlink` output, our own marker file, our marker pair, and a command string naming our path - never by a path pattern an operator could forge by naming a directory `ship-tasks`. No user-supplied string is interpolated into a removal command; nothing printed is executed.
