# TASK-IMP-121: Uninstall must leave the repo as it found it

## Summary

Uninstall deletes the machine and leaves things pointing at it. A byte-level harness on a fresh git repo found five dangling skill symlinks and two MCP registrations naming a server that is gone. Two further defects are byte-level: stripping our managed block from an operator's pre-commit hook leaves one newline behind every cycle, and doing the same to a pre-existing `.gitignore` leaves one behind once. A fifth defect runs the other way and the harness could not see it: `:118-119` `rmdir`s `.agents/skills` and `.agents` for emptiness alone, with no ownership test, so an operator's pre-existing empty `.agents/skills` is DESTROYED - reproduced end to end this round, and `.agents` with it when `CYBEROS_AGENTS` filters the antigravity pointer off. The harness's zero-casualty census holds only over its six cases as run, none of which had a pre-existing `.agents/skills`; it is not the general claim its wording invites. So the repo is left broken rather than merely untidy, two files an operator owns do not come back byte-identical, and one directory an operator owns does not come back at all.

## Problem

Measured 2026-07-18. `uninstall.sh:151` runs `rm -rf "$CY"`; `:155-158` then recreates `.cyberos/memory/` and moves the stashed BRAIN back. **`.cyberos/` survives by default** - what is gone in both modes is `plugin/`, `mcp/`, `cuo/`, `lib/`, `docs-tools/` and the root scripts. Every pointer install writes targets `.cyberos/plugin/skills/...` or `.cyberos/mcp/cyberos-mcp.mjs`, and both live in the part that is removed under `CYBEROS_UNINSTALL_KEEP_BRAIN=1` and `=0` alike. So the question is never "does `.cyberos/` still exist" - it is "does this pointer name a path we deleted", and that is answered by the pointer's own text.

**1. Five dangling skill symlinks.** Install writes six channels: `.agents/skills/`, `.claude/skills/`, `.grok/skills/`, `.commandcode/skills/`, `.codex/skills/`, `.opencode/skill/` (`:623-627`, `:631-632`, `:650-679`; note OpenCode's singular `skill`). Uninstall's loop (`:94`) looks at two. `.agents/skills/ship-tasks` is removed by `:98-99`; `.claude/skills/{task-author,task-audit}` by `:113-114`. `.claude/skills/ship-tasks` is exempted by name at `:112` (`[ "$_sc" != "ship-tasks" ]`), and `.grok`, `.commandcode`, `.codex`, `.opencode` are never looked at - `grep -ci` returns **0** for each of them in `uninstall.sh`. Five survivors, all `-> ../../.cyberos/plugin/skills/ship-tasks`, all confirmed dangling.

**2. Two dead MCP registrations.** `.mcp.json` and `.cursor/mcp.json` both register `node .cyberos/mcp/cyberos-mcp.mjs` (`:685-688`). Uninstall never unregisters - `grep -ci mcp` returns 0. Every agent reading those configs launches a missing server on every startup, in a repo the operator just cleaned.

**3. The hook gains a newline every cycle.** Install appends its block with `cat >> "$hk" <<'HOOK'` whose heredoc's **first line is blank** (`:851-853`), so the file gains `\n# >>> cyberos-status-hook v2`. `uninstall.sh:68`'s sed range starts **at** the marker, so the blank above it survives. Proven by the harness at 29 -> 30 -> 31 -> 32 bytes over three cycles, final tail `exit 0\n\n\n\n`; reproduced this round at 7 -> 8 -> 9 -> 10 on a 7-byte fixture. Unbounded: nothing ever trims it.

**4. The same defect in `.gitignore`, which the first draft missed.** `install.sh:747` writes `[ -s "$gi.cyberos.tmp" ] && printf '\n'` before the block, and `uninstall.sh:80`'s sed range also starts at the marker. Measured by replaying both verbatim: a pre-existing `.gitignore` goes **14 -> 15 bytes** and stays at 15 across three cycles - `install.sh:745`'s awk trims the trailing blank run on re-install, so this leak is one-time rather than accumulating. It is still a mutation of an operator's file, and it is why the fix belongs to the strip, not to the hook.

The `[ -s ]` guard at `:747` also explains the separate `.gitignore` symptom the harness saw: where install **created** the file (`:706`), the tmp is empty, no separator is written, and the strip leaves a 0-byte file.

**5. A directory an operator owns is destroyed - the one defect that runs the other way.** `uninstall.sh:118-119` `rmdir`s `.agents/skills` and then `.agents`, each `2>/dev/null || true`. Neither carries an ownership test: `rmdir`'s refusal on a non-empty directory is the ONLY guard, and it does not fire when the directory is an operator's and happens to be empty. Reproduced end to end this round: a fresh git repo with an operator's pre-existing empty `.agents/skills`, install, then uninstall - `.agents/skills` is GONE. In the default config `.agents` survives, but only incidentally: the `:588` antigravity pointer writes `.agents/rules/cyberos.md`, no uninstall code path removes it (`grep -n 'rules/' uninstall.sh` returns only `:92`, a comment, and that comment names `.devin/rules/`, `.windsurf/rules/` and `.windsurfrules` - not `.agents/rules/`), and the directory is therefore never empty. Filter antigravity off (`CYBEROS_AGENTS=agents`) and `find .agents` after uninstall returns nothing - the operator's `.agents` goes too. This is a §1.9 casualty, and it is the reason the container rule below is a marker gate rather than a blanket refusal to prune.

**Against section 6.** `uninstall.sh:162` reads `# 6. skill symlinks into .cyberos (dangling) - leave dirs; operator cleans`. The dangling is **known and deliberate**, and `:111`'s comment records the same posture for `.claude/skills/ship-tasks`. This is a decision to overturn, not a gap to fill. It is overturned because its premise does not survive contact with `:113-114`. The restraint section 6 encodes is the right instinct about *operator files* - and a symlink whose target text names our own machine is not an operator file. `:113-114` already trusts exactly that proof to remove `task-author` and `task-audit`, in the same run that leaves `ship-tasks` dangling. Section 6 is not restraint applied consistently; it is one channel's entry escaping a test its two siblings pass. Leaving a pointer to a file we deleted protects nobody.

**What ownership can and cannot be proven from - and what install can be made to record.** The discipline `:98-99` and `:113-114` use is: read the artefact, decide from what it says. It generalises to every artefact here - a marker-delimited block, an MCP command naming our path, a `.cyberos-owned` file. Where it stops is the *container*, and it stops there for a contingent reason rather than a necessary one. Whether install *created* `.gitignore` or `.mcp.json` is recorded nowhere: `install.sh`'s `wrote`, `AGENT_FILES` and `SKILL_DIRS` (`:479`, `:580`, `:621`, `:686`) are shell variables that die with the process, and an operator who ran `touch .gitignore` before install is byte-indistinguishable from install's `: > "$gi"` (`:706`). No clause below conditions on that.

But the container question is not undecidable in general - it is unrecorded, and install is the thing that would record it. **It already does, one directory too low.** `:672-675` writes a `.cyberos-owned` marker into each `.agents/skills/<cmd>` copy dir, `:101` reads it, and `:137` states the principle the marker exists to serve: *"what we did not create, we do not silently destroy."* Measured: `.agents/skills/.cyberos-owned` is ABSENT while `.agents/skills` EXISTS, because `:675` redirects into `$_sdest` - the skill dir (`:653`), not its parent. The parent carries no proof, which is exactly why `:118`'s `rmdir` is blind and deletes an operator's pre-existing empty `.agents/skills`. §1.11 moves that same marker up one level, under a create-test; §1.6 gates the `rmdir` on it. This is TASK-IMP-094's mechanism applied to the dir above, not a new one.

**Why the marker set is two dirs and not six.** `grep -n 'rmdir' uninstall.sh` returns `:118` (`.agents/skills`), `:119` (`.agents`) and `:158` (the BRAIN stash tmpdir). Those are the only directories uninstall can remove, so they are the only ones a marker changes a decision about. The five native channel parents are never `rmdir`'d - `grep -ci` returns 0 for grok/commandcode/codex/opencode/cursor - so marking them would write a file nothing reads and, because those dirs are never removed, would leave every one of those markers behind as a new permanent leftover. That is the defect this task exists to remove, arriving through the fix for it. The rule is therefore: mark exactly what the prune can delete.

**Two proofs, not one, and they do not generalise alike.** The direct form (`:613`) writes a target naming `.cyberos/plugin/skills/<entry>`: that text names the machine we deleted, so it proves ownership for *any* entry under *any* channel. The chained form (`:662`) writes `.agents/skills/<cmd> -> ../../.claude/skills/<cmd>`, which names no `.cyberos/` path at all - it is ours only *by construction*, and only for the three names `:649`'s `SHARED_CMDS` lists. Generalising the chained pattern to every entry in the directory would keep the pattern and destroy the justification: an operator's `.agents/skills/my-notes -> ../../.claude/skills/my-notes` matches it exactly and is not ours. The two forms are therefore separated below, and only the direct one is name-blind.

## Proposed Solution

Extend the readlink ownership test to every channel install writes, and apply the same read-the-artefact discipline to the MCP registrations and to both managed blocks. Four proofs, each evaluated at uninstall time from the artefact itself:

| artefact | proof it is ours | generalises? |
|---|---|---|
| skill symlink, direct form | `readlink` target text naming `.cyberos/plugin/skills/<entry>` (`:98`) | yes - the text names our deleted machine |
| skill symlink, chained form | ours by construction (`install.sh:662`) | no - only the three `:649` names, only under `.agents/skills/` |
| `.agents/skills` copy dir | the `.cyberos-owned` file it carries (`:101`) | yes - the marker is ours |
| `.agents/skills` and `.agents` themselves | the `.cyberos-owned` marker §1.11 has install write when, and only when, install CREATES the dir | no - only these two dirs, because only these two can be `rmdir`'d (`:118-119`) |
| `.gitignore` / hook block | the `# >>>` / `# <<<` marker pair delimiting it | yes - the markers are ours |
| MCP registration | the command naming `.cyberos/mcp/cyberos-mcp.mjs` | yes - the command names our deleted machine |

**The predicate is textual, never resolution.** `readlink` returns a string; a target naming our machine proves ownership whether or not it still resolves. This is what makes the rule identical under `CYBEROS_UNINSTALL_KEEP_BRAIN=1` (where `.cyberos/` survives with BRAIN inside) and `=0` (where it does not), and it is why an operator's own broken symlink pointing elsewhere is untouched: unresolvable is not the same as unowned, in either direction.

**What is provably ours is removed; a container is removed only when its creation is proven too.** The block comes out of `.gitignore`, the file stays. The registration comes out of `.mcp.json`, the file stays with an empty registry. The symlink goes, the channel dir stays. For `.gitignore`, `.mcp.json` and the five native channel parents the line is drawn where the proof runs out, and nothing below moves it: an emptied container is inert, and unprovable ownership is not a licence to delete.

`.agents/skills` and `.agents` are the two exceptions, because they are the only containers uninstall can remove (`:118-119`) and therefore the only ones where the proof is worth recording. **Install records it.** §1.11 has install write `.cyberos-owned` into each of those two dirs when, and only when, install itself creates the dir; §1.6 then permits the `rmdir` only for a dir carrying that marker and only once it is empty. Where install created the dir, uninstall takes it back and leaves nothing behind. Where the operator's dir predated install, there is no marker, the dir is kept, and uninstall says so - which is what `:118` fails to do today.

**The trap this must not walk into, and the test that avoids it.** `mkdir -p` succeeds silently on a directory that already exists (measured: exit 0). So `mkdir -p "$d"` followed by an unconditional marker write would stamp an operator's pre-existing directory as ours - the current bug inverted, and strictly worse, because today's defect merely deletes an unproven dir while that one would manufacture the proof first. The create-test must therefore run BEFORE the `mkdir`, in the shape `[ -d "$d" ] || { mkdir -p "$d"; marker "$d"; }`. Two measured facts stop that shape from being applied once and reused: `mkdir -p a/b` creates **both** levels, so a marker in `.agents/skills` says nothing about `.agents`; and `.agents` has a **second creator** at `:566` (the `:588` antigravity pointer, whose dirname is `.agents/rules`) which runs before `:660`. `.agents` therefore needs its own test, evaluated before the first line that can create it. §1.11 states both.

**Ownership and capability are different questions, and `.mcp.json` needs both.** What proves the *registration* is ours is its command text, exactly as `readlink` output proves a symlink. What byte-identity to `install.sh:685`'s form decides is something else entirely: whether `uninstall.sh` can rewrite the file **without a JSON parser it does not have**. So byte-identity is a capability gate, not an ownership test - which is why the prior draft's own objection ("byte-identity proves the content is ours; it cannot prove the file is") is correct and does not bite: nothing below uses byte-identity to prove the file is ours, and the file is never deleted either way.

**Strip the separator with the block, by bytes and not by lines.** Install always appends exactly one `\n` immediately before the `# >>>` marker (`:851-853`, `:913-915`: the heredoc's first line is blank), so the byte immediately before the marker is always ours. The strip must delete **that byte**, not that line. Measured: the line-oriented rule restores a newline-terminated hook exactly but adds a byte to a hook with no trailing newline (6B -> 7B); the byte-oriented rule is byte-exact for both, and for a hook ending in three newlines, and for a hook with content above and no trailing newline.

## 3. Edge cases

- **RETRACTED: "an operator hook with NO trailing newline cannot be restored byte-exact."** The prior draft asserted this as an impossibility proof and it is FALSE. Install's `cat >>` appends exactly `\n` + block (`:851-853`: the heredoc's first line is blank), so the byte immediately before the marker is ALWAYS ours, for every shape. Deleting **that byte** rather than **that line** inverts the append. Measured this round, install replayed verbatim then each strip: line-oriented 6B -> 7B DIFFERS / byte-oriented 6B -> 6B BYTE-EXACT; newline-terminated 7B -> 7B both; three-newline 9B -> 9B both; no-newline-with-content-above 24B -> 25B line / 24B -> 24B byte. The information is not destroyed at append time - it survives as the presence or absence of the blank line above the marker, and install's append is injective on hook content. The prior author's error was a level confusion: the rule was reasoned about in line-space, where the no-newline case has no blank line to consume, and "my line rule cannot" became "no rule can". §1.7 therefore binds on ALL pre-install hook content.
- **The impossibility is real, but it is `.gitignore`'s and its cause is on the install side.** `install.sh:733-746`'s awk does not append to `.gitignore`, it REWRITES it: `:744` buffers each line and `print`s it back with a terminator, and `:745` trims the trailing blank run. Measured with that awk applied ALONE (no block, no separator): `*.log\n` (6B) -> 6B unchanged; `*.log\n\n\n` (8B) -> 6B; `*.log` (5B) -> 6B. Three distinct pre-install files collapse to one post-install state, so no uninstall-side rule can tell them apart - this is the pigeonhole the prior draft claimed for the hook, sitting on the other file. §1.7 is scoped to the newline-terminated, no-trailing-blank-run shape for `.gitignore` because that is the measured domain where install is injective. Closing the rest means changing `install.sh:744-745` to stop rewriting the file - a different and larger change than §1.11's two marker writes, and out of scope even though `install.sh` is now in scope for those. Recorded as an uncovered gap, not handed to a clause that cannot honour it. Re-measured this round and reproduced exactly.
- **CORRECTED: the "limit of the byte-oriented rule" is a limit of the awk EMIT, not of the rule.** The third author recorded that where an operator has added content BELOW our block with no trailing newline, "an awk END clause re-adds one: want 17B, got 18B" - and flagged it as their own rule's limit, which is the right instinct. **The measurement reproduces; the attribution does not.** Re-measured this round on the same shape with the same `sub()` and only the emit varied: `print $0` -> **18B DIFFERS**; `printf "%s", $0` -> **17B EXACT**. `print` appends `ORS`; `printf "%s"` appends nothing. So the shape IS recoverable, `printf "%s"` remains byte-exact on all four base shapes (6B/7B/9B/23B), and nothing here narrows §1.7. Kept as an edge case because the trap is real for anyone who reaches for `print`, and because it is the exact failure the round keeps repeating one level down: the author tested their own rule, which audit.md §3.2 records as the session's first instance of that discipline, and then stopped one step short of the cause.
- **UNVERIFIED, and named rather than assumed: the byte-oriented strip is measured only on GNU awk.** The expression above uses `RS="^$"` whole-file slurp, which is a GNU extension, and it was measured on GNU Awk 5.1.0 on Linux. `install.sh:98` records that "the payload's floor is POSIX" and `:153` already branches GNU `-c` against BSD `-f`, so the payload runs on macOS's BSD awk, where `RS="^$"` does not slurp. Whether the byte-oriented rule is expressible in BSD awk is NOT tested here. §1.7 binds on the BEHAVIOUR - delete exactly one byte, never a line - and not on any tool, so the clause stands and is implementable by other means; the implementer must not carry the gawk form over untested. Recorded because it is this author's own construct, and an originated claim is the one that arrives already believed.
- `CYBEROS_UNINSTALL_KEEP_BRAIN=0` vs the default: in the default path `.cyberos/` SURVIVES (`:151` removes it, `:155-158` recreates it with BRAIN inside); under `=0` it does not. §1.1's predicate is textual, so it is the same rule in both, and both are covered by AC 9. This is the premise the first draft inverted; the fix is not to pick the other mode but to stop conditioning on the mode at all.
- A symlink into `.cyberos/` whose target was ALREADY absent before uninstall (a prior partial install): still provably ours by target text, still removed. Unresolvable is not unowned.
- **The chained form is bounded by name, and that is not a name exemption.** `.agents/skills/<cmd> -> ../../.claude/skills/<cmd>` names no `.cyberos/` path, so it is ours by construction (install writes exactly that shape at `:662`, for exactly the three names at `:649`) rather than by target text. §1.1's "no entry may be exempt by name" is therefore about the DIRECT form only - it is what removes `:112`'s guard. Widening the chained rule to every entry under `.agents/skills/` would keep the pattern and lose the justification: an operator's `.agents/skills/my-notes -> ../../.claude/skills/my-notes` matches it exactly. Bounding it to `:649`'s three names under `.agents/skills/` is precisely today's shipped behaviour at `:94`+`:98` - genuinely carried forward unchanged, and now stated as a scope limit rather than left as an accident of the loop.
- `CYBEROS_COPY_SKILLS=1`: the five native channels' copy fallback (`:613`'s `|| cp -R`) writes NO `.cyberos-owned` marker - only `.agents/skills/<cmd>` gets one (`:672-675`). So a copied `ship-tasks` under `.claude/skills/` is not provably ours and §1.3 keeps and reports it. It is untidy, not broken: a copy is self-contained and dangles at nothing. **Its exclusion no longer rests on "install.sh is out of scope"** - option C puts install.sh in scope, so that justification is retired and the honest one stated instead: this is an ENTRY-level ownership question under channel dirs the prune cannot reach (`grep -n 'rmdir'` returns only `:118`/`:119`/`:158`), so §1.6's container rule never fires on it and no deletion this spec authorises turns on it. Marking those copies would widen the task without changing any outcome §1.1-§1.12 decide. Named as an uncovered gap, not folded in for symmetry.
- **`.mcp.json` an operator hand-registered BEFORE install, byte-identical to ours.** `install.sh:687` is create-if-absent, so install leaves it untouched; `:689`'s summary points operators at `.cyberos/mcp/README.md` to hand-register, so this is a supported path. §1.4 fires and removes the registration, mutating a path present before install. That is permitted by §1.9's carve-out and by nothing else: the registration's command names the machine we deleted, so the CONTENT is provably ours even though the FILE is not, and the file itself survives (§1.6). AC 9 pins the boundary - the file must still exist afterwards. An operator who wants the registration kept should keep the server.
- `.mcp.json` containing both our registration and an operator's: §1.5, not §1.4 - the file is not byte-identical to install's form, so it is left alone and the dead registration is reported. Surgically removing one key from arbitrary JSON needs a parser `uninstall.sh` does not have and `node` it must not require; reporting is what it can honestly do, and TASK-IMP-106's kept-list is the idiom for saying it.
- **An operator's pre-existing EMPTY `.agents/skills` (or `.agents`) - the case the operator's decision turns on.** `uninstall.sh:118-119` rmdirs both when they empty, with no ownership check: rmdir's failure on a non-empty dir is the only guard, and it does not fire when the dir is an operator's and happens to be empty. Measured end to end this round: with a pre-existing empty `.agents/skills`, install then uninstall DELETES it. That is a §1.9 casualty, not a tension - a path present before install goes missing. Under option C the marker decides it: no marker on that dir (install did not create it), so §1.6 keeps it and reports it. Where install DID create it, the marker is there, uninstall takes it back, and no empty dir is left behind - which is what option A would have cost.
- **`:119` is reachable, so gating it is not dead code.** In the DEFAULT config `.agents` survives regardless of any marker, because the `:588` antigravity pointer writes `.agents/rules/cyberos.md` and NO uninstall code path removes it - so `.agents` is never empty and `:119`'s rmdir no-ops. (The survival is by absence of code: `grep -n 'rules/' uninstall.sh` returns only `:92`, a comment naming `.devin/rules/`, `.windsurf/rules/` and `.windsurfrules`, none of which is `.agents/rules/`. Citing `:92-93` here would be audit 1's ISS-006 - a three-path comment stretched to cover a fourth - so it is not cited.) That made `:119` look inert. It is not: measured with `CYBEROS_AGENTS=agents` (antigravity filtered off, no `.agents/rules/` written), `find .agents` after uninstall returns NOTHING - the operator's own pre-existing `.agents` AND `.agents/skills` are both destroyed. Both dirs therefore need their own marker and their own gate.
- **`mkdir -p` creates the whole chain, which is why `.agents` cannot inherit `.agents/skills`'s proof.** Measured: `mkdir -p chain/agents/skills` creates `chain/agents` as well as `chain/agents/skills`. So when install runs `:660` in a repo with no `.agents` at all, ONE command creates both levels and a single marker in the child would leave `:119` exactly as blind as today. Compounding it, `.agents` has a SECOND creator - `:566`'s `mkdir -p "$(dirname "$abs")"` via the `:588` antigravity pointer, whose dirname is `.agents/rules` - and that one runs FIRST (566 < 588 < 660). Measured with `CYBEROS_AGENTS=antigravity`: `.agents` and `.agents/rules/cyberos.md` are created with no `.agents/skills`. §1.11 therefore tests each directory's own existence before any `mkdir` that could create it, rather than testing once at `:660`.
- **The marker on a KEPT directory would be a new leftover, so it comes off either way.** Where install created `.agents` and the antigravity pointer is installed, uninstall keeps `.agents` (its `rules/cyberos.md` survives by design) - and a marker left inside it would be a permanent file this task's own title forbids. §1.12 therefore removes the marker BEFORE the `rmdir` and regardless of whether the `rmdir` succeeds: the marker is provably ours, so §1.9 permits removing it, and the dir is simply kept without it. The sequencing is forced in the other direction too - the marker is a file, so while it is present the dir is never empty and the `rmdir` could never fire.
- **Re-install after adoption keeps the adoption.** If the operator deletes `.agents/skills/.cyberos-owned` to adopt the dir and later re-installs, `:660`'s `[ -d "$d" ]` test is TRUE, so install neither mkdirs nor re-marks, and the dir stays adopted across re-installs. That falls out of §1.11's create-test rather than needing its own rule, and it is the behaviour `:674-675`'s adoption sentence promises. The same test makes the marker write idempotent: a dir install created and marked is not re-marked on re-install.
- An operator hook whose pre-install content already ended in multiple newlines: restore is byte-exact to THAT content - §1.7 consumes exactly one byte, never normalises. Measured byte-exact for a hook ending in three newlines (9B -> 9B). Note this does NOT hold for `.gitignore`, where `:745` already trimmed those newlines at install time - see the second bullet.
- `CYBEROS_GLOBAL_SKILLS=1` copies `ship-tasks` into `$HOME/.claude/skills` and three siblings (`:634-637`); uninstall never touches them. Outside the repo is outside this task's contract ("leave the repo as it found it"); named so it is not mistaken for coverage.
- Uninstall run twice: the second run finds nothing provably ours and removes nothing; §1.8's byte-identity still holds.
- §1.9 compares post-uninstall bytes to PRE-INSTALL bytes, which is only meaningful for a file the operator did not edit between install and uninstall. An operator who edits their hook after install owns that edit and uninstall must not revert it; AC 9's fixtures do not edit between the two, and that is an assumption of the measurement, not a proven property of the code.
- Security-class: ownership is decided by `readlink` output, our own marker file, our marker pair, and a command string naming our path - never by a path pattern an operator could forge by naming a directory `ship-tasks`. No user-supplied string is interpolated into a removal command; nothing printed is executed.
