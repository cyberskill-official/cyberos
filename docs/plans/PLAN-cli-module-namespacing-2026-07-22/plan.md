# plan@1: CLI module namespacing (fix the `cyberos` name collision)

Status: REVISED 2026-07-22. Original decision (rename `modules/memory`'s console script, add `memory`/
`cuo` verbs under the `cyberos` bin) approved, then superseded same day when the founder clarified the
public CLI's bin name itself changes. See section 4 for the revised decision and section 9 for the
revision log. Awaiting `/create-tasks` to turn section 6 into a real backlog.

Process note: this plan was authored without a locally installed `.cyberos/` machine (this checkout
is a fresh, throwaway clone; `.cyberos/` is gitignored and regenerated by `cyberos install`, which was
not run here). The repo-wide scan below was done by direct reads (Read/Grep/Bash) rather than through
`repo-context-map-author`, so there is no `repo-context-map@1` artifact to cite as `scan_ref` — this is
a deviation from the standard `plan-author` flow, flagged rather than faked. Section 8 (BRAIN rows) is
likewise not appended to any BRAIN store for the same reason; that append should happen for real the
first time `/create-tasks` or `/ship-tasks` runs this against an installed repo.

## 1. Intent

Give the public CyberOS CLI its own name (`cs`), separate from the internal BRAIN-store CLI that
happens to already use the name `cyberos`, and give the `memory` and `cuo` command families a real,
addressable place inside `cs`. Do this without breaking the existing npm trusted-publishing pipeline,
and without claiming `cuo` commands can run standalone outside Claude Code before that's actually true.

## 2. Context

Raised by the founder (Stephen) after `brew install --cask cyberos` + `brew trust` succeeded locally,
but `cyberos -h` kept showing an unrelated-looking 39-subcommand BRAIN-store CLI instead of the
Homebrew-installed one. Root cause, confirmed by direct repo inspection (manual scan, see process
note above):

- `tools/install/plugin/commands/*.md` — the Claude Code plugin's slash commands: `/install`,
  `/uninstall`, `/version`, `/status`, `/help`, `/plan`, `/create-tasks`, `/improve`, `/ship-tasks`.
- `tools/install/cli/bin/cli.mjs` — the npm-published `@cyberskill/cyberos` CLI, currently exposed to
  users as the bin command `cyberos`. Its own header comment states the design intent directly: "the
  commands below mirror the payload's own surface (help.sh) and the plugin's slash commands 1:1, so the
  three channels cannot drift apart." Dispatch is a flat `SCRIPTS` table keyed on `argv[0]`: `install`,
  `uninstall`, `version`, `status`, `help`, `create`, plus `mcp` and `gates` handled inline.
  `docs/index.md` confirms: "via the npm package the same eight commands are `npx cyberos <command>`."
- `modules/memory/pyproject.toml` — a separate Python package, `cyberos-memory`, confirmed **not**
  published to PyPI (`pip index versions cyberos-memory` returns no matching distribution) and
  confirmed by the founder to be internal-only, with no plan to publish it. Its `[project.scripts]`
  entry is `cyberos = "cyberos.__main__:main"` — the 39-subcommand BRAIN-store CLI (init, view, put,
  move, delete, verify, export, audit, search, semantic-sync, checkpoint, import, backup, prune, prove,
  verify-proof, sth-wrap, state, consolidate, doctor, crypto-mode, session, serve, publish, digest,
  resolve-conflict, validate, history, transcript, put-if, acl, dream, dream-apply, episode,
  recall-similar, self-update, skill, workflow). It reaches a PATH only via local/dev editable installs
  (which is exactly what happened on the founder's machine, through pyenv).
- `modules/cuo/pyproject.toml` — `cyberos-cuo`, "persona-aware orchestration above SKILL module", built
  on `click`. This backs `/plan`, `/create-tasks`, `/ship-tasks`, `/improve`. Those slash commands are
  multi-step, LLM-orchestrated skills (interview, draft, self-audit, HITL approval gate) — not
  deterministic scripts — so they cannot be wrapped as a literal shell subcommand today without
  deciding how (or whether) to run an agentic session headlessly from a CLI invocation.

The original round of this plan (2026-07-22, morning) proposed renaming `modules/memory`'s colliding
console-script entry so the public npm CLI could safely keep the name `cyberos`. The founder then
clarified the actual target state: `cs` is the only public command; `cyberos-memory` is confirmed
internal-only and was never going to be published regardless. That inverts which name needs to move —
see section 4.

## 3. Options

**A. Node absorbs everything.** Extend `cli.mjs` to add `memory` (and eventually `cuo`) verbs that
`spawnSync` into the Python packages as subprocesses. Evidence: `cli.mjs` already does this pattern
for `mcp` and `gates`. Cost: makes Python a hard runtime dependency of what is currently a pure-Node
CLI, and adds a subprocess/stdio boundary for every memory operation. (Independent of the naming
question below — this option is about which runtime dispatches `memory`/`cuo`, not about the bin name.)

**B. Python absorbs everything.** Port `install`/`uninstall`/`version`/`status`/`help` off Node,
publish the CLI from Python instead. Evidence: those five commands are already thin wrappers around
`.sh` scripts (`install.sh`, `status.sh`, `uninstall.sh`, `version.sh`), so the logic itself is
portable independent of dispatch language. Cost: breaks the existing OIDC trusted-publishing setup on
the npm package — `release.yml`'s own comments flag this wiring as fragile (pinned to
org+repo+workflow-filename trust, breaks silently on a rename). The Homebrew formula just shipped
(`cyberos-cli.rb`, npm-based) would need a different packaging model entirely.

**C. Keep two runtimes; rename the PUBLIC bin to `cs`; add namespaces under it.** Change the npm
package's exposed bin command from `cyberos` to `cs` (package.json's `bin` field, generated by
`build.sh`); add `memory` and `cuo` as new top-level verbs in `cs`'s (formerly `cyberos`'s) dispatch
table. `modules/memory`'s internal console-script entry does **not** need renaming for collision
reasons anymore, since the public command is moving off the name `cyberos` entirely — but `cs memory`
still needs a real dispatch path into that module's logic (subprocess spawn, per option A, or a direct
library call if Node ever absorbs it, per option B). `cuo`, since its commands are LLM-orchestrated and
not deterministic, starts as a redirect stub — it prints which slash command to run inside Claude Code,
rather than faking standalone execution. Evidence: reuses the dispatch pattern that already exists;
`modules/memory` staying internal-only (confirmed) means renaming it was never actually necessary to
solve the founder's real goal, which is a single clean public name.

## 4. Decision

Option C, revised: rename the **public** bin (`cyberos` → `cs`) rather than the internal one.
Confidence: medium-high.

Rationale: the founder's actual target state is "the only public [command] is `cs`" — `cyberos-memory`
was already confirmed internal-only with no publishing plans, so it was never the thing that needed to
change for correctness. Renaming the public bin directly delivers a single, unambiguous public command
and sidesteps the PATH-collision problem entirely (nothing internal claims `cs`). It is a more visible,
more breaking change than the original plan (anyone who has already typed `npx cyberos` or `cyberos
install` needs to learn the new name), but it is honest about what "the only public command" actually
requires, rather than working around it with an internal rename that leaves two public-facing names
in play. The `cuo`-as-stub caveat from the original decision still applies unchanged.

Operator verdict: **APPROVE** (recorded 2026-07-22, via direct conversation with the founder — not
through an installed `.cyberos` HITL gate, since none is installed in this checkout). This supersedes
the original APPROVE recorded earlier the same day against the pre-revision decision (see section 9).

Open design question not resolved by this plan (left for `/create-tasks` or implementation to decide):
whether the npm **package name** itself (`@cyberskill/cyberos`) also changes, or only the `bin` field
changes while the package name stays the same (npm allows a package name to differ from its bin
command name; many packages do this). This plan only commits to the bin/command name becoming `cs`.

## 5. Scope

In scope:
- Change the npm package's `bin` field (rendered by `build.sh`) so the installed command is `cs`
  instead of `cyberos`. Decide (as part of task 1, not pre-decided here) whether the underlying npm
  package name also changes or only the bin entry does.
- Update `tools/install/cli/bin/cli.mjs`'s own usage text and every internal reference from `cyberos
  <command>` to `cs <command>`.
- Add `memory` as a new top-level verb under `cs`, dispatching into `modules/memory`'s existing logic
  (mechanism — subprocess vs. other — is an implementation detail for `/create-tasks`/`/ship-tasks`,
  not decided here).
- Add `cuo` as a new top-level verb under `cs`, as a documentation/redirect stub only.
- Update `tools/install/help.sh`, `docs/index.md`'s "same eight commands"/`npx cyberos` references, and
  the plugin's `help.md` to describe the new `cs`-based surface (`install`/`uninstall`/`version`/
  `status`/`help` stay flat top-level under `cs`; `memory` and `cuo` become namespaces).
- README/CHANGELOG updates for the rename.

Out of scope (explicitly not part of this plan):
- Renaming `modules/memory`'s own console-script entry — not needed for the collision problem anymore,
  and the founder confirmed it stays internal, so left alone unless a future need arises.
- Option A or B's runtime-consolidation question (which runtime ultimately executes `memory`/`cuo`) —
  deferred, not decided against permanently, just not blocking this pass.
- Building real headless/agentic execution for `cuo` commands from a bare shell invocation — deferred
  as its own follow-up decision.
- Any change to BRAIN store internals, schema, or the memory subcommands' own behavior.
- Any change to `release-pkgmgr-pr.yml` or its Homebrew/winget manifest generation logic itself — flagged
  as a downstream consumer (section 7), not modified by this plan.
- Deciding the npm package-name question in section 4's open design question — flagged for
  implementation, not resolved here.

## 6. Proposed Task Set

1. Decide and implement the npm package's new public identity: rename the `bin` field from `cyberos`
   to `cs` in `build.sh`'s package.json generation (and decide whether `@cyberskill/cyberos` the
   package name also changes, or only the bin command does).
2. Update `tools/install/cli/bin/cli.mjs` usage text and all `cyberos <command>` examples to `cs
   <command>`.
3. Add a `memory` verb to `cs`'s dispatch table that reaches `modules/memory`'s BRAIN-store operations
   (dispatch mechanism is an implementation decision, not fixed by this plan).
4. Add a `cuo` verb to `cs`'s dispatch table as a redirect stub (prints the matching slash command to
   run inside Claude Code); do not implement standalone execution in this task.
5. Update `help.sh`, `docs/index.md`, the plugin's `help.md`, README, and CHANGELOG for the `cs` rename
   and the new `memory`/`cuo` namespaces.
6. Follow-up note (tracked here, not implemented by this plan) against the Homebrew tap
   (`cyberskill-official/homebrew-tap`): `Formula/cyberos-cli.rb`'s `test do` block currently asserts
   `bin/"cyberos"`, and its description assumes the old flat eight-command `cyberos` surface. Both need
   updating once the npm package's bin name actually changes — Homebrew's `bin.install_symlink
   Dir["#{libexec}/bin/*"]` will pick up whatever bin name the package declares automatically, but the
   Formula's own test assertion and prose will not update themselves. The Cask's `depends_on formula:
   "cyberos-cli"` relationship and description likely also want a follow-up pass for naming
   consistency, though the dependency mechanism itself does not break.
7. End-to-end regression test on a clean machine (fresh Python env + fresh Homebrew install once the
   tap is updated): confirm `cs <top-level cmd>` works, `cyberos` cleanly no longer resolves to the
   public installer CLI, and `cs memory <cmd>` reaches the memory module without collision.

## 7. Risks

- This is a breaking rename for anyone who has already run `npx cyberos` or installed the old bin name
  — no transition alias is planned (the founder explicitly rejected keeping `cyberos` as an alias
  alongside `cs`). Mitigated in severity by how new and low-distribution this CLI still is (npm
  publishing only began with the 1.0.9 release cycle), but worth a deliberate CHANGELOG callout rather
  than a silent swap.
- Touches `tools/install/cli/bin/cli.mjs` and the package.json generation in `build.sh`, both of which
  ship through the OIDC trusted-publishing pipeline in `release.yml`. That pipeline's own comments
  describe the trust wiring as fragile (pinned to org+repo+workflow-filename); a bin-name or
  package-name change should go through the same release flow and be re-tested end to end.
- Downstream break: the Homebrew tap's `Formula/cyberos-cli.rb` (shipped this session) hardcodes the
  `cyberos` bin name in its `test do` assertion and its description. It will silently start asserting
  against a binary that no longer exists once the npm package renames, unless task 6 lands in the same
  window. Flagged explicitly so it isn't discovered later as a broken `brew test`.
- The `cuo` verb as a stub does not fully satisfy "the CLI and slash commands behave the same" — only
  namespaces it under the new public name. Flagged explicitly rather than silently scoped out; a real
  answer requires a separate decision on headless execution.
