# plan@1: CLI module namespacing (fix the `cyberos` name collision)

Status: decision recorded, operator verdict APPROVE (2026-07-22). Awaiting `/create-tasks` to turn
section 6 into a real backlog.

Process note: this plan was authored without a locally installed `.cyberos/` machine (this checkout
is a fresh, throwaway clone; `.cyberos/` is gitignored and regenerated by `cyberos install`, which was
not run here). The repo-wide scan below was done by direct reads (Read/Grep/Bash) rather than through
`repo-context-map-author`, so there is no `repo-context-map@1` artifact to cite as `scan_ref` — this is
a deviation from the standard `plan-author` flow, flagged rather than faked. Section 8 (BRAIN rows) is
likewise not appended to any BRAIN store for the same reason; that append should happen for real the
first time `/create-tasks` or `/ship-tasks` runs this against an installed repo.

## 1. Intent

Stop two independently-installable packages in this repo from both claiming the executable name
`cyberos` on a user's PATH, and give the `memory` and `cuo` command families a real, addressable
place inside the CLI, without breaking the existing npm trusted-publishing pipeline or claiming
`cuo` commands can run standalone outside Claude Code before that's actually true.

## 2. Context

Raised by the founder (Stephen) after `brew install --cask cyberos` + `brew trust` succeeded locally,
but `cyberos -h` kept showing an unrelated-looking 39-subcommand BRAIN-store CLI instead of the
Homebrew-installed one. Root cause, confirmed by direct repo inspection (manual scan, see process
note above):

- `tools/install/plugin/commands/*.md` — the Claude Code plugin's slash commands: `/install`,
  `/uninstall`, `/version`, `/status`, `/help`, `/plan`, `/create-tasks`, `/improve`, `/ship-tasks`.
- `tools/install/cli/bin/cli.mjs` — the npm-published `@cyberskill/cyberos` CLI. Its own header
  comment states the design intent directly: "the commands below mirror the payload's own surface
  (help.sh) and the plugin's slash commands 1:1, so the three channels cannot drift apart." Dispatch
  is a flat `SCRIPTS` table keyed on `argv[0]`: `install`, `uninstall`, `version`, `status`, `help`,
  `create`, plus `mcp` and `gates` handled inline. `docs/index.md` confirms: "via the npm package the
  same eight commands are `npx cyberos <command>`."
- `modules/memory/pyproject.toml` — a separate Python package, `cyberos-memory`, whose
  `[project.scripts]` entry is `cyberos = "cyberos.__main__:main"`. This is the 39-subcommand BRAIN-store
  CLI (init, view, put, move, delete, verify, export, audit, search, semantic-sync, checkpoint, import,
  backup, prune, prove, verify-proof, sth-wrap, state, consolidate, doctor, crypto-mode, session, serve,
  publish, digest, resolve-conflict, validate, history, transcript, put-if, acl, dream, dream-apply,
  episode, recall-similar, self-update, skill, workflow — confirmed against `modules/memory/cyberos/`
  source files, e.g. `core/episode.py`). It is a different runtime (Python vs. Node) and a different
  product surface (direct memory-store plumbing vs. repo-orchestration), but claims the same bin name.
- Verified via `pip index versions cyberos-memory`: no matching distribution — this package is not
  published to PyPI. It only reaches a PATH via a local/dev editable install (which is exactly what
  happened on the founder's machine, through pyenv). No outside consumer depends on the current name.
- `modules/cuo/pyproject.toml` — `cyberos-cuo`, "persona-aware orchestration above SKILL module",
  built on `click`. This is what backs `/plan`, `/create-tasks`, `/ship-tasks`, `/improve`. Those
  slash commands are multi-step, LLM-orchestrated skills (interview, draft, self-audit, HITL approval
  gate) — not deterministic scripts — so they cannot be wrapped as a literal shell subcommand today
  without deciding how (or whether) to run an agentic session headlessly from a CLI invocation.

## 3. Options

**A. Node absorbs everything.** Extend `cli.mjs` to add `memory` (and eventually `cuo`) verbs that
`spawnSync` into the Python packages as subprocesses. Evidence: `cli.mjs` already does this pattern
for `mcp` and `gates`. Cost: makes Python a hard runtime dependency of what is currently a pure-Node
CLI, and adds a subprocess/stdio boundary for every memory operation.

**B. Python absorbs everything.** Port `install`/`uninstall`/`version`/`status`/`help` off Node,
publish the CLI from Python instead. Evidence: those five commands are already thin wrappers around
`.sh` scripts (`install.sh`, `status.sh`, `uninstall.sh`, `version.sh`), so the logic itself is
portable independent of dispatch language. Cost: breaks the existing OIDC trusted-publishing setup on
`@cyberskill/cyberos` — `release.yml`'s own comments flag this wiring as fragile (pinned to
org+repo+workflow-filename trust, breaks silently on a rename). The Homebrew formula just shipped
(`cyberos-cli.rb`, npm-based) would need a different packaging model entirely.

**C. Keep two runtimes; fix the collision; add namespaces.** Rename `modules/memory`'s console-script
entry point away from the bare name `cyberos` (e.g. to `cyberos-memory-cli`); add `memory` and `cuo` as
new top-level verbs in `cli.mjs`'s `SCRIPTS`-style dispatch. `memory` execs into the renamed Python
binary. `cuo`, since its commands are LLM-orchestrated and not deterministic, starts as a redirect
stub — it prints which slash command to run inside Claude Code, rather than faking standalone
execution. Evidence: smallest diff, reuses the dispatch pattern that already exists and that the file's
own comments say is meant to stay 1:1 across channels; the rename is a non-breaking change today per
the PyPI check above.

## 4. Decision

Option C. Confidence: medium-high.

Rationale: it fixes the actual, reproduced bug (silent PATH shadowing between two same-named
executables) with the smallest blast radius, and it is honest about the one thing that genuinely
cannot be papered over in this pass — `cuo` commands need a real decision on headless/agentic
execution before "the CLI and the slash commands behave the same" is fully true for that half of the
surface. That decision is flagged as follow-up work (section 5), not silently deferred.

Operator verdict: **APPROVE** (recorded 2026-07-22, via direct conversation with the founder — not
through an installed `.cyberos` HITL gate, since none is installed in this checkout).

## 5. Scope

In scope:
- Rename `modules/memory`'s `[project.scripts]` entry away from the bare `cyberos` name.
- Update every internal reference to the old bare-name invocation (docs, tests, scripts) inside
  `modules/memory` and anywhere else in the repo that assumes `cyberos <memory-subcommand>` directly.
- Add `memory` as a new top-level verb in `tools/install/cli/bin/cli.mjs`, dispatching to the renamed
  binary.
- Add `cuo` as a new top-level verb in the same file, as a documentation/redirect stub only.
- Update `cli.mjs`'s own usage text, `tools/install/help.sh`, `docs/index.md`'s "same eight commands"
  line, and the plugin's `help.md` to describe the new module-namespaced surface.

Out of scope (explicitly not part of this plan):
- Option A or B (single-runtime consolidation) — deferred, not decided against permanently, just not
  this pass.
- Building real headless/agentic execution for `cuo` commands from a bare shell invocation — deferred
  as its own follow-up decision (needs a design for running an agent session outside Claude Code,
  including model/API access, which is a materially different problem than namespacing).
- Any change to BRAIN store internals, schema, or the memory subcommands' own behavior — this plan
  only touches how the binary is named and reached, not what it does.
- Any change to `release-pkgmgr-pr.yml` or its Homebrew/winget manifest generation logic — that
  workflow is only a downstream consumer to notify, not something this plan modifies.

## 6. Proposed Task Set

1. Rename `modules/memory/pyproject.toml`'s `[project.scripts]` entry from `cyberos` to
   `cyberos-memory-cli` (or equivalent non-colliding name); update every internal reference.
2. Extend `tools/install/cli/bin/cli.mjs` to add a `memory` verb that execs the renamed console
   script, with a clear, actionable error if it is not installed/on PATH.
3. Add a `cuo` verb to `cli.mjs` as a redirect stub (prints the matching slash command to run inside
   Claude Code); do not implement standalone execution in this task.
4. Update `cli.mjs` usage text, `help.sh`, `docs/index.md`'s "same eight commands" line, and the
   plugin's `help.md` to reflect the new module-namespaced surface (`install`/`uninstall`/`version`/
   `status`/`help` stay flat top-level; `memory` and `cuo` become namespaces).
5. Update README/CHANGELOG; open a follow-up note against the Homebrew tap's `cyberos-cli.rb`
   description, since its current text assumes only the flat eight-command surface.
6. End-to-end regression test on a clean machine (fresh Python env + fresh Homebrew install): confirm
   `which cyberos` resolves consistently and both `cyberos <top-level cmd>` and
   `cyberos memory <cmd>` work without collision.

## 7. Risks

- Machines that already have the old-named Python console script installed (editable/dev installs)
  keep colliding until the developer reinstalls; the rename does not retroactively fix machines that
  already resolved the old name first on PATH. Mitigated in severity by the PyPI check above — no
  external consumer exists yet, so the affected population is internal dev machines only.
- Touches `tools/install/cli/bin/cli.mjs`, which ships through the OIDC trusted-publishing pipeline in
  `release.yml`. That pipeline's own comments describe the trust wiring as fragile (pinned to
  org+repo+workflow-filename); any change to this file should go through the same release flow and be
  re-tested against the "must mirror slash commands 1:1" intent already documented in the file.
- The `cuo` verb as a stub does not fully satisfy "the CLI and slash commands behave the same" — only
  namespaces it. Flagged explicitly rather than silently scoped out; a real answer requires a separate
  decision on headless execution.
