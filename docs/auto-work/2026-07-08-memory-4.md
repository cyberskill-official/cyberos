## MEM-060 - Fix real product bugs surfaced by the now-runnable brain DB suite -> in_review

Once MEM-059 made the brain DB suite runnable, it exposed pre-existing failures. Two are genuine PRODUCT bugs (not test bugs), both fixed here; both were invisible for as long as the suite couldn't run.

### Bug 1 - rolling summaries never built for subject/channel scopes (F3)

`summarize::scope_event_filter` returned a SQL fragment hardcoding `$3`. That is correct for `uncovered_event_count` (`WHERE tenant_id=$1 AND source_seq>$2 AND <frag>`), but the SAME fragment is reused in `build_and_supersede`'s gather query `WHERE tenant_id=$1 AND <frag>`, which has no `$2` -> the query referenced `$1` and `$3` with only 2 binds -> Postgres "bind message supplies 2 parameters, but prepared statement requires 3". So `build_and_supersede` ALWAYS errored for subject/channel scopes; only the `time_window` scope (no scope bind) ran. Rolling summaries - a core BRAIN feature - never built against a real DB.

Fix (`src/brain/summarize.rs`): `scope_event_filter(scope_kind, scope_ph: u8)` renders the caller's placeholder index; `uncovered_event_count` passes 3, `build_and_supersede` passes 2. Added a unit test pinning the rendered placeholder.

### Bug 2 - the intra-tenant access predicate errored for granted callers

`access_scope::caller_may_see` / `deny_reason` decoded `SELECT 1` (Postgres INT4) as `Option<i64>` (INT8). When a founder/manager grant actually matched, the row decode raised "Rust type i64 is not compatible with SQL type INT4", so `caller_may_see` returned a DB error instead of `true` - i.e. the DEC-2722 security boundary's ALLOW path was broken: a founder or a manager would get a recall error instead of their entitled results.

Fix (`src/brain/access_scope.rs`): decode the three `SELECT 1` probes as `Option<i32>`.

### Test fix - summary version assertions encoded the broken behaviour

`brain_summaries_test.rs` pinned exact summary version numbers (v1; exactly 2 versions), which held only because ingest-time auto-summarisation (`touch_windows`, throttled at 5 events) silently errored (Bug 1). With summaries now building at ingest, the explicit `run_summarize_once` supersedes to a higher version. Relaxed the two brittle pins to assert the real ACs (#4 covered range + exactly one current row; #5 a new event writes a new version, priors retained + superseded, exactly one current = highest) rather than a fixed number. The invariants are unchanged and still fail if superseding breaks.

### Discovered -> filed as MEM-061 (NOT fixed here; out of MEM-060 scope + a distinct subsystem)

`brain_tiering_test` (2 tests) still fails: seeded old events stay `hot` and are never demoted. Root cause: the interaction `emit` path drops `occurred_at_ns`. `emit()` calls `cyberos_audit_chain::emit_genesis_with_op(...)` WITHOUT the event's occurred-at, so `l1_audit_log.ts_ns` is set to INGESTION time (now), not occurrence time. The brain ingest reads `ts_ns` from that column, so every event is "recent" and age-based tiering never fires. This also undermines recency-based recall ranking (future MEM-021). The fix belongs in the emit/audit-chain path (propagate `occurred_at_ns` -> `l1_audit_log.ts_ns`), a distinct subsystem from MEM-060's summarize/access changes. Filed as MEM-061 (P0).
