## MEM-059 - Fix brain_common test harness so the brain DB suite is runnable -> in_review

Discovered while verifying MEM-002: `tests/brain_common.rs` (the shared harness for every DB-backed brain test) could not apply the migrations, so the WHOLE `#[ignore]` brain suite errored at setup and had (silently) never run - despite memory notes claiming these tests were "proven green."

Two harness bugs:
1. `apply_lenient` used `sqlx::query` (extended protocol) on multi-statement migration files -> "cannot insert multiple commands into a prepared statement".
2. Fixing #1 unmasked a race: parallel `#[tokio::test]` threads each call `Env::new()` and apply migrations concurrently, colliding on `pg_proc` (`CREATE OR REPLACE FUNCTION` in 0003/0004/0005) -> "duplicate key value violates unique constraint pg_proc_proname_args_nsp_index".

Fix (`services/memory/tests/brain_common.rs`):
- `apply_lenient` now uses `sqlx::raw_sql` (simple query protocol -> multi-statement OK).
- New `ensure_schema(pool)` applies the full schema set exactly once per test process, guarded by a process-global `tokio::sync::OnceCell`, so the concurrent-apply race cannot happen. `Env::new` calls it instead of the inline apply block.
- Added migration `0009_rls_fail_closed.sql` to the set so the DB tests run against the SHIPPED schema (superuser bypasses RLS, so they still pass; enforcement is proven separately in `brain_rls_test.rs`).

### Discovered by making the suite runnable -> filed as MEM-060 (NOT fixed here)

With the harness fixed, the suite runs and honestly reports pre-existing failures that were invisible while it couldn't run:
- `brain_summaries_test`: 0/3 (all fail)
- `brain_provenance_test`: 1/3 (`provenance_points_back_to_audit_rows` = INT4/i64 ColumnDecode at test line 30; `worker_never_writes_audit_chain` = the summarize bind bug)
- `brain_recall_access_scope_test`: 2/4 (`founder_sees_any_subject_self_sees_own`, `recall_excludes_closest_neighbour_outside_access_scope` - cascade from summaries not building)
- `brain_ingest_test`: 3/3 pass; `brain_rls_test`: pass.

Root cause (single product bug, diagnosed by inspection): `summarize::scope_event_filter` returns a SQL fragment hardcoding `$3` (correct for `uncovered_event_count`'s `WHERE tenant_id=$1 AND source_seq>$2 AND <frag>`), but the SAME fragment is reused in `build_and_supersede`'s gather query `WHERE tenant_id=$1 AND <frag>` which has no `$2` -> placeholders are `$1` and `$3`, only 2 binds supplied -> Postgres "bind message supplies 2 parameters, but prepared statement requires 3". Effect: `build_and_supersede` has ALWAYS errored for subject/channel scopes, so rolling summaries were never built against a real DB (only the `time_window` scope, which binds no scope param, would run). Plus a test-side INT4/i64 decode in `brain_provenance_test.rs:30`.

This is a genuine, significant latent product bug (a core feature - rolling summaries - is broken), kept invisible for as long as the brain DB suite couldn't run. Filed as **MEM-060** (P0, critical, deps MEM-059). It, not the harness, is now the real blocker for MEM-005/006/009 gating on a green brain suite.
