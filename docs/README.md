# `docs/` — FR/NFR specs + documentation tooling

This folder holds the **raw FR and NFR specification files** — workflow deliverables generated by the `feature-request-author` skill. These are inputs to the website build pipeline, not redundant documentation.

**All user-facing documentation lives on the [docs site](https://cyberos-wiki.cyberskill.world/).**

| Folder / File | Purpose | Entry point |
|---|---|---|
| [`feature-requests/`](feature-requests/) | ~268 FRs across 26 domains. Each FR has a spec + audit companion. Authored via `feature-request-author` skill. | [`BACKLOG.md`](feature-requests/BACKLOG.md) |
| [`non-functional-requirements/`](non-functional-requirements/) | ~157 NFRs across 18 domains. Each NFR has a spec + audit companion. | `README.md` |

### What's already on the website

The website build pipeline reads FR/NFR frontmatter and generates:

| Source | Build script | Website page |
|---|---|---|
| `feature-requests/` frontmatter | `data-extract.mjs` → `render-fr-catalog.mjs` | [`reference/fr-catalog.html`](../website/docs/reference/fr-catalog.html) |
| `non-functional-requirements/` frontmatter | `nfr-extract.mjs` → `render-nfr-catalog.mjs` | [`reference/nfr-catalog.html`](../website/docs/reference/nfr-catalog.html) |
| `knowledge/VN_GLOSSARY.md` | manual | [`reference/glossary.html`](../website/docs/reference/glossary.html) |
| Risk data | manual | [`reference/risk-register.html`](../website/docs/reference/risk-register.html) |

### Why FR/NFR specs stay as raw markdown

These files are **workflow deliverables**, not documentation. The `feature-request-author` skill generates them, the `feature-request-audit` skill scores them, and the website build pipeline reads their frontmatter to render catalog pages. Deleting them would break the workflow pipeline.

**After creating or modifying FRs or NFRs**, regenerate the website catalog:

```bash
./website/build/build.sh        # full build (FR + NFR catalogs)
./website/build/build.sh --fr   # FR catalog only
./website/build/build.sh --nfr  # NFR catalog only
```

## How the layers relate

```
SDP ── normative ──▶ FRs ── authority ──▶ Skill catalog ── compose ──▶ CUO workflows
                                                                            │
                                                                            ▼
                                                                      memory (memory module)
                                                                            │
                                                                            ▼
                                                                   Implementation modules
                                                                   (ai-gateway, auth, mcp, …)
```

- **SDP** — defines the 14 stages every deliverable flows through (see [CUO appendices](https://cyberos-wiki.cyberskill.world/modules/cuo/appendices.html)).
- **C-Suite Reference** — defines the 48 personas + the 9-block schema each persona spec must render (see [CUO appendices](https://cyberos-wiki.cyberskill.world/modules/cuo/appendices.html)).
- **FRs** (`feature-requests/`) capture every concrete change request, tagged by phase + module.
- **Skill catalog** (`modules/skill/`) ships 104 author+audit pairs that materialise SDP stages into agentic Skills.
- **CUO workflows** (`modules/cuo/`) chain Skills into persona-owned deliverables (223 workflows live).
- **memory** (`modules/memory/`) records every chain decision in an append-only audit chain.
- **Implementation modules** (planned in FRs) are the runtime services (services/auth, services/ai-gateway, etc.) that satisfy the FRs.

## Naming convention

| Convention | Filename | Purpose |
|---|---|---|
| Top-level folder index | `README.md` | What's in this folder |
| Module spec (Layer-1) | `AGENTS.md` | Normative protocol — currently only the memory module has one |
| Skill manifest | `SKILL.md` | Skill body + frontmatter (Anthropic Agent Skills standard) |
| Audit rubric | `RUBRIC.md` | Per-skill rubric for the audit-loop |
| Contract schema | `CONTRACT.md` | Versioned artefact schema (e.g. `feature_request@1`) |
| FR markdown | `FR-{MOD}-{NNN}-{slug}.md` | One FR per file (e.g. `FR-AUTH-001-magic-link.md`) |
| Per-folder log | `CHANGELOG.md` | Newest-first release history |

## Roadmap reference

CyberOS ships in five gated phases — P0 (Foundation) → P1 (Productivity) → P2 (Operations) → P3 (SaaS-ready) → P4 (Client-facing GA). The phase-by-phase milestone arc is documented on the docs site at [`../website/docs/architecture/milestones.html`](../website/docs/architecture/milestones.html). All work in `feature-requests/` is tagged by phase.

## Docx ↔ markdown utility

If you need to convert any future Markdown to/from docx (e.g. for sharing with non-technical reviewers):

```bash
cd docs
make docx INPUT=path/to/file.md
make md   INPUT=path/to/file.docx
```

Requires `pandoc` (`brew install pandoc` / `apt-get install pandoc`). See [`Makefile`](Makefile) for the exact commands.
