DEC-095 — Sync-class boundary: four classes, scope defaults, identity model

Decision

Every memory carries a `sync_class` (frontmatter §5.1, 28th field) declaring whether it leaves the local machine and where it goes. Four classes: `local-only`, `publishable`, `shared`, `client-visible`. Defaults are scope-driven (§17.2 table); per-file overrides allowed. Subject sovereignty: subjects may always downgrade their own writes to `local-only`; agents may not upgrade above the scope default without explicit chat instruction.

Identity is **per-subject**, not per-machine. A subject (e.g., `subject:stephen-cheng`) may operate from multiple machines; each has its own `.cyberos-memory/` and its own linear audit chain. The org BRAIN re-chains incoming memories per §11.6 (preserving `original_chain` for traceability).

Offboarding follows the **absorb-then-discard** pattern: published `shared` memories stay (org property by virtue of having been published); personal `member/<id>/` and `local-only` content is garbage-collected from the org BRAIN's mirror; the leaving employee retains their personal BRAIN as a portable export.

Why

The pre-A protocol had:

  • An obvious org-vs-personal scope structure (`company/`, `member/<id>/`, `client/<id>/`, `module/<name>/`) but no rule classifying which scopes flow to a shared store.
  • A direct contradiction between §11.8 ("Concurrent multi-machine editing of the same project is unsupported; pick one authoritative machine") and FACT-004 ("CRDT sync across machines"). Both lived in the system; an agent reading both got contradictory signals.
  • No identity model for cross-actor flow. `signed_by: ed25519` was reserved in §5.1 but never used. The moment a second laptop received a memory claiming "Stephen authored this," there was no way to verify cryptographically.
  • No explicit publish/subscribe boundary. Every memory written between the start of multi-person work and now would have been classified retroactively at significant migration cost.

    • How

    §17 declares four sync classes and per-scope defaults. §11.8 rewritten to clarify "this protocol governs the personal layer; continuous multi-machine sync of shared scopes happens through the runtime BRAIN module." §5.1 frontmatter extended with `sync_class`. §14 end-of-response block extended with sync-class-summary line.

    §17.6 explicitly bounds what this protocol does NOT define: the wire protocol of `brain.publish`/`brain.pull` (BRAIN module's domain), ACL enforcement on PORTAL surfaces (PORTAL module's domain), conflict resolution between two subjects' concurrent edits (uses §9.1 with org BRAIN as eventual-consistency arbiter), cryptographic key rotation (separate `meta/key-policy.md`).

    Scope defaults

    `local-only`: `meta/`, `audit/`, `index/`, `exports/`, `conflicts/`, `.lock`, `member/<self>/private/`, `memories/drift/`.

    `publishable`: `member/<self>/` (non-private), `memories/preferences/`, `memories/refinements/`, `memories/decisions/`, `memories/facts/`, `memories/people/`, `memories/projects/`.

    `shared`: `project/`, `company/`, `module/<name>/`, `client/<id>/` (internal), `persona/<role>/`.

    `client-visible`: `client/<id>/portal-visible/`.

    `project/` defaulting to `shared` reflects the answer to "does CyberOS architecture flow to every employee?" — yes, this project IS the company's product. Different projects may override.

    Trade-offs accepted

  • **Per-subject identity** (rather than per-machine): one logical BRAIN per person across all their devices. Cleaner mental model; requires the BRAIN module to handle the mirror-per-subject sync. Audit chains stay per-store-linear; org BRAIN re-chains.
  • **Subject sovereignty over personal scope**: agents do not contest a subject's edits to `member/<their-own-id>/`. Risk: subject can hide a memory from the org. Mitigation: anything `publishable` is what a subject *chose* to share; if they didn't share, that's their right.
  • **Absorb-then-discard offboarding**: keeps the org's institutional knowledge while respecting the leaver's personal sovereignty. Risk: a leaver who feels wronged could pre-emptively tombstone their `publishable` memories before publishing. Acceptance: published copies on the org BRAIN are independently authoritative; locally-tombstoning a `shared` memory does not propagate up.
  • **Mechanism deferred to BRAIN module P1**: until then, `sync_class` is metadata-only. Risk: misuse during the metadata-only window. Mitigation: defaults are conservative (most personal scopes default to `publishable`/`local-only`, not `shared`).

    • Closes

  • §11.8 ↔ FACT-004 contradiction (FACT-004 updated separately to cross-link §17).

    • How to use this memory

    When designing a new memory bucket, declare its scope and let §17.2 pick the default sync_class. When in doubt about whether a fact is personal or org, ask: would a teammate joining tomorrow benefit from seeing this? If yes, `publishable` or `shared`. If no, `local-only`. When the BRAIN module ships at P1, the `sync_class` field starts to load-bear; before then, get the defaults right so migration is automatic.
