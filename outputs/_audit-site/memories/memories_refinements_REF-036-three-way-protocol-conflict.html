REF-036 — Three-way protocol-conflict handling (§0.5 + §13.0)

Real-world trigger

User question (Stephen, 2026-05-06): *"did we take care of the case when local BRAIN conflict with upstream BRAIN when update?"*

Honest diagnosis: the post-cascade §0.5 mechanism handled scenarios A (local hand-edit; 2-way mismatch fires §13.0 tripwire) and B (clean upstream upgrade). It did NOT handle scenario C (loaded ≠ pinned ≠ upstream): the agent would have silently overwritten local hand-edits with upstream pull. Scenarios D (multi-actor protocol-version skew), E (upstream rollback), F (upstream tightens validators), G (key rotation operational details) were also unhandled but lower-priority.

Refinement adopted

TIER 1 (three-way conflict detection + structured user-choice prompt + §13.0 classifier row). TIER 2 (cross-actor skew) and TIER 3 (key rotation flow) deferred — both gain operational relevance only when the BRAIN module's network surface ships at P1.

What changed in AGENTS.md

  • §0.5 gained a "Three-way conflict (loaded ≠ pinned ≠ upstream)" subsection (between the signed-upstream-releases subsection and the bootstrap-behaviour subsection): defines detection condition, structured prompt with three options, no-auto-merge rule, freeze-write linkage to §13.0.
  • §13.0 state-classifier table gained a new row: `INCOMPATIBLE:three-way-protocol-conflict`. Same freeze-write handling as `protocol-sha256-mismatch`.

    • Why this is the right fit

  • Extends the existing conservative §13.0 discipline (writes-frozen-until-explicit-resolution) from 2-way to 3-way without inventing new mechanisms.
  • The three explicit options map cleanly onto the existing §0.5 vocabulary (revert; approve via phrase; manual merge then approve via phrase).
  • No automated merge: matches the protocol's broader aversion to silent automated decisions on contested state.

    • Trade-offs

  • Friction during 3-way conflicts (user MUST choose). Acceptance: silent overwrite of hand-edits is worse than friction.
  • No detection without an upstream-release-list endpoint. Pre-P1 this is moot (no upstream check happens); post-P1 this becomes the canonical path.

    • How to use this memory

    When the BRAIN module's `protocol.releases.list` endpoint ships, ensure §0.5's three-way detection is wired into the upgrade-check flow, NOT after the user has approved (otherwise approval would race against three-way state). When testing the upgrade flow end-to-end at P1, include a 3-way scenario in the test matrix.
