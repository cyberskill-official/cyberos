DEC-093 — Manual fine-tune playbook (registry v0.2.0)

A human-driven, structured 7-step cycle for skill improvement. Complements DEC-092's auto-refinement: auto catches mechanical drift; manual fine-tune handles patterns that need human judgement. Frontmatter `human_fine_tune:` block declares who can fine-tune, what triggers initiation, what artefacts are required, and what windows are blackouts.

What landed in frontmatter

human_fine_tune:
  fine_tuner_role: cpo                # who has commit access
  review_required:
    on_minor_bump:    false
    on_major_bump:    true            # owner_role + registry maintainer
    on_safety_change: true            # cuo-cseco + cuo-clo
    # auditor skills add: on_rubric_rule_added/removed
  signals_to_initiate:
    - acceptance_rate_below: 0.6
    - hitl_pause_rate_above:  0.4
    - drift_signal_count_above: 3
    - user_complaint_received
    - regulator_inquiry_received
    - self_audit_refinement_proposal_count_above: 2
  procedure_ref: null                  # null = registry README Part 7 default
  required_artifacts:
    - changelog_entry
    - acceptance_test_added
    - memory_refinement_entry
  blackout_windows: []                 # ISO date ranges; edits frozen

The 7-step playbook (registry README Part 7)

1. **Pause** — set skill to maintenance mode; routes return `E_SKILL_MAINTENANCE`.

2. **Diagnose** — read `genie.action_log` for the affected window; cluster failures by mode.

3. **Add regression** — write golden-input + golden-output for each distinct failure mode; land under `acceptance/`.

4. **Edit** — `SKILL.md` body / `INVARIANTS.md` / `RUBRIC.md` (for auditor skills); aim: pass new regressions without breaking old.

5. **Re-run acceptance suite** — old + new golden cases; ALL must pass.

6. **Bump + log** — PATCH if no behaviour change; MINOR if new invariant; MAJOR if envelope changes. CHANGELOG entry with `### Driver` citing the trigger.

7. **Resume** — set skill to active; monitor acceptance over next 24-48h.

Required artefacts (validator-enforced)

Every fine-tune commit MUST include:

  • **`changelog_entry`** — Keep-a-Changelog 1.1.0 with `### Driver` section.
  • **`acceptance_test_added`** — at least one new regression in `acceptance/regression-<YYYY-MM-DD>-<slug>.md`.
  • **`memory_refinement_entry`** — append to `memories/refinements/REF-NNN-<slug>.md` in the BRAIN per AGENTS.md §10 step 6. Future agents read this.

    • Optional but recommended: `rubric_rule_diff` (auditor skills), `drift_record` (when a drift signal triggered the cycle).

    Blackout windows

    Any ISO date range listed in `blackout_windows` freezes edits. Useful for audit weeks, production launches, holiday weeks. The validator rejects any commit touching the skill during a blackout with `op:rejected reason:blackout-window-active`.

    Why this is structured (not freeform)

    Skills look like markdown that anyone can edit, technically true, operationally false. A skill in production has acceptance fixtures, drift telemetry, downstream consumers, and partner exposures. Editing it without the playbook breaks trust calibration and audit chain. The 7-step playbook is the minimum discipline to keep skills evolvable without becoming brittle.

    Auto-refinement vs. manual fine-tune

    | Auto-refinement (DEC-092) | Manual fine-tune (this DEC) |

    | --- | --- |

    | Triggered by invariant breach or anomaly threshold | Triggered by acceptance / HITL / drift signals OR auto-refinement escalation |

    | Skill emits `refinement_proposal` envelope; supervisor pauses | Human runs the 7-step playbook |

    | Single-diff change, ≤1 file, machine-readable | Multi-file, may add invariants / rubric rules / new acceptance cases |

    | PATCH bump | PATCH / MINOR / MAJOR depending on scope |

    | Resume from LangGraph checkpoint | Re-deploy after re-run-suite passes |

    How to use this memory

    When considering a fine-tune for a skill:

  • First check `signals_to_initiate` — has any threshold tripped? If not, you're doing a routine cycle (typically every 2 weeks during the first month, monthly thereafter for production skills).
  • Check `blackout_windows` — are we inside one? If yes, defer.
  • Check `review_required` — what gates apply for this bump kind? MAJOR + safety = always two reviewers.
  • Run the 7-step playbook in registry README Part 7. Don't skip steps. The required_artifacts validation runs at commit time.
