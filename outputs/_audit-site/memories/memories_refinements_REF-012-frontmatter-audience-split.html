REF-012 — Split SKILL.md frontmatter contract by audience (registry README Part 2.2)

What changed

Registry README v0.2.0 Part 2.2 now organises the 33 frontmatter fields into three named groups by audience:

  • **§3.1 Portable Anthropic-skill fields** — `name`, `description`, `skill_version`. Load anywhere a SKILL.md is read (Claude Code, Antigravity, vanilla Anthropic skill loaders).
  • **§3.2 CyberOS runtime extensions** — `persona`, `owner_role`, `allowed_brain_scopes`, `allowed_mcp_tools`, `escalation`, `expects/produces`, `audit`, `confidence_band`, `untrusted_inputs`, `determinism`, `emitted_source_freshness_tier`, `gated_until_phase`. Require the CyberOS runtime to enforce.
  • **§3.3 v0.2.0 portability + governance** — `invocation_modes`, `depends_on_contracts`, `exposable_as`, `self_audit`, `human_fine_tune`, the new `expects.standalone_interview_ref`, the new `produces.human_summary_ref`.

    • Why

    Without the split, a future reader (or the user, six months from now) hits the question "which of these fields will Antigravity honor?" and has to manually classify. The split makes it machine-readable and human-obvious: only §3.1 fields are guaranteed to load on a non-CyberOS host; §3.2 are silently ignored; §3.3 are the new v0.2.0 governance/portability surface.

    Trigger

    User asked in conversation: *"if i copy both skills folder into .gemini/antigravity/skills, will they work? or they only work with Claude structure"*. The honest answer required exactly this audience distinction. The README §1 had previously claimed "skills are standalone" without making the runtime-vs-format distinction explicit. That ambiguity caused the question; the refinement prevents future readers from hitting the same confusion.

    Tier

    **Tier 1** — directly prevents the observed failure (next reader would ask the same question). Adopted in the same response, as part of registry v0.2.0.

    How to apply

    When authoring or reviewing skills documentation:

  • Any reference to "the frontmatter contract" should specify which group (§3.1 / §3.2 / §3.3) is meant.
  • New fields go into one of the three groups explicitly. If a proposed field doesn't fit any of them, that's a smell — most likely it belongs in `_contracts/` or in a side-channel like `INVARIANTS.md`, not in the SKILL.md frontmatter.
  • The CCSM-to-anthropic transpiler reads ONLY §3.1 fields. The CCSM-to-mcp-tool transpiler reads §3.1 + relevant §3.3 fields (mostly `expects/produces`). The CyberOS-native runtime reads all three.
