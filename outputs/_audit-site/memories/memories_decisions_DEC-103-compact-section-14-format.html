DEC-103 — Compact §14 format gated by operational_mode

Decision

§14 splits into two formats: **§14.1 compact** (default when `manifest.operational_mode: normal`) shows only changed paths in a `Δ Changes:` block plus a single `unchanged:` roll-up line plus a Status block; **§14.2 full** (used when mode is `verbose | debug | maintenance`) retains the pre-Bundle-I per-scope-explicit format. The audit ledger stays authoritative; format changes don't affect chain integrity. Coverage stat for ingestion ops (§14.3) is mandatory in both formats.

Why

The pre-I §14 block was 14+ lines verbatim, of which ~9 typically read "no change". The user surfaced real readability friction during a routine review. Signal-to-noise was poor for daily use; full format only makes sense during protocol-development sessions where every gap matters.

The `operational_mode` field already exists (Bundle C). Reusing it as the format discriminator avoids inventing new mechanism. Default `normal` gets compact rendering; protocol developers and debuggers use `verbose`/`debug` to get the full format.

How

§14 rewritten with two-format spec:

  • §14 opening paragraph declares the split and points at `operational_mode`.
  • §14.1 specifies compact format (3-block layout: `Δ Changes` / `Status` / `unchanged`).
  • §14.2 specifies full format (pre-I per-scope-explicit; with `maintenance` banner addition).
  • §14.3 coverage stat (renumbered from prose paragraph; unchanged content).
  • §14.4 authority clarifier (audit ledger is authoritative; format is human-summary only).

    • Trade-offs accepted

  • **Two formats means future agents must support both at parse time.** Acceptance: the §14 block is for HUMAN consumption, not chain-recovery; future agents primarily read the audit ledger. The block is a courtesy summary.
  • **Compact `unchanged:` roll-up loses per-file granularity.** Acceptance: full format is one toggle away (`operational_mode: verbose`); analysis-only turns where details matter are infrequent.
  • **Bullet/marker variation across modes.** Acceptance: clarity over consistency; readability gain outweighs.

    • What this does NOT change

  • Audit ledger format and chain semantics — unchanged.
  • §14 mandatory status (still required after every substantive reply).
  • Coverage stat for ingestion ops (still mandatory; just renumbered §14.3).
  • Per-mode behaviour outside §14 (DEBUG banners, MAINTENANCE permissions unchanged).

    • How to use this memory

    When rendering §14, check `manifest.operational_mode`. Choose §14.1 (compact) for `normal`; §14.2 (full) for the other three modes. Always include the §14.3 coverage suffix on ingestion ops. When designing future protocol changes that affect §14 content, update both §14.1 and §14.2 examples to keep them in sync.
