DEC-102 — Strict uppercase BRAIN alias (§0.3 case-sensitivity)

Decision

§0.3's BRAIN-alias activation requires literal uppercase B-R-A-I-N. Lowercase 'brain' (in any of "the brain", "human brain", "brain freeze", "what does the brain do", etc.) does NOT trigger memory-store interpretation; it's plain language about anatomy, metaphor, or general topic. When context strongly implies memory-store but casing is lowercase, the agent surfaces the ambiguity and asks a clarifying question rather than silently assuming.

Why

The pre-H §0.3 wording — *"the BRAIN" or "your BRAIN"* — used implied capitals but didn't enforce them. A literal reader (or a too-eager agent) could have matched lowercase "brain" too. Examples of false-positive risk:

  • *"What does the brain do during sleep?"* — anatomy question; should not trigger `view`-walk of `.cyberos-memory/`.
  • *"I had a brain freeze"* — metaphor; not a memory-store reference.
  • *"Brain training apps"* — product topic; not a memory-store reference.
  • *"Right side of the brain"* — anatomy; not a memory-store reference.

    • Stephen noticed this gap during a casual review post-Bundle-G and asked for the strict version. The fix is a single-paragraph amendment with no other knock-on effects.

    How

    §0.3 first paragraph amended with explicit case-sensitivity clause. New "Lowercase 'brain' is normal language" clarifier paragraph added before the existing examples list. Includes an ambiguity-disambiguation rule: lowercase + memory-context → clarifying question; never silent assumption.

    Trade-offs accepted

  • **Users won't naturally type ALL CAPS in casual conversation.** Acceptance: the BRAIN alias is a deliberate-action signal (like "@channel" or "/command"); the casing is the signal.
  • **Marginally more friction for power users.** Acceptance: the existing §0.3 examples ("save to the BRAIN", "what's in the BRAIN?") are short and easy to capitalise; the clarifying question for ambiguous lowercase is one-line friction worth the false-positive prevention.

    • What this does NOT change

  • **§1 step 2 diagnostic-verb list** (Bundle G) — keeps lowercase "check brain", "show brain", "view brain". Those trigger `PRISTINE-DIAGNOSTIC-HOLD` based on verb intent, NOT BRAIN-alias activation. Independent mechanisms.
  • **Prose readability** — written documentation elsewhere in the protocol can use either case for natural reading. The case-sensitivity rule applies only at the alias-activation boundary.

    • How to use this memory

    When the user types "the BRAIN", "your BRAIN", or any uppercase form, treat as memory-store reference per §0.3 examples. When the user types lowercase "brain", treat as plain language; if context is ambiguous (memory-discussion in surrounding turns), ask "Did you mean 'BRAIN' (the memory store)?" before acting. Never silently assume the alias from lowercase.
