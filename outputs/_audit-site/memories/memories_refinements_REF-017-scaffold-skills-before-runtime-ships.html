REF-017 — Scaffold the next skill in a chain before the runtime ships; gate it with `gated_until_phase`

When designing a multi-skill pipeline (e.g. `cuo/cpo/fr-create` → `cuo/cpo/fr-audit` → `cuo/cto/fr-to-tech-spec`), there is real value in scaffolding the downstream skills BEFORE the runtime that would execute them ships. The scaffold IS a contract — it documents what the future runtime must satisfy, what envelopes the future supervisor will route, what invariants must hold, and what acceptance fixtures the harness will need. The cost of writing the scaffold is low (~30 minutes per skill) and the benefit is high: the chain shape becomes legible immediately, contract dependencies become explicit, and the team can review + iterate on the skill's design without the runtime as a forcing function.

How to use this memory

When proposing a new skill that's part of a chain whose runtime isn't ready yet, default to scaffolding it now. Use these rules:

1. **Set `gated_until_phase: <runtime_milestone>` in frontmatter.** The supervisor + validator will refuse to route to this skill until the gate clears. The skill exists as documentation, not as routable code.

2. **Author the full v0.2.0 frontmatter contract.** All 33 fields. INVARIANTS.md, envelopes, STANDALONE_INTERVIEW.md, HUMAN_SUMMARY.md, acceptance/README.md. Don't half-scaffold — the contract loses value if it's incomplete.

3. **Name the things the scaffold deliberately omits in CHANGELOG.md.** Reference docs (HITL_PROTOCOL.md, etc.), the artefact's contract (e.g. `tech_spec@1`), the worked PIPELINE.md example. These land at v0.2.0 with the runtime; the scaffold's CHANGELOG should call them out as intentional omissions so a future reader doesn't think they were forgotten.

4. **Declare contract dependencies via `depends_on_contracts:` even if no consumer code exists yet.** This makes the dependency graph machine-readable from day one and exercises the validator's pin-path check.

5. **Add the new skill to the registry README's Part 23.1 index with explicit `(scaffold)` annotation.** The lifecycle state diagram in Part 14 implicitly accepts a "scaffold" state below v0.1.0 / Adolescent.

Why it matters

The pre-existing pattern was "skills appear when they can run." The cost of that pattern: when the runtime ships, skill design happens under deadline pressure, and the team often discovers contract gaps mid-implementation. Scaffolding-first reverses the pressure — by the time the runtime lands, the skill's contract is already battle-tested through readings and reviews.

This is the same logic that already applies to contracts (per DEC-090): they exist before all consumers do. Scaffolds extend the principle from contracts to skills.
