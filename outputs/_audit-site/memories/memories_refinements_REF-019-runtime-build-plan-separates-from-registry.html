REF-019 — The runtime build plan is separate from the registry; lifecycles differ; mixing them is wrong

When CyberOS reached the point where 7 skills + 5 contracts were scaffolded but no runtime existed, the natural temptation was to either (a) start writing runtime code under `cyberos/docs/skills/` because that's where the design lives, or (b) inflate `cyberos/docs/skills/README.md` Part 12 (runtime architecture) into a build plan with task breakdowns, sequencing, definitions-of-done. Both are wrong.

What the right separation looks like

**`cyberos/docs/skills/` + `cyberos/docs/contracts/`** = the SPEC. Stable. Versioned. Each skill carries `gated_until_phase: runtime_v0_3_0` until the runtime ships. Updates are CHANGELOG events with discrete impact.

**`cyberos/runtime/`** = the BUILD PLAN + eventually the implementation. Different lifecycle: phases land over weeks; engineering picks pieces in parallel; refactoring is normal. PLAN.md, INTERFACES.md, BUILD_ORDER.md sit in this folder as the engineering hand-off. When phase J (acceptance harness) turns green and skills run end-to-end, the folder gets archived and replaced with a thin operations README pointing at the actual code.

**The key principle:** the registry is what the runtime MUST satisfy. The runtime folder is HOW it gets satisfied. If they ever conflict, the registry wins; the runtime updates first, the registry updates afterward only if the contract genuinely needs to change.

How to use this memory

When designing a system that has both a contract layer and an implementation layer:

1. **Put the contract layer in its own namespace.** Make it small, stable, versioned with care. Every change is a CHANGELOG entry.

2. **Put the implementation layer in its own namespace.** Allow chaos here — multiple iterations per phase, refactoring without ceremony, lessons-learned notes.

3. **Forbid the implementation from updating the contract.** Contract changes go through the contract's own CHANGELOG first; implementation follows.

4. **Forbid the contract from describing the implementation.** The contract says "what the runtime MUST satisfy"; details of HOW are out-of-scope.

5. **Mark every contract item with a "ready-for-implementation" gate** (e.g. `gated_until_phase: runtime_v0_3_0`). The gate's removal is the moment the contract crosses into "the runtime is there to back this up."

This separation scales. CyberOS will eventually have dozens of skills, multiple contract namespaces, hundreds of acceptance fixtures, and an active runtime folder. Mixing them collapses into a mess where every change touches everything.

When to revisit

If the runtime build runs into a wall because the contract is wrong (e.g., a skill's interface is impractical to implement), the right response is:

1. Pause the runtime build for that skill.

2. File a registry CHANGELOG entry proposing the contract change.

3. Get the change approved (per `human_fine_tune.review_required` gates in the relevant SKILL.md).

4. Update the contract.

5. Resume the runtime build.

NOT: silently change the contract while implementing. NOT: implement the wrong thing because the contract said so. Bilateral updates with explicit ceremony.
