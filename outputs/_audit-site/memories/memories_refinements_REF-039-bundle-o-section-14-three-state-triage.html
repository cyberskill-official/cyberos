REF-039 — Bundle O: §14 three-state triage (silent / files-only-compact / issues-verbose)

Trigger

User feedback during cowork-session 2026-05-10, immediately after Bundle N (TIER 1+2) landed. Stephen reviewed the post-Bundle-N §14 output and found three more axes to compress:

1. *"Status: unchanged section seem not necessary since there is 'Δ Changes' section"* — `Status:` block + `unchanged:` line are redundant signal.

2. *"In normal mode no need to should Δ Changes if no issues arise too"* — `Δ Changes:` itself is redundant given the parallel `📁 Files changed:` block.

3. *"only show Files changed (no need to implied outside BRAIN), only turn on maintenance mode and show full memory verbose (arrange them smartly too) status when issues arise"* — single merged file list, smart re-arrangement, auto-trigger verbose on issues.

Plus an open question on token tracking. All four feedback points are §0.4 standing-rule triggers.

Adoption

Adopted via §0.5 chat-turn approval (2026-05-10):

> approve protocol upgrade to sha256:b0d9ad3adc35ec1b74bad1407532873db828adc5161d7f05e23914e76096c1d6

Tier choice: **TIER 1+2+3 in one Bundle** (full implementation in a single upgrade because the changes are entangled — you can't reduce §14.1 to `📁 Files changed:` only without also defining where the audit/Status detail goes, which means defining §14.2's auto-trigger and arrangement).

Changes

  • **§14 heading**: `(conditional in normal mode)` → `(silent by default; verbose when issues)` — emphasises the trigger semantics rather than the mode dependency.
  • **3-state triage table** added at top of §14 — explicit decision matrix (omit / compact / verbose) replaces implicit per-condition logic.
  • **§14.1 compact rewritten** — now contains ONLY a `📁 Files changed:` block (BRAIN + non-BRAIN paths merged, no qualifier) plus optional `Tokens:` line. Removed: `Δ Changes:` heading, `Status:` block (with all 4 sub-lines), `unchanged:` line, `audit/<YYYY-MM>.jsonl: …` line.
  • **§14.2 verbose trigger broadened** — now fires on ANY of: `op:rejected|revert|warn|health_check` this turn; latest §8.7 reports CRITICAL or WARN; `operational_mode != normal`. Pre-Bundle-O the trigger was mode-only.
  • **§14.2 arrangement** smartly ordered — `⚠️ Findings:` block first (head of output for actionability), then `📁 Files changed:` (mirroring §14.1), then `Δ Changes (BRAIN detail):` for the audit-curious, then `Status:` block, then optional `Tokens:`.
  • **`unchanged:` line removed** entirely from §14 — the implicit signal (absence-from-list) replaces the explicit roll-up.
  • **`Tokens:` slot reserved** in both §14.1 and §14.2 — emitted only when a runtime token counter is wired up via MCP tool exposing `usage.input_tokens`/`usage.output_tokens`. Approximation via `tiktoken` or character count is forbidden because misleading.
  • **§14.3 (coverage stat)** unchanged in semantic content; cross-reference text updated to refer to §14.1 and §14.2 individually.

    • Why this works

    §14 now has three precise output volumes corresponding to three precise situations: (1) nothing happened that needs surfacing, (2) routine mutations the user already approved, (3) something needs attention. Each transition is automatic — no operator action required to escalate from compact to verbose when a finding arises. The trade-off compared to Bundle N: agents must track per-turn finding counts cheaply (single integer), and the audit-curious user must read the audit ledger via tools rather than getting the head SHA in chat. Both trade-offs were explicitly accepted by Stephen in the approval.

    How to use this memory

    The next agent reading this BRAIN must:

    1. Maintain a per-turn `finding_count` integer (incremented on `rejected|revert|warn|health_check` ops).

    2. At end of turn, evaluate the 3-state triage in DEC-107.

    3. When emitting §14.2, lead with `⚠️ Findings:` only when findings exist; omit the block in pure-mode-trigger cases (e.g., `verbose` mode with no findings).

    4. Never emit a `Tokens:` line until a runtime token counter is configured. The protocol slot exists for forward-compat; using it without backing data is misleading.

    5. When non-BRAIN files are modified in a routine turn, the §14.1 `📁 Files changed:` block IS the only output — no other sections appear.
