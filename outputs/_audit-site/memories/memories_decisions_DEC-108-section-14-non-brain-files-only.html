DEC-108 — §14 `📁 Files changed:` shows non-BRAIN paths only (Bundle P correction to Bundle O)

Decision

The §14 `📁 Files changed:` block lists **non-BRAIN paths ONLY** — always — in both §14.1 compact and §14.2 verbose formats. BRAIN-internal mutations (memory writes, audit rows, manifest updates, health reports, protocol-history archives) are agent housekeeping and surface only in §14.2's `Δ Changes (BRAIN detail):` block when verbose is otherwise triggered. Three structural changes to §14:

1. **`📁 Files changed:` narrowed to non-BRAIN paths only** in both §14.1 and §14.2. Bundle O's wording ("merged list, no qualifier") was a misread of user feedback — corrected here.

2. **§14.0 omission condition (c) updated**: BRAIN-only mutation turns no longer trigger §14.1. The condition now reads "no non-BRAIN file was modified this turn" rather than "no mutations". A turn that ONLY writes a DEC/REF/preference produces no §14 output.

3. **`Δ Changes (BRAIN detail):` always present in §14.2.** It is now the sole place BRAIN-internal mutations surface in chat. §14.1 never shows BRAIN paths.

Why

User correction during cowork-session 2026-05-10, immediately after Bundle O landed:

> "no need to implied outside BRAIN" i mean only show changes outside the brain, no need to show inside BRAIN changes

Bundle O interpreted this as "merge into one list, no qualifier"; Stephen meant "show only outside-BRAIN changes — drop the BRAIN paths entirely from §14.1". The semantic difference matters: under Bundle O, every BRAIN mutation generated a §14.1 line; under Bundle P, BRAIN mutations are silent unless something else escalates the output to §14.2.

Why this is the right framing:

  • BRAIN housekeeping (writing memories, appending audit rows, refreshing manifest checkpoints) is the agent's job. The user does not need to track it on every turn.
  • The audit ledger is the authoritative record (§7); the user can run `runtime/tools/cyberos_validate.py` or the on-demand healthcheck phrase to inspect BRAIN state when interested.
  • Issues automatically promote output to §14.2 verbose, which DOES surface BRAIN detail under its own section. The compact format is only for the routine case where the user actively wants to know "what did you touch in MY project?".

    Trade-offs

  • **Pro**: routine BRAIN-only-mutation turns are now silent. Writing a DEC + REF + preference triple on a healthy chain produces zero §14 output. Significant noise reduction for protocol-development sessions.
  • **Pro**: clean conceptual separation — `📁` always means "your project files"; BRAIN housekeeping has its own section under verbose only.
  • **Pro**: §14.1 stays minimal (single block, single purpose).
  • **Con**: a user who actively wants to see BRAIN writes on every turn must flip `operational_mode` to `verbose` or run the on-demand healthcheck. Mitigation: most users prefer to be told only when something needs attention, and verbose mode is one config change away.
  • **Con**: agents that don't carefully classify "BRAIN path vs non-BRAIN path" may incorrectly include BRAIN entries under `📁`. Mitigation: classification is mechanical (`.cyberos-memory/...` prefix); §8.7 self-audit catches drift.

    1. Classify each path mutated this turn as **BRAIN** (path starts with `.cyberos-memory/`) or **non-BRAIN** (everything else).

    2. Evaluate §14.0 conditions: omit if normal mode + no findings + no NON-BRAIN file changes + clean self-audit. (BRAIN mutations alone do NOT trigger §14.1.)

    3. If §14.0 doesn't apply: choose §14.1 (non-BRAIN changes, no issues) or §14.2 (issues OR mode != normal) per the trigger table.

    5. In §14.2, list non-BRAIN paths under `📁 Files changed:` (omit block if none) AND BRAIN paths under `Δ Changes (BRAIN detail):` (always present in §14.2).

    When in doubt about whether a path is BRAIN or non-BRAIN, the rule is mechanical: prefix-match `.cyberos-memory/`. No exceptions.
