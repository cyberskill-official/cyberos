REF-013 — Declare cross-skill dependencies in frontmatter (`depends_on_contracts:`)

What changed

New frontmatter field `depends_on_contracts:` — a list pinning the contract folders + major versions a skill consumes. Validators reject skill bodies that reference contract paths not declared in this list.

depends_on_contracts:
  - id:        feature-request
    version:   v1
    purpose:   generation_skeleton
    pin_path:  cyberos/docs/contracts/feature-request/v1/

Why

Before this, fr-create and fr-audit hard-coded the path `cuo/_shared/feature-request-template/template.md` in their bodies. A naive `cp -r cuo/cpo/fr-create/` to another instance broke silently — the template wasn't co-located. Bumping the template to `feature_request@2` had no machine-readable way to force consumers to acknowledge the bump.

`depends_on_contracts:` makes the dependency explicit, machine- readable, version-pinned, and bundleable. The build pipeline can ship skill + declared contracts as one zip. The validator can warn on undeclared body references.

Trigger

User asked: *"both skills have outside reference to cuo/_shared/feature-request-template. Is there any solution for this?"* The previous architecture conflated two questions: (1) where should shared schemas live? (2) how should consumers declare they need them? The answer to (1) is DEC-090 (move to `_contracts/`); the answer to (2) is this REF (declare via frontmatter). Both landed together in registry v0.2.0.

Tier

**Tier 1** — directly prevents broken extractions and silent schema-drift between consumer and contract. Adopted in same response as DEC-090.

How to apply

When a skill needs to read a versioned schema:

  • The schema goes in `cyberos/docs/contracts/<id>/v<n>/`, NOT in `cuo/_shared/`. (DEC-090)
  • The skill declares the dependency in its frontmatter: `depends_on_contracts: [{id, version, purpose, pin_path}]`.
  • The skill body references the path declared in `pin_path`. The validator rejects body references to a contract not declared.
  • On a contract MAJOR bump, every declared consumer is updated (or explicitly opts in to staying on the older version with a CHANGELOG entry).
