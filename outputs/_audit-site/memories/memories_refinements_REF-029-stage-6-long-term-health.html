REF-029 — Stage 6: Long-term BRAIN health

Trigger

Local-optimization plan (`docs/CyberOS-AGENTS.LOCAL-OPTIMIZATION.md`) Stage 6 ships the operational-polish layer that lets the BRAIN's footprint grow sub-linearly with age and lets multiple agents work concurrently on the same store. Three primitives land together because each depends on the others: Merkle checkpoints anchor the proofs that compaction relies on; compaction needs `.lock.shared` so other agents can `view` while the compaction holds exclusive `.lock` for the manifest update; `.lock.shared` is the precondition for safe concurrent `cyberos-validate` + `cyberos-index` runs against a live store.

Changes (5 additive amendments)

  • **Change A — §7.6 Merkle checkpoints.** Every `op:"consolidation_run"` row gains a `merkle_root` field — SHA-256 tree over rows since the previous checkpoint. Linear `chain` LINK invariant remains canonical; Merkle is a derived index for O(log N) prefix verification.
  • **Change B — §7.7 Audit ledger compaction.** Opt-in, phrase-triggered. Once a ledger month is Merkle-checkpointed AND older than `manifest.compaction_policy.minimum_age_months` (default 12), `audit/<YYYY-MM>.jsonl` collapses to a per-memory `audit/<YYYY-MM>.compacted.jsonl` + Merkle proof; original verbatim preserved at `archive/<YYYY-MM>.jsonl.zst`. ~80% disk savings on year-old ledgers; reversible.
  • **Change C — §8.9 Ledger compaction phase.** Phase 8.9 documents the explicit pre-conditions, atomic phase steps, and MAINTENANCE-mode re-expansion path. Refuses on missing Merkle checkpoint, age below threshold, or unresolved §8.7 phase 4 critical findings.
  • **Change D — §4.9.1 `.lock.shared`.** Sibling lock file for shared-read mode. Read-only ops (`view` per §4) acquire `.lock.shared` only; mutation ops continue with exclusive `.lock`. Consolidation phases §8.1–§8.4 acquire shared, upgrade to exclusive for §8.5–§8.7. Older agents that don't honour shared mode fall back to exclusive — always safe; just without the concurrency benefit.
  • **Change E — §8.7 phase 4 extension.** Self-audit phase 4 now also recomputes Merkle roots and verifies compacted-ledger proofs. CRITICAL on `merkle-checkpoint-divergence` or `merkle-proof-divergence`.

    • What's dormant until first triggered

  • No `merkle_root` rows yet — first one will appear at the next `op:"consolidation_run"` (next session-end or 25-row threshold).
  • No compacted ledgers — earliest possible compaction window is 2027-05 (the 2026-05 ledger needs to age past 12 months).
  • `.lock.shared` is available but unused until two agents work the same store concurrently.

    • Implementation work this unblocks

    No further §0.5 approvals needed for any of these (regenerable derived state on top of new protocol primitives):

  • `cyberos_validate.py` — add `_check_merkle_checkpoints()` walking `op:"consolidation_run"` rows + recomputing roots; add `_check_compacted_ledger()` for `audit/*.compacted.jsonl` files.
  • `cyberos_doctor.py` — new repair op `R5-rebuild-merkle-checkpoint` (re-derive `merkle_root` from period rows under MAINTENANCE mode); new CLI `cyberos-doctor decompact-ledger --month <YYYY-MM>` for §7.7 reverse path.
  • `cyberos_index.py` — new table `merkle_checkpoints(audit_id, root, period_start_audit_id, period_end_audit_id)`; new query `cyberos-index query merkle-proof <chain>` returns inclusion path.

    • How to use this memory

    When ledger growth or session-start cost regresses past Stage-1's incremental-walk improvements, reference this REF: Merkle checkpoints make prefix verification bounded; compaction (after the 12-month wait) cuts disk bloat ~80%. Older agents reading this BRAIN must run AGENTS.md whose canonical SHA is `sha256:77eda214d687f8fd8eb826b8699e62614c3b606e980486c7fcd8496f92ce6dfa` or later; otherwise they trip §13.0 INCOMPATIBLE on the new manifest.compaction_policy field.
