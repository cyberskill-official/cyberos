REF-026 — Deprecate `.protocol-signing-key` file (Bundle K)

Real-world trigger

Stephen, post-Bundle-J: *"is there any way that no need one more separate file .protocol-signing-key?"* Honest re-evaluation: yes — the file was placeholder weight pre-BRAIN-module-P1. Stephen picked Option A (delete now, defer real mechanism to P1) over alternatives.

Refinement adopted

TIER 1: §0.5 TOFU paragraph rewrite + file deprecation marker + DEC-094 v=2 + README update. Single coordinated bundle.

Why this is the right fit

The file's value was zero today (placeholder; signed nothing). Removing it doesn't weaken security — first-time TOFU is always MITM-vulnerable regardless of source. Deferring the canonical mechanism to P1 lets the design happen with real constraints rather than speculation.

This is the **first Bundle that REMOVES protocol surface area** rather than adding. Bundles B/A/C/D/E/F/G/H/I/J all added; Bundle K subtracts. Sets a useful precedent: the §0.4 propose-then-adopt loop can shrink the protocol when something proves to be aspirational rather than load-bearing.

What changed in AGENTS.md

  • §0.5 TOFU paragraph: removed the `.protocol-signing-key` e.g. clause; new wording invites any trusted channel; explicit pre-P1-no-canonical-source note.

    • What changed elsewhere

  • `cyberos/.protocol-signing-key`: overwritten with deprecation marker (cowork sandbox can't `rm`; user can `rm` manually).
  • `cyberos/.cyberos-memory/memories/decisions/DEC-094-protocol-update-policy.md`: v=1 → v=2; signing_keys bullet + bootstrap-auto-pin paragraph updated; History entry appended.
  • `cyberos/docs/CyberOS-AGENTS.README.md`: Part 6 paragraph updated to match new §0.5 wording; Part 8 gains Bundle K as twelfth real-world trigger.
  • `cyberos/docs/CyberOS-AGENTS.CHANGELOG.md`: Bundle K entry appended.

    • Trade-offs

  • First-clone TOFU friction at P1+ when a real key exists: the "somewhere" to paste from isn't defined. Acceptable; designing it now would be premature.
  • Deprecation-marker rather than full deletion: practical compromise given the cowork sandbox's permissions.

    • How to use this memory

    When evaluating whether a piece of protocol surface area is load-bearing or speculative, the Bundle K test is useful: *"What concrete failure would this prevent right now? If the answer is 'none, but it might at P1,' defer."* Adding speculative machinery accumulates carrying cost; removing it is cheap until you're sure you need it.

    When working on BRAIN module P1, design the canonical TOFU-source mechanism then — with real signing-key constraints in hand. AGENTS.md frontmatter, separate file, MCP endpoint, manifest-seed pattern all remain options.
