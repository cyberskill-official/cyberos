DEC-107 — §14 three-state triage: silent / files-only-compact / issues-verbose (Bundle O)

Decision

The §14 end-of-response block now operates as a **three-state triage** (silent → compact → verbose) where the trigger for the verbose state is broadened beyond `operational_mode` to include any issue/finding signal:

| State | Trigger | Output |

|---|---|---|

| **§14.0 omitted** | normal mode + no findings + no mutations + clean self-audit | (nothing) |

| **§14.1 compact** | normal mode + memory mutation OR non-BRAIN file change, AND no issues | `📁 Files changed:` block ONLY (BRAIN + non-BRAIN merged) + optional `Tokens:` line |

| **§14.2 verbose** | ANY of: `op:rejected\|revert\|warn\|health_check` this turn; latest §8.7 reports CRITICAL/WARN; `operational_mode != normal` | full block — `⚠️ Findings:` first, then `📁 Files changed:`, then `Δ Changes (BRAIN detail):`, then `Status:`, then optional `Tokens:` |

Three structural changes vs Bundle N's §14:

1. **§14.1 compact reduced to `📁 Files changed:` only.** No `Δ Changes:` heading, no `Status:` block, no `unchanged:` line, no audit-trail line. BRAIN paths and non-BRAIN paths appear in one merged list — path itself disambiguates.

2. **§14.2 auto-triggers on issues** even when `operational_mode == normal`. Pre-Bundle-O the user had to flip the mode field to see verbose output; post-Bundle-O the agent escalates output format automatically when something needs attention.

3. **`Tokens:` slot reserved** in both §14.1 and §14.2 but emitted only when a real token counter is wired up via runtime tool. Pre-availability the line is omitted — never estimated via `tiktoken` or character count, since approximations would mislead.

Why

User feedback during cowork-session 2026-05-10 (immediately after Bundle N landed):

1. *"Status: unchanged section seem not necessary since there is 'Δ Changes' section, no need to show status if no issues arise too because 📁 Files changed i already know that the BRAIN performing well"* — Status block + `unchanged:` line are redundant when the user can see the file list directly.

2. *"In normal mode no need to should Δ Changes if no issues arise too"* — `Δ Changes:` itself is redundant given `📁 Files changed:`; merge them.

3. *"only show Files changed (no need to implied outside BRAIN), only turn on maintenance mode and show full memory verbose (arrange them smartly too) status when issues arise"* — file paths self-disambiguate; verbose should auto-trigger on issues, not require manual mode flip.

4. *"Is it possible to know/track tokens consumed? if can show it after 📁 Files changed section, if not then skip it"* — token tracking is desired but should not be faked.

The §14 block has been on a noise-reduction journey since Bundle I (compact format), through Bundle N (omission rule + audit-trail suppression), and Bundle O completes it by reducing the routine-mutation case to a single `📁 Files changed:` block.

Trade-offs

  • **Pro**: routine mutation turns now produce ~3 lines of §14 output (header + 1-2 file bullets) instead of ~10 lines (`📝` header + `Δ Changes:` heading + bullets + `📁 Files changed:` heading + bullets + `Status:` heading + 4 status lines + `unchanged:` line).
  • **Pro**: issues automatically promote output to verbose without operator intervention — the user always sees findings in full context.
  • **Pro**: token-tracking slot reserved without committing to a specific implementation.
  • **Con**: `Δ Changes (BRAIN detail):` is now only visible in §14.2 (issues / verbose modes). For routine mutations, the user trusts the audit ledger as the auditable record (which it always was per §7).
  • **Con**: agents that don't track per-turn finding counts may incorrectly emit §14.1 when they should emit §14.2. Mitigation: per-turn finding count is cheap (just count `rejected`/`revert`/`warn`/`health_check` rows appended this turn); §8.7 self-audit catches schema-level drift if any agent gets it wrong.

    1. Count this turn's rows by op: `rejected_count + revert_count + warn_count + health_check_count`. If > 0 → §14.2.

    2. Read `meta/health/<latest>.md` summary line. If CRITICAL > 0 OR WARN > 0 → §14.2.

    4. Check whether any memory file was created/modified/renamed/deleted OR any non-BRAIN file was modified. If yes → §14.1; if no → §14.0 omit.

    When in doubt, escalate one tier (compact when omit might fit; verbose when compact might fit). The user's stated preference is "less is more" — but ALSO "show full when something needs attention".
