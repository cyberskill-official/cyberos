Classification rules

Every memory file carries a `classification` that determines retention and consent posture. The four classes are:

`personnel` — about a specific Member (employee, contractor, founder). Retention default 3 years. Subject consent required (`has_consent: true` plus a non-empty `consent_scope`). The subject can edit or delete their own at will. Never auto-resolved on conflict.

`client` — about an external client account, contact, or engagement. Retention default 7 years. Active engagement OR explicit consent required. Never auto-resolved on conflict.

`operational` — internal facts about the company, modules, decisions, processes that are not personal. Retention default 1 year. No consent gate.

`public` — protocol files, glossary, generic project documentation. Retention indefinite. No consent gate.

The denylist (§9.3 of `docs/CyberOS-AGENTS.md`) is structurally enforced at the write boundary regardless of classification: compensation, government IDs, bank/card numbers, home addresses, health PII, secrets, and external-party PII without explicit consent are never written to memory — pointers to where the value lives are stored instead.

How to use this memory

Set the `classification` field correctly when creating a new memory file: be conservative — when in doubt, choose the stricter class. The same rule decides whether `consent.has_consent` and `consent_scope` must be present and non-empty.
