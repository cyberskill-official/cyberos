DEC-105 — Deprecate `.protocol-signing-key` file (defer canonical TOFU source to BRAIN module P1)

Decision

The `cyberos/.protocol-signing-key` file (introduced by Bundle B / DEC-094 as a placeholder) is deprecated. §0.5 TOFU paragraph rewritten to remove the file reference. Pre-BRAIN-module-P1, no canonical out-of-band source is mandated by the protocol; users paste fingerprints from any trusted channel. The canonical mechanism will be designed when P1 ships and a real CyberSkill signing keypair is generated.

Why

Stephen surveyed the project surface area post-Bundle-J and asked: *"is there any way that no need one more separate file .protocol-signing-key?"* The honest answer was yes — the file was placeholder weight:

  • No real CyberSkill upstream signing key exists yet (BRAIN module P1 hasn't shipped).
  • The file's content was comment-and-placeholder; it signed nothing.
  • It documented an aspiration rather than enforcing real trust.
  • Sale-noti and any future downstream project would have had to copy it (extra friction for zero benefit).

    • When P1 ships, the canonical distribution mechanism gets designed with real constraints (where does the keypair live? how does it propagate? does it go in AGENTS.md frontmatter, the manifest directly, an MCP endpoint, or recreated as a separate file?). Designing it speculatively now would lock in choices we can't yet defend.

    How

    §0.5 TOFU paragraph rewritten — removed the e.g. clause about the file, replaced with general language about "any trusted out-of-band channel" and an explicit P1-deferral note. The file itself overwritten with a deprecation marker pointing at this DEC + REF-026 + the CHANGELOG entry. The `manifest.protocol.signing_keys[]` array stays unchanged in §6 — only the population mechanism is deferred.

    DEC-094 (the originating decision for the file) bumped to v=2 with a History entry documenting the Bundle K deprecation. README Part 6 updated to match.

    Trade-offs accepted

  • **First-clone TOFU friction at P1+**: when a real key exists, new employees will need to paste it from somewhere. The "somewhere" isn't defined yet. Acceptance: defining it now without P1's actual constraints would be premature; the friction lands when there's actual signing to do, which is the right time.
  • **Deprecation-marker file rather than full deletion**: the cowork sandbox can't `rm` files outside `.cyberos-memory/`. The marker file documents the deprecation; user can manually delete from local clone. Acceptance: cleaner than leaving the placeholder pretending to be active.
  • **First-time TOFU is always vulnerable to MITM regardless of source.** Whether you paste from `.protocol-signing-key`, an email, a Slack message, or an in-person exchange, the threat model is the same — an attacker with the right access can inject a malicious fingerprint. The file's "convenience" was illusory comfort, not real security.

    • What this does NOT change

  • `manifest.protocol.signing_keys[]` array structure — unchanged.
  • §0.5 SHA-pinning, three-way-conflict handling, rollback flow — unchanged.
  • The protocol's overall TOFU model — unchanged; only the population mechanism is deferred.
  • BRAIN-module-P1 scope — when P1 designs the canonical distribution mechanism, all options (frontmatter, separate file, MCP endpoint, manifest seed) remain on the table.

    • How to use this memory

    When designing BRAIN module P1's protocol-release endpoint, decide on the canonical distribution mechanism with real constraints in hand. Consider trade-offs between AGENTS.md frontmatter (couples trust to protocol evolution), separate file (small surface area but extra config), MCP endpoint (server-required), or manifest-seed pattern (per-machine first-bootstrap). Bundle K leaves the door open; P1 picks.

    When a downstream project asks "where do I get the upstream signing fingerprint?" pre-P1, answer: *"From any trusted channel CyberSkill explicitly designates. Pre-P1, there is no canonical source — that's by design."*
