REF-027 — Bundle L TIER 2: Legacy memory_id carve-out via meta/legacy-ids.md registry

Trigger

2026-05-07 BRAIN healthcheck (this conversation) surfaced 4 invalid `memory_id` values per §5.2 — three meta files (`tombstones.md`, `retention-rules.md`, `classification-rules.md`) and `memories/facts/FACT-005-doc-changelog-convention.md`, all carrying pre-protocol mnemonic IDs (`mem_01HSXX0…` ULID-style and `mem_F005…` purely-mnemonic). The healthcheck additionally found 13 §4.7 SHA-mismatched files; closing the SHA-mismatch finding required appending corrective `op:str_replace` audit rows, but for `meta/tombstones.md` the corrective row would itself fail §5.2 (it would carry the legacy `memory_id`).

Decision

Carve out a closed-set legacy-IDs registry rather than tombstone-and-recreate the 4 files. Tombstone-and-recreate would have cascaded `relationships:` rewrites across adjacent memories citing `mem_F005…` as a `cites`/`refines` target. The registry approach localises the change to one new file and one new validators-table row.

What landed (canonical SHA `sha256:599e1097…`)

  • §4.2 denylist exemption set extended with `meta/legacy-ids.md` (one entry added to the existing list).
  • §5.2 validators table extended with one new row defining the legacy-IDs carve-out: closed-set, registry-gated, future writes still must use UUIDv7/ULID.
  • §13.1 step 7a creates an empty `meta/legacy-ids.md` at bootstrap (so newly-bootstrapped BRAINs have the registry from day 1, even if it stays empty).

    • How to use this memory

    When future agents encounter a non-conforming `memory_id`:

    1. Check `meta/legacy-ids.md`. Found → accept and continue. Not found → reject per §5.2.

    2. If a user wants to add a NEW entry to the registry, that's a §0.5 protocol upgrade — entries don't land via routine writes. The closed-set property is what keeps this from becoming an escape hatch.

    3. When tombstoning a file with a legacy ID, the tombstone reference in `meta/tombstones.md` keeps the legacy ID for traceability — the registry entry stays even after tombstone.

    Side-finding (deferred to a follow-up bundle)

    The healthcheck identified that the 269-row pre-upgrade ledger was written by 3 distinct canonicalisations (Python `json.dumps` with two exclusion conventions; RFC 8785 strict). LINK invariant holds across all three — chain integrity is intact — but cross-writer hash recompute fails for ~190 rows under §7.2 strict JCS. This is INFO-severity per §7.2 (the LINK invariant is the authoritative guarantee). A follow-up TIER 1 amendment was proposed: *"writers MUST match `manifest.protocol.last_writer_canonicalization` once set; switching emits `op:warn reason:canonicalization-drift`"*. Not part of Bundle L; queued for a future bundle.
