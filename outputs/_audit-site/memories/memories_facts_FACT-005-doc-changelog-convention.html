FACT-005 — Doc-CHANGELOG convention (no inline versioning)

Each maintained design document in `docs/` carries a sibling `<DocName>.CHANGELOG.md` that holds the day-by-day record of changes. The doc itself does NOT carry an inline version marker (no "v1.0", no "## 2.3.1"). Improvements land continuously; the changelog is the canonical record of evolution; git history is the canonical record of the diff.

The convention applies to: `CyberOS-AGENTS.md`, `CyberOS-PRD.docx`, `CyberOS-SRS.docx`. Each has its own `*.CHANGELOG.md` next to it. The format is inspired by Keep a Changelog 1.1.0 but date-stamped rather than version-stamped: top-level headings are `## YYYY-MM-DD — <one-line summary>`, with optional same-day rounds noted as `## YYYY-MM-DD (afternoon revisions)` etc. Sections are `### Added`, `### Changed`, `### Removed`, `### Real-world trigger`.

Why

Inline version markers in living protocol documents create three failure modes: (a) cross-machine `INCOMPATIBLE` noise when versions tick on every minor amendment; (b) "false stability" when the version number doesn't bump despite real changes; (c) cognitive overhead reading vs. writing the document — the version becomes the headline rather than the actual content. The CHANGELOG approach moves the audit trail to its own file where it can grow without polluting the doc.

Origin

Codified on 2026-05-04 alongside DEC-086 (which removed `schema_version` and `compatible_runtimes` from the manifest for the same reason). The trigger was Stephen asking "is `compatible_runtimes` and `schema_version` necessary?" — the analysis revealed both were vestigial and inconsistent with the day-by-day protocol-evolution model. Captured as the feedback memory `memories/preferences/feedback-no-inline-versioning.md`.

How to use this memory

When opening or maintaining a doc in `docs/`, look for the sibling `<DocName>.CHANGELOG.md` to find the latest amendment landed. When making a change to a doc, add a new dated section to the changelog in the same commit; never add an inline version marker. When ingesting a doc into BRAIN, both the doc and its changelog are valid sources — record the changelog's most-recent date in the digest's `provenance.source_ref` so future drift detection can compare against it.
