REF-033 — Self-audit pass + DEBUG/MAINTENANCE modes (§8.7, §8.8)

Real-world trigger

User question (Stephen, 2026-05-06): *"Can the BRAIN audit itself? While users are using the BRAIN and unexpected issues happen, I should be notified so I can fix it asap. For now maybe we can use DEBUG or ROOT mode."*

Diagnosis: pre-C protocol had partial self-audit elements but no integrated full-store integrity pass, no notification escalation channel, and no clear distinction between read-side verbosity and write-side repair authority.

Refinement adopted

TIER 1 (§8.7 self-audit pass + DEBUG mode) and TIER 3 (§8.8 MAINTENANCE mode for operator-confirmed repairs). TIER 2 (org-level escalation channel) deferred to BRAIN-module P1 ship.

What changed in AGENTS.md

  • New §8.7 (Self-audit pass) — sev-1 — sixth phase of consolidation. Six checks. Three severity buckets. Three operational modes (normal/debug/verbose).
  • New §8.8 (MAINTENANCE mode) — sev-0 — time-limited repair authority; per-op chat confirmation; logged as `actor_kind: maintainer` with `maintenance_session_id`.
  • §6 manifest gained `operational_mode` and `health_check_policy` fields.
  • §7.1 op enum gained six new ops: `health_check`, `warn`, `drift_candidate`, `shallow_candidate`, `maintenance.start`, `maintenance.end`.
  • §14 end-of-response block gained a `health` line.
  • New `meta/health/` directory (deterministic health reports).

    • Why this is the right fit

    The naming distinction (DEBUG = read-side, MAINTENANCE = write-side) avoids the Linux-root footgun pattern. The six-check ordering goes from cheapest (schema validate) to most expensive (chain end-to-end), so partial runs degrade gracefully. The severity-routing into `manifest.operational_mode` lets a founder enable DEBUG on a remote employee's BRAIN over a video call without elevating any privileges. The `meta/health/<YYYY-MM-DD>-<sha>.md` deterministic naming makes reports diff-able across sessions.

    Why MAINTENANCE mode is sev-0 and time-limited

    Repair ops touch exactly the things the protocol normally forbids. A long-lived maintenance session is indistinguishable from a quietly-modified protocol. Time-limited (1 hour or session end), per-op confirmation, and audit logging with maintainer actor-kind together make the elevated-privilege window observable and bounded.

    Trade-offs

  • **DEBUG-mode banner clutter**: explicit purpose of the mode; user controls.
  • **MAINTENANCE per-op friction**: appropriate for high-stakes recovery.
  • **TIER 2 escalation deferred**: requires the BRAIN module's network surface; not implementable on the filesystem layer alone.
  • **Six checks may be slow on large stores**: deferred concern; once any per-tenant total exceeds 1MB soft cap (§5.5), the consolidation cadence already needs revisiting.

    • How to use this memory

    When the user reports a recurring oddity, suggest DEBUG mode. When the user encounters genuine corruption (CRITICAL severity from a health-check), guide them through MAINTENANCE entry. When designing the BRAIN module's P1 escalation channel, this REF is the canonical reference for what to forward (CRITICAL + aggregated WARN, metadata-only) and what to NOT forward (memory content, INFO).
