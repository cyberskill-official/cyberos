DEC-100 — Comprehensive audit-fix pass + §0.6 related-files update rule

Decision

A comprehensive whole-document audit of AGENTS.md found 19 issues across three tiers. All adopted in Bundle F. Three new sections added (§0.6, §7.5, §8.1–§8.5 explicit headers); 10 stale claims fixed; 5 cross-references repaired; 4 compressions applied. AGENTS.md SHA went `sha256:f9328b7…cb1022` (post-E) → `sha256:f7f3934…f4f1b7` (post-F).

§0.6 — the most consequential addition — codifies the rule that **every successful protocol upgrade MUST be accompanied in the same chat turn by updates to**: CHANGELOG (dated entry), README (any tracked Part), cross-linked FACT memories, and implementation files (e.g., `brain_writer.py` for §7.2). The rule is sev-1 (recoverable failure mode), positioned in §0 alongside the other standing-rule sections (§0.4 propose-refinements; §0.5 protocol distribution).

Why

The cumulative effect of 5 bundles + 4 follow-on tasks across 2026-05-06 had introduced doc inconsistencies the agent didn't surface during normal work: the §5.1 heading still claimed "27 fields" after sync_class made it 28; §8 still claimed "5 phases" after §8.6 + §8.7 made it 7; §8.7 used the pre-Bundle-D chain-hash formula; §4.7's orphan-manifest check would have falsely flagged every protocol_upgrade as crash-mid-consolidation; §11.5 referenced a non-existent §8.5; §9.7 mentioned an undefined "30-day legal hold." These were silent bugs — the protocol still worked because no implementation strictly followed every word of every section, but a clean-room implementation reading just the doc would have failed in specific ways.

The §0.6 rule prevents the same drift from recurring. By making related-files updates a sev-1 protocol obligation rather than a discretionary courtesy, future protocol upgrades cannot quietly accumulate doc-level debt.

How (the 19+1 changes)

**TIER 1 — bugs / stale claims** (10 items): §0 precedence statement, §5.1 field count, §8 phase count, §8.7 hash formula, §4.7 orphan-manifest pairing, §9.7 30-day-hold removal, §11.5 §8.5 reference, §11.6 schema extensions declared, §17.5 actor_keys clarified, §7.5 corrects-vs-correction_to defined.

**TIER 2 — stale or inconsistent** (5 items): §3 layout adds protocol-history + health, §13.1 null vs "", §16 op:warn vocabulary, §0.2 schema_version removed, §9.7 cite §17 sync_class.

**TIER 3 — compression / consolidation** (4 items): §0.5 forbidden-by-§0.2 paragraph compressed, §4.10 forbidden-tools compressed, §4.1/§11.7 path-constraints merged, §9.4 project-specific examples generalised.

**Plus §0.6 (Related-files update rule)** as the headline addition.

Trade-offs accepted

  • **§8 renumbering means future cross-references to "phase 5" might be ambiguous** (§8.5 now means the explicit subsection; "phase 5" still means the recompute-manifest action). Acceptance: §8.5 subsection is explicitly named "Update manifest" so the meaning is clear.
  • **§0.6 sev-1 is intentionally below sev-0**: failing to update related files is recoverable (back-fill works); making it sev-0 would freeze writes whenever someone forgot to update CHANGELOG, which is friction without proportional value. The standing-rule status (always applied) is what makes drift unlikely.
  • **§7.5 introduces a new subsection in §7 just for `corrects` vs `correction_to`**. Could have folded into §7.1 prose. Decision: dedicated subsection makes the distinction discoverable for anyone debugging an audit chain (where both mechanisms appear).

    • How to use this memory

    When proposing future AGENTS.md changes, ALWAYS update CHANGELOG and README in the same chat turn (per §0.6). When debugging an audit chain that has both `op:"corrects"` and `correction_to` populated, see §7.5 for the disambiguation. When extending §6 manifest schema (e.g., for BRAIN module P1's `actor_keys`), follow §0.5 protocol upgrade flow strictly — the §13.0 `INCOMPATIBLE:<field>` tripwire will catch any premature additions.
