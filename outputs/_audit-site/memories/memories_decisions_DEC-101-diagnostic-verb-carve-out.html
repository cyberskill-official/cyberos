DEC-101 ã Diagnostic-verb carve-out for PRISTINE auto-bootstrap

Decision

ôÏ1 step 2's `Auto-bootstrap silently if PRISTINE` directive carves out an exception when the user's current-turn message contains a recognised diagnostic verb. Default list: `healthcheck`, `status`, `inspect`, `audit`, `check brain`, `show brain`, `view brain`, plus the configured `manifest.health_check_policy.on_demand_phrase`. Project may override via `manifest.health_check_policy.diagnostic_verbs[]`.

When the carve-out applies, ôÏ13.0 returns `PRISTINE-DIAGNOSTIC-HOLD` (sub-state of `PRISTINE`); the agent surfaces what ôÏ13.1 would create and waits for explicit user consent before bootstrapping.

Why

A fresh Cowork session at `/Users/stephencheng/Projects/CyberSkill/sale-noti/` (the first downstream consumer of the post-Bundle-F protocol) ran `healthcheck` against a `PRISTINE` BRAIN. The agent there reasoned correctly: bootstrapping silently would have changed the state being diagnosed (after bootstrap, the BRAIN would no longer be PRISTINE ã the diagnostic answer becomes obsolete the moment it's run). The agent held off and surfaced the friction as an ôÏ0.4 refinement candidate. Stephen approved upstreaming so future downstream projects don't re-encounter it.

The pre-G protocol's silent auto-bootstrap directive was correct for *task-oriented* first-use ("create a memory", "remember that X") ã bootstrap is a prerequisite, no point prompting. But for *diagnostic* first-use ("healthcheck", "status", "inspect"), the silent bootstrap masks the very answer the user asked for. Carving out an exception preserves the diagnostic discipline.

How

ôÏ1 step 2 amended with the carve-out clause naming the default verb list. ôÏ13.0 gains `PRISTINE-DIAGNOSTIC-HOLD` as an explicit sub-state with action: surface verdict + list bootstrap effects + await consent + do not write. ôÏ6 manifest gains `health_check_policy.diagnostic_verbs[]` for project-level override.

Consent phrases that resume bootstrap: `bootstrap and continue`, `just bootstrap`, or any task-oriented instruction (the diagnostic intent has yielded to a work intent; bootstrap proceeds normally).

Trade-offs accepted

  • **Diagnostic verb list is fuzzy at edges** ã what about "is the BRAIN ready?" or "show me what you have"? The rule is conservative: only literal verb matches trigger the carve-out. Ambiguous phrasings still trigger silent bootstrap. The `manifest.health_check_policy.diagnostic_verbs[]` override lets each project tune.
  • **Friction for first-time-task users who happen to mention "status"** ã e.g., "what's the status of this project?" might trigger PRISTINE-DIAGNOSTIC-HOLD even though they want a task. Acceptance: one-line consent ("bootstrap and continue") resumes; small friction. Better than silent overwrite of diagnostic answer.
  • **Default verb list ships in ôÏ1 prose AND in manifest.health_check_policy.diagnostic_verbs[]** ã duplication. The ôÏ1 list is the bootstrap-default (used before manifest exists); the manifest list is the configured-default (used afterwards). They CAN diverge if a project edits one without the other; this is a minor drift concern.

    • What this does NOT do

  • Does NOT change behaviour for `READY` or `CORRUPT` states ã those still operate per their existing ôÏ13.0 actions.
  • Does NOT extend the carve-out to non-PRISTINE states. Mid-session diagnostic invocations against a READY BRAIN proceed normally.
  • Does NOT prevent the user from explicitly asking for both at once ("bootstrap then run healthcheck"); the agent treats that as a sequenced task.

    • How to use this memory

    When a fresh Cowork session opens at a `PRISTINE` project root and the user's first message contains a diagnostic verb, surface the absent-state verdict per ôÏ13.0's `PRISTINE-DIAGNOSTIC-HOLD` action. Wait for consent. When other downstream projects encounter similar fresh-session friction, propose ôÏ0.4 refinements; this DEC is the canonical example of the propose-then-upstream loop firing on real downstream feedback.
