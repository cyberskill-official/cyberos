DEC-094 — Protocol update policy: SHA-pinning, signed releases, consent-gated upgrades

Decision

`AGENTS.md` is content-addressable: identified by the canonical SHA-256 of its normalised text, not by an inline version marker. The manifest pins the currently-approved SHA; every session start verifies the pin; protocol changes happen only via the explicit chat-turn approval phrase *"approve protocol upgrade to `<sha256:…>`"*.

Why

The pre-B protocol was silent on its own distribution. There was no tripwire for: hand-edits to AGENTS.md outside the consent flow, host-platform silent updates, malicious replacements, or accidental drift between machines. The consent gate (§0.2) protected against runtime injection but did not protect the protocol document itself.

Forced auto-update would defeat §0.2 — anyone who controls the publish pipeline silently rewrites every BRAIN's rules. Pure manual updates work but offer no upstream-authenticity guarantee. The chosen design layers three properties: **authenticity** (Ed25519 signatures over canonical SHA, verified against pinned fingerprints), **authorization** (explicit chat-turn approval phrase per §0.2), **auditability** (every upgrade is an `op:"protocol_upgrade"` audit row; prior version archived under `meta/protocol-history/`).

How

Manifest carries a `protocol` block:

  • `sha256` — currently-approved canonical SHA.
  • `approved_at` / `approved_by` — when and who.
  • `loaded_path` — where AGENTS.md lives (default `docs/CyberOS-AGENTS.md`).
  • `signing_keys` — array of pinned Ed25519 fingerprints (TOFU; first add via explicit user paste; cyberos repo bakes CyberSkill upstream key into `cyberos/.protocol-signing-key`).
  • `last_checked_at` — last upstream-release-check time.

    • §13.0 state classifier gains `INCOMPATIBLE:protocol-sha256-mismatch`. §7.1 op enum gains `protocol_upgrade` and `protocol_rollback`. §13.1 bootstrap gains step 12 (auto-pin at first run) and step 13 (seed `meta/protocol-history/`).

    Trade-offs accepted

  • **TOFU trust establishment** rather than CA-signed cert chain: simpler, sufficient for the threat model (compromise of signing key is detectable via fingerprint comparison).
  • **Silent-check, manual-apply** rather than fully-automated: friction at adopt time, but absolute protection against silent protocol rewrites.
  • **Content-addressable rather than version-numbered**: aligns with the no-inline-version philosophy (FACT-005); avoids version-bump fatigue or version-lying; SHA is a name, not a comparison.
  • **Bootstrap auto-pin without prompt**: friendly for solo use today; first-clone gets a baseline pin (just the SHA of whatever AGENTS.md is loaded); user adds upstream signing key later when one exists. Pre-Bundle-K the cyberos repo carried `.protocol-signing-key` as a placeholder; Bundle K deferred this to BRAIN module P1.

    • Bootstrap behaviour

    First-run bootstrap silently auto-pins the canonical SHA of whatever `AGENTS.md` is loaded. `signing_keys: []` initially. User pastes CyberSkill upstream fingerprint when ready (or first-clones a cyberos repo that bakes it in).

    What this does NOT do

  • Does not auto-fetch upstream releases on a schedule (silent-fetch is fine; silent-apply is forbidden).
  • Does not bypass §9.3 denylist or §4.2 content gate (those stay sev-0 inviolable).
  • Does not re-validate previously-written memories under new rules (handled by additive-only protocol-evolution discipline; see Bundle C / DEC-096 once landed).

    • How to use this memory

    When the user proposes an AGENTS.md edit, follow §0.5: archive prior version, edit, compute new canonical SHA, ask for the chat-turn approval phrase with the exact target SHA, str_replace manifest, append `op:"protocol_upgrade"`. When session-start computes a SHA mismatch, freeze writes and surface the diff — never auto-resolve. When the user pastes a fingerprint, validate format (`ed25519:<base64>`), append to `signing_keys` with provenance.
