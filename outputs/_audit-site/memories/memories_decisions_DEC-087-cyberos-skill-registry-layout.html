CyberOS skill-registry layout — Option B + fr-create/fr-audit split

The CyberOS skill registry lives at `cyberos/docs/skills/`, organised as `cuo/<role>/<workflow-skill>/SKILL.md` (Option B from the 2026-05-05 trade-off diagram). The PRD §3.2 mandate `~/.cyberos/skills/cuo/<role>/` is interpreted as a runtime install path; the source-of-truth is the docs registry. Cross-persona reusable skills live in `cuo/_shared/` (per DEC-061's `draft-payslip-explanation` worked example).

The Feature Request workflow ports out of `feature-request/v2.0.0` as two atomic skills, NOT one:

  • `cuo/cpo/fr-create/` — owns sections §0–§14 + §18 of v2.0.0.
  • `cuo/cpo/fr-audit/` — owns sections §15–§17.

    • The split is intentional: each skill is a standalone trigger AND chainable via the LangGraph supervisor's `next_skill_recommendation` field. The chain `fr-create → fr-audit` reproduces v2.0.0's `create_and_audit` mode; standalone `fr-audit` reproduces v2.0.0's `audit_only` mode. The shared `feature_request@1` template lives at `cuo/_shared/feature-request-template/template.md` so both skills (and any future `cuo/cto/fr-to-tech-spec`) load the same authoritative schema.

    How to use this memory

    When making layout / split / interface decisions for new CUO skills:

  • New workflow under an existing role → `cuo/<role>/<workflow>/SKILL.md`.
  • New cross-persona reusable workflow → `cuo/_shared/<workflow>/SKILL.md`.
  • Each skill carries its own `CHANGELOG.md` (Keep-a-Changelog 1.1.0).
  • Each skill declares `expects:` + `produces:` envelopes so the supervisor can chain it safely.
  • Audit row schema (`genie.action_log`) maps 1:1 to the workflow leaf via `(persona_id, skill_id, skill_version)`.

    • When asked "should this be one skill or two?": apply the standalone-trigger test. If the user could plausibly want to invoke just one half, split. The fr-create/fr-audit split is the canonical example.
