DEC-097 — Canonical-JSON tightening: cite RFC 8785 JCS as authoritative algorithm

Decision

§7.2 canonicalisation now cites **RFC 8785 (JSON Canonicalization Scheme, JCS)** as the authoritative algorithm. Implementations MUST conform to RFC 8785 in full. Hand-rolled approximations (`json.dumps(sort_keys=True, separators=(',', ':'), ensure_ascii=False)`) MUST run the JCS test vectors and confirm bit-identical output before being trusted to chain audit rows.

The chain LINK invariant (`row[N].prev_chain == row[N-1].chain`) is reaffirmed as the authoritative integrity guarantee. Hash *recomputation* across writer versions MAY fail (different writers may have produced different canonical-JSON outputs for logically-identical rows); this is informational, NOT a chain break. §8.7 self-audit reports hash-recompute differences at INFO severity.

Why

The 2026-05-06 cascade verifier surfaced 149 pre-existing audit rows that fail bit-perfect hash recomputation against the new `brain_writer.py`. Both writers followed the pre-D §7.2 wording (keys sorted, compact separators, non-ASCII preserved, "shortest IEEE-754") yet produced different output bytes. Root cause: §7.2 was underspecified — multiple legal interpretations of "shortest IEEE-754" exist (Python `1.0` vs JCS `1`); UTF-16 code-unit vs byte-lex key ordering was unspecified; boolean serialisation (`True` vs `true`) was unspecified.

This is exactly the §0.4 trigger pattern: a real, observed failure exposing a protocol underspecification. The fix is small, specific, and additive.

How

§7.2 rewrite cites RFC 8785 JCS, calls out the four most common divergence points (number toString, key UTF-16 ordering, boolean lowercase, NFC string normalisation), and names reference implementations (`rfc8785` PyPI package, `canonicalize` npm package). The LINK-vs-recompute distinction is made explicit so future implementers don't waste time hunting "phantom" chain breaks that are actually writer-version differences.

Trade-offs accepted

  • **Test-vector dependency** — implementations adopting `json.dumps`-based shortcuts must validate against JCS test vectors before being trusted. Cost: a one-time test pass. Benefit: cross-writer reproducibility from this point forward.
  • **Pre-existing rows remain hash-non-reproducible** — we do NOT retroactively re-chain pre-D rows. The LINK invariant holds; that's sufficient. Forcing a re-chain would itself break older external exports that pinned chain values.
  • **Number-serialisation surprise** — Python implementations using `json.dumps` will produce `1.0` for `float(1)`, but JCS requires `1`. This is the most common gotcha; implementations must use a JCS-conformant library OR a custom number serialiser.

    • What this does NOT do

  • Does NOT invalidate pre-D audit rows. They remain on the chain.
  • Does NOT require a one-time chain rebuild. LINK integrity is preserved.
  • Does NOT require writer libraries to be re-licensed (RFC 8785 is freely implementable; reference libraries are MIT/Apache).

    • How to use this memory

    When implementing or auditing a §7.2 hash, first ask: am I checking LINK integrity (authoritative) or hash recomputation (informational)? When integrating a new writer language (e.g., Go, Rust), pull a JCS-conformant library; do not hand-roll. When debugging a hash-recompute mismatch, check the four common divergence points before suspecting a real bug.
