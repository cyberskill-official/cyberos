DEC-086 — Remove schema_version + compatible_runtimes from manifest; field-presence forward-compat (AGENTS.md §6, §4.3, §13.0)

**Status: Locked. Deciders: Founder.**

**Decision.** Two manifest fields are removed:

  • `compatible_runtimes` — vestigial; never referenced anywhere in protocol logic.
  • `schema_version` — conceptually misaligned with day-by-day protocol evolution. Discrete-version-number model would either bump daily (constantly tripping cross-machine `INCOMPATIBLE`) or never bump (lie about state). Both options worse than removal.

    • **Replacement forward-compat mechanism.** Field-presence detection at the validator level:

  • §4.3 hygiene: unknown frontmatter fields → `op:rejected reason:unknown-frontmatter-field:<name>` + surface to user.
  • §13.0 state classifier: `INCOMPATIBLE:<field>` triggered when `manifest.json` carries any field not in the agent's loaded §6 schema.

    • Same forward-compat protection (agent too old → refuse), no inline version markers, aligned with day-by-day evolution.

    **How to apply.** Manifest str_replace on cyberos + styx BRAINs to drop both fields. Future manifests carry only the fields in the current §6 schema. Agents reading a manifest with unknown fields surface "agent likely too old; run the latest AGENTS.md."

    **Cross-ref:** AGENTS.md §6 + §4.3 + §13.0; tombstones DEC-082 + REF-007 same session; PRD/SRS DEC-072 reference also updated.
