REF-020 — First end-to-end pipeline run validates scaffolds + surfaces 7 actionable findings

The CyberOS pipeline (registry v0.2.9 — 8 skills + 6 contracts + chain-selector) ran end-to-end against a real project for the first time on 2026-05-06. Input: a 34-page PDF plan for SaleNoti (CyberSkill's first product venture, a Vietnamese Shopee deal-tracker). Output: 11 markdown artefacts spanning project_brief → chain_plan → PRD → audit → 3 FRs → 3 audits → 1 tech-spec → 1 impl-plan, all under `/Users/stephencheng/Projects/CyberSkill/sale-noti/cyberos-pipeline-output/`.

The run was manual (no runtime; Claude played the role of every skill in chat) — but that's exactly what made it useful. The scaffolds were tested as PROMPTS, not as code, and the findings below are about prompt quality, contract completeness, and chain-flow ergonomics — all of which transfer 1:1 to the future runtime.

How to use this memory

When designing new skills, when refining existing skills' STANDALONE_INTERVIEW.md, when authoring new contracts, consult this memory's findings. It captures lessons from a REAL document → REAL pipeline pass that wouldn't surface from theoretical scaffolding alone.

7 actionable findings

F1 — `tech_spec@1` is referenced inline in fr-to-tech-spec/SKILL.md but is NOT yet a registered contract

The skill's body describes the artefact shape with 6 H2 sections. That's good documentation but bad contract management — fr-audit's pattern (where the rubric IS in RUBRIC.md and the artefact IS in `cyberos/docs/contracts/feature-request/template.md`) didn't apply to tech-specs.

**Recommended action (registry v0.3.0):** Promote `tech_spec@1` to `cyberos/docs/contracts/tech-spec/` (CONTRACT.md + template.md + CHANGELOG.md). Same pattern as `feature-request`, `prd`, `srs`, `impl-plan`. Add to fr-to-tech-spec's `depends_on_contracts:` block.

F2 — `chain_profile: standard` defaults work for software products; needs validation for non-software project_kinds

The SaleNoti run validated the standard profile (software_product → standard chain). But the lean profile (internal_tooling, research_spike) and full profile (regulated, eu_ai_act_high) haven't been tested with real inputs. The selection rules in chain-selector/SKILL.md are first-match-wins logic; one wrong rule could mis-route entire project classes.

**Recommended action (when more projects run):** Capture each new project's `project_kind` + final `chain_profile` + whether the user overrode → after 5-10 real projects, recompute the rule precedence and confidence.

F3 — The PRD authority-elevation pass is critical — without it, llm-implicit goals leak in

prd-author's STANDALONE_INTERVIEW.md Q5 (authority-elevation pass) is the gate that prevents `llm-implicit` from appearing on Goals. In the SaleNoti run, the founder's strong opinions made this easy ("Stephen wants to capture 10K MAU" → `human-edited`). For future projects with weaker founder input, the elevation prompt becomes the difference between a defensible PRD and a vague-claims PRD.

**Recommended action:** Strengthen the elevation prompt. Currently options are (a) elevate to human-edited if user confirms, (b) elevate to llm-explicit if cite a source, (c) reword, (d) drop. Add (e) "park for v2" — sometimes the best answer is "we don't know yet, defer to PRD iteration 2 after research".

F4 — Vietnam-context BRAIN reads matter

requirements-discovery's Q15 (regulatory touchpoints) hit on Vietnam PDPL (Decree 13/2023). For a Vietnamese-domiciled project, this is non-negotiable. The skill's STANDALONE_INTERVIEW.md doesn't currently flag Vietnam-specific compliance triggers (PDPL, e-commerce-licensing, FDI rules for foreign-capital-percentage thresholds, etc.).

**Recommended action (registry v0.2.x patch):** Add a Vietnam-context micro-checklist to Q15 that triggers when `manifest.tenant.residency == 'vn'`. Mirrors how EU AI Act checks fire on EU-residency.

F5 — fr-author's per-FR scope is correctly granular

The 3 FRs (FR-001 extension UI, FR-002 backend API, FR-003 magic-link auth) decomposed cleanly from the PRD. Each is one engineer × 2-3 weeks; each has clean dependency declarations; each has independent test plans. The granularity rule of thumb worked: "one FR per natural API or UI surface."

**Recommended action:** Document this granularity heuristic in fr-author's SKILL.md or in a new REF entry. Currently the skill is silent on "how big should an FR be?" — implicit knowledge that a future agent will lose.

F6 — Tech-spec sizing accuracy is unknowable until real engineering happens

TS-001 estimated 13 engineer-days for FR-001. The PDF plan estimated 7 weeks for the full Phase 1 MVP with 1 senior + 2 interns; that's ~21 engineer-weeks ÷ ~12 FRs = ~1.7 weeks/FR average. TS-001's 2.5-week estimate is in-band but unverifiable until real coding happens.

**Recommended action (post-launch):** Run a retrospective after Phase 1 MVP ships: `actual_engineer_days ÷ estimated_engineer_days` per ticket. Surface the calibration ratio as an anomaly_signal in spec-to-impl-plan's `human_fine_tune.signals_to_initiate` (already there as `sprint_estimate_accuracy_below: 0.5`).

F7 — INV-002 of spec-to-impl-plan (never auto-create tickets) is the right rule but needs UI work

The HALT_BEFORE_CREATE_TICKETS prompt is a sev-0 invariant. In the manual run, I emitted impl-plan with `tickets_created: false` because no PROJ MCP backend was wired — the question never came up. When the runtime ships AND PROJ MCP is connected AND a real engineer hits "create tickets", the prompt MUST be both noticeable and easy to confirm. UX matters — if it's a tiny "y/n" buried in a chat thread, engineers will just say "y" reflexively.

**Recommended action (runtime build, Phase F supervisor):** The HALT_BEFORE_CREATE prompt should render as a Review primitive with: spec-summary preview, ticket count, total engineer-days, total cost (proxy: tickets × $0.10 PROJ MCP cost), explicit "Yes, create N tickets in {backend}" button vs "No, write markdown only" button. Default to the latter.
