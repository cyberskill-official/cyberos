DEC-096 — Self-audit pass + DEBUG/MAINTENANCE operational modes

Decision

The BRAIN audits itself on every consolidation cycle as a sixth phase (§8.7) running under `.lock`. Six checks: schema validate, supersedes-graph integrity, relationships-graph integrity, audit chain integrity, orphan files, resource caps. Three severity buckets: CRITICAL (freezes writes), WARN (surfaced), INFO (logged).

Three operational modes via `manifest.operational_mode`:

  • `normal` — WARN/CRITICAL in §14 block; INFO not surfaced.
  • `debug` — every reject/revert/warn this session floats to top of next response as a banner.
  • `verbose` — adds successful-op tracing.

    • A separate `maintenance` mode (§8.8) — distinct from DEBUG — is the safe version of "ROOT": time-limited (1 hour or session end), allows specific repair ops normally forbidden (chain rebuild, orphan tombstone, force-resolve, rollback, frontmatter migration edit), each repair requires per-op chat confirmation and is logged with `actor_kind: maintainer` plus a `maintenance_session_id`. NEVER bypasses the §9.3 denylist or §4.2 content gate.

    Org-level escalation (TIER 2) deferred to BRAIN module P1 — when shipped, CRITICAL and aggregated WARN events forward to a CyberSkill admin channel. Privacy boundary: only metadata escalates, never memory content.

    Why

    User question (Stephen, 2026-05-06): *"Can the BRAIN audit itself? While users are using the BRAIN and unexpected issues happen, I should be notified so I can fix it asap. For now maybe we can use DEBUG or ROOT mode."*

    The pre-C protocol had self-audit elements scattered (§4.7 reconciliation, §8.6 source-coverage validator, §13.0 state classifier, §0.4 standing rule, §1.10 user-completeness-challenge response) but no:

  • Periodic full-store integrity scan (orphan files, dangling supersedes, schema drift).
  • Notification channel beyond the §14 block, which is easy to miss when issues compound silently.
  • Clear distinction between read-side verbosity ("show me everything") and write-side elevated permissions ("let me repair this").
  • Recovery toolkit for genuinely-corrupt states (chain breaks, orphan rows, stuck conflicts).

    • The naming distinction matters: DEBUG = verbose alerting (read-side), MAINTENANCE = repair authority (write-side). Conflating them as "ROOT" creates the same blur that turns Linux root usage into a footgun.

    What this NOT

  • Not a free-pass to bypass denylist or content gate (those stay sev-0 inviolable even in MAINTENANCE).
  • Not a vector for prompt injection (mode changes require explicit chat phrase per §0.2).
  • Not org-level today (escalation channel is a TIER 2 follow-on; needs BRAIN module's network surface).
  • Not a substitute for §0.4's standing rule (refinements still come from observed failures, not from health-check spam).

    • Trade-offs accepted

  • **Per-op chat confirmation in MAINTENANCE mode**: friction during recovery. Acceptance: recovery is rare and high-stakes; friction is appropriate.
  • **DEBUG mode banner above every response**: visual noise during training/debugging. Acceptance: this is the explicit purpose of DEBUG mode; user can return to `normal` at any time.
  • **No automatic CRITICAL → freeze write threshold tuning**: a false-positive on schema validation could freeze writes spuriously. Acceptance: `MAINTENANCE` mode is the escape valve; spurious CRITICAL is an §0.4 trigger for refining the validator.

    • How to use this memory

    When the user reports unexpected behaviour, suggest enabling DEBUG mode for the next session to see all rejects/reverts/warns at the top of every response. When the user encounters genuinely-corrupt state (chain break, orphan files), guide them through MAINTENANCE mode entry with the specific repair op needed. When the BRAIN module ships at P1, reopen TIER 2 (org-level escalation) as a follow-on REF.
