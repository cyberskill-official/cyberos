DEC-110 brain_writer.session-end should recompute manifest.memory_count

Context

`cyberos status` dashboard (Aspect 2.1) surfaced a real manifest drift: manifest.memory_count = 81 but real on-disk count = 158 (drift = 77). Investigation showed that brain_writer.cmd_session_end() updates audit_chain_head, last_updated_at, and reconciliation_checkpoint but never recomputes memory_count. Each write op:create increments memory_count drift further until the next manual fix.

Decision

brain_writer.cmd_session_end() MUST recompute memory_count by walking .cyberos-memory/ for *.md files at session end, and include the recomputed value in the manifest str_replace that already runs in step 2.

Alternatives Considered

Alternative 1: Compute on every write (op:create / op:str_replace / op:delete)

  • **Pros:** always-fresh count
  • **Cons:** O(N) cost on every single mutation; serialization bottleneck under concurrent writes
  • **Why not:** session-end is the natural sync point; recompute there matches existing manifest-update cadence

    • Alternative 2: Cyberos status auto-runs manifest-update when drift detected

  • **Pros:** zero brain_writer change
  • **Cons:** status is read-only by contract (HARD GATE per /health pattern); should never mutate
  • **Why not:** violates Aspect 2.1 dashboard's read-only contract

    • Alternative 3: Manifest-bump as separate brain_writer command

  • **Pros:** explicit, no auto-magic
  • **Cons:** every contributor must remember to run it; bug-prone
  • **Why not:** session-end is the canonical sync point; making it implicit there matches §4.7 reconciliation expectations

    • Consequences

    Positive

  • cyberos status dashboard always shows accurate counts
  • Eliminates the recurring "manifest drift" bottleneck (currently appears on every status invocation)
  • Matches the pattern of audit_chain_head + last_updated_at being session-end-updated

    • Negative

  • session-end gains O(N) file walk cost (mitigated: walked anyway during §8.7 self-audit phase 6)
  • One more place that touches manifest — must remain atomic with existing str_replace

    • Risks

  • Concurrent session.end + write race: must hold .lock per §4.9.1
  • Mitigation: session-end already holds BrainLock; this addition is within the same lock scope
