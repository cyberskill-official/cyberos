DEC-099 — Three-way protocol-conflict handling

Decision

§0.5 gains a Three-way conflict subsection covering the case where loaded SHA `Y`, pinned SHA `X`, and upstream-available SHA `Z` all differ. The agent refuses to apply upstream, surfaces a structured prompt offering three explicit user options (revert local; approve local as upgrade; manual three-way merge), and waits for an explicit chat-turn choice. No automated merge.

§13.0 gains a corresponding state-classifier row: `INCOMPATIBLE:three-way-protocol-conflict`. Same handling as 2-way `protocol-sha256-mismatch` (freeze writes; surface diff).

Why

Pre-E §0.5 handled only the 2-way case (loaded vs pinned). The post-cascade analysis surfaced a real risk: a user with hand-edited AGENTS.md who runs "check for protocol updates" could have those edits silently overwritten by an upstream pull. This is exactly the failure mode the protocol's existing §13.0 tripwire was designed to prevent for the 2-way case — extending the same conservative discipline to the 3-way case is the natural follow-on.

How

The Three-way conflict subsection of §0.5 specifies:

  • Detection: an upstream release SHA `Z` AND `manifest.protocol.sha256 == X` AND `canonical_sha(loaded) == Y` AND `X ≠ Y ≠ Z`.
  • Behaviour: state classifier returns `INCOMPATIBLE:three-way-protocol-conflict`; writes frozen.
  • Resolution: user picks one of three options in chat — option 1 reverts local to pinned; option 2 approves local as upgrade (bringing pinned to `Y`); option 3 lets the user merge manually offline and approve the merged SHA via the standard §0.5 phrase.
  • No automated merge: explicit user choice is required.

    • Trade-offs accepted

  • **Friction during 3-way conflicts**: the user must explicitly choose. Acceptance: silent overwrite is worse than friction.
  • **No auto-merge**: simpler protocol; matches the existing 2-way conservative discipline. Auto-merge would require defining merge semantics for arbitrary text diffs, which is infeasible at the protocol layer.
  • **TIER 2 (cross-actor protocol-version skew) and TIER 3 (key rotation operational flow) deferred**: TIER 2 becomes relevant once `brain.publish` ships at BRAIN module P1; TIER 3 becomes relevant once a real signing key is in use (placeholder today). Adopting them speculatively today would add bulk without proportional value.

    • What this does NOT do

  • Does NOT address scenario E (upstream rollback): mechanically works via standard §0.5 flow; no special semantic added.
  • Does NOT address scenario F (upstream tightens validators): Bundle D's additive-only rule is the protection; if CyberSkill violates it, the answer is "fix CyberSkill," not "make the protocol forgiving."
  • Does NOT address scenarios D and G: those are explicitly deferred (TIER 2 and TIER 3 of the proposal).

    • How to use this memory

    When designing the BRAIN module's `protocol.releases.list` MCP endpoint at P1, ensure it returns the SHA + signature + changelog so the agent can run the §0.5 three-way check before presenting upgrade options. When the user reports that "the agent overwrote my hand-edits," verify the §13.0 `INCOMPATIBLE:three-way-protocol-conflict` tripwire fires correctly and the freeze-write behaviour is operational.
