REF-018 — Defer structural complexity until it earns itself; flat folders beat per-major folders at small scale

The original v0.2.0 contracts layout had `<contract-id>/v<n>/<files>` with each major version in its own folder, justified by parallel-version maintenance + self-contained bundles + per-major examples. At registry v0.2.4 the user asked: why folder-per-version instead of flat-folder-with-CHANGELOG? The audit confirmed the answer: the per-major-folder layout was over-engineered for current scale.

What scale assumptions were wrong

The v0.2.0 layout solved a parallel-version maintenance problem we don't have:

  • CyberSkill = 10 employees, 2 active projects.
  • No partner connectors deployed.
  • No tenant operating two majors of the same contract simultaneously.
  • The realistic transition pattern when `feature_request@2` lands is "migrate everything in one batch", not "support v1 + v2 in parallel for months."

    • Solving a problem we won't have for a year (and may never have) costs complexity today: every `pin_path:` carried a dead-weight `/v1/` suffix; every layout diagram in every README documented sub-folder structure that didn't pay for itself.

    How to use this memory

    When designing a new structural decision (folder layout, naming convention, dependency graph), apply this lens:

    1. **Name the failure mode the structure prevents.** ("Without v<n>/ folders, parallel major versions would collide.")

    2. **Estimate when that failure mode actually occurs at current scale.** ("CyberSkill won't run two majors in parallel for at least a year.")

    3. **Estimate the daily cost of carrying the structure.** ("Every pin_path has /v1/ as dead weight; readers see deeper hierarchy than they need.")

    4. **If [time-to-failure-mode] >> [time-to-current-pain], defer the structure.** Add it back when evidence of the failure mode emerges, not preemptively.

    The general pattern: **defer structural complexity until it earns itself**. This is not "YAGNI" applied carelessly — it's "YAGNI calibrated by scale." If you're going to operate at scale where the structure is needed, build it. If you're not, document the simpler structure with an explicit note about the upgrade path.

    When to revisit (the upgrade path)

    Reintroduce the `v<n>/` folder structure when ANY of:

  • A consumer skill is intentionally pinned to v1 while internal skills migrate to v2 (parallel maintenance becomes real, not hypothetical).
  • A partner connector pins a contract version that diverges from internal majors.
  • A contract's example directory grows beyond 10 files and v2 examples are sufficiently different to confuse readers.
  • Three or more skills are blocked on a coordinated v1→v2 migration that would benefit from rolling adoption.

    • When that day comes, the migration is mechanical: `mkdir <contract-id>/v1/`; `mv <contract-id>/{CONTRACT.md,template.md,etc.} <contract-id>/v1/`; create `<contract-id>/v2/` with the new files; update consumers' `pin_path:` declarations.
