FACT-004 — BRAIN three-layer architecture

BRAIN is the per-tenant universal memory layer. Per-tenant means residency-pinned and isolated by RLS; one logical BRAIN per tenant, three storage tiers within each BRAIN.

Layer 1 — Filesystem `.cyberos-memory`

Claude-style local-edge filesystem store. YAML frontmatter + Markdown body per file. The six file operations are the only legal mutators: `view`, `create`, `str_replace`, `insert`, `delete` (soft tombstone), `rename`. Each mutating op produces exactly one Merkle-chained audit row. **Per-subject sync across the subject's machines via the org BRAIN's per-subject mirror** (mechanism in the runtime BRAIN module per §17.5; this filesystem layer is single-master per machine — see AGENTS.md §11.8 + §17). Portable export as a deterministic zip bundle. Anti-injection content gate at the write boundary blocks prompt-injection markers, mixed-script confusables, base64 blobs, and the denylist.

The `.cyberos-memory/` folder we are writing in right now IS Layer 1 — the spec we follow is in `docs/CyberOS-AGENTS.md`. (CyberOS production builds the same contract into BRAIN; the protocol is identical.)

Layer 2 — Vector + graph fact memory

Mem0-style active retrieval substrate. Storage: pgvector with HNSW index for vector search; Apache AGE for graph relationships; PGroonga for multilingual full-text mirror. Four operations: ADD, UPDATE, DELETE, NOOP — applied by an LLM judge with a 0.85 confidence threshold for conflict resolution. Hybrid retrieval pipeline: vector + lexical + graph, then bge-reranker-v2-m3, then context expansion via Anthropic Contextual Retrieval. GraphRAG community summaries provide canonical-entity navigation across communities.

Layer 3 — Archival corpus

Letta-style cold-tier archival of all ingested content (chat threads, project artefacts, KB pages, email summaries, learning records). Used for replay, historical Q&A, and DSAR fulfillment. Stored on S3-compatible object storage with retention rules per classification.

Conflict resolution

Detection per §5.6 — same-scope contradiction with different facts. Personnel/client classifications never auto-resolve; both memories are kept linked `kind:contradicts` and a `conflicts/<YYYY-MM-DD>-<slug>.json` is written. UI presents 4 options: Keep A, Keep B, Keep both as disputed pair, or Edit & replace both. Authority strictly cannot be raised by an agent — only marked down on uncertainty.

Auto-Dream nightly

A nightly consolidation job runs the 5-phase consolidation per §8: Surface candidates, Detect conflicts, Conservative merge (only operational/public, never personnel/client), Reorganise large files, Update manifest. Scaffolding only at P0; first real run at P1 entry.

Performance budgets

p95 brain.search ≤ 250 ms. BRAIN indexes every eligible module write within p95 ≤ 5 s. CUO citation rate ≥ 98% — answers always cite or admit ignorance.

Denylist (DEC-036)

Compensation, government IDs, bank/card numbers, home addresses, health PII, secrets, and external-party PII without consent are structurally excluded at ingestion. If a memory must reference a denylisted item, BRAIN stores a pointer (e.g. "see Vault → folder → entry; held by HR/Ops Lead") instead of the value.

How to use this memory

This is the canonical mental model for BRAIN. When proposing a new feature that touches memory (a new ingestion source, a new query mode, a new scope kind), validate it against the three-layer architecture and the denylist. Performance must hit the p95 budgets above; correctness must hit the citation rate.
