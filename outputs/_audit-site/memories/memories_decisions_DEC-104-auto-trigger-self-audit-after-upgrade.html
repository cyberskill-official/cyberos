DEC-104 — Auto-trigger §8.7 after `protocol_upgrade` + uppercase BRAIN in trigger phrases

Decision

§0.5's `protocol_upgrade` flow gains step 4: **auto-trigger §8.7 self-audit immediately after manifest pin and the protocol_upgrade audit row**. This is the post-upgrade migration check — §8.7 phase 1 (schema validate) already catches memories failing the new §5.1 schema; auto-invocation closes the loop. Skip only with explicit phrase *"skip post-upgrade scan"*. Manual trigger via new `manifest.health_check_policy.post_upgrade_phrase` (default *"rescan BRAIN"*).

Plus a Bundle-H consistency fix: `on_demand_phrase` default updated from *"run brain healthcheck"* (lowercase) to *"run BRAIN healthcheck"* (uppercase), matching §0.3's strict-uppercase BRAIN-alias rule. `diagnostic_verbs[]` entries that mention BRAIN switch to uppercase too. Lowercase versions are NOT diagnostic triggers — they're anatomy/metaphor per §0.3.

Why

Stephen's question post-Bundle-I: *"can we auto trigger scan and re-arrange/refine the .cyberos-memory after AGENTS.md update, because there maybe breaking changes or rules that need to adapt, and how to manual trigger that?"*

The pre-J protocol left a real gap: after `protocol_upgrade`, the new schema applies but old memories haven't been audited for compatibility. A new required field, a tightened validator, a renamed scope — all could leave memories silently invalid. §8.7 already had the schema-validate check that catches this, but it ran only on-demand or at session end. Auto-triggering §8.7 after `protocol_upgrade` makes the migration check non-skippable-by-default.

The uppercase-phrase fix is independent: Stephen also said *"for manual i want 'run BRAIN healthcheck' instead"*. Three default phrases still used lowercase "brain" inconsistent with §0.3 (Bundle H). Updating them now closes the loop.

How

§0.5 protocol_upgrade flow gains step 4 — one paragraph naming what it does and the skip phrase. §6 manifest schema gains `health_check_policy.post_upgrade_phrase`. §8.7 gains a "Post-upgrade scan" subsection distinguishing report naming. §1 step 2's diagnostic-verb list updates to uppercase BRAIN. §6 manifest defaults all switch to uppercase BRAIN.

The cyberos running manifest's `on_demand_phrase` and `diagnostic_verbs[]` also update as part of Bundle J's manifest re-pin (so this protocol upgrade's own self-audit will use the new uppercase phrase if Stephen invokes it).

Trade-offs accepted

  • **Auto-trigger adds latency to every protocol_upgrade**: a §8.7 pass (~0.5s on a small BRAIN; ~3s on a large one). Acceptance: the migration check is too important to defer. Skip phrase exists for when the user knows the upgrade is purely cosmetic.
  • **Project-level overrides not migrated**: if a project configured `on_demand_phrase: "run brain healthcheck"` (lowercase) before Bundle J, it stays that way. Only the protocol's DEFAULT changes. Acceptance: consistent with Bundle H's general principle (uppercase is the default; per-project override always allowed).
  • **Two report-file naming conventions** (`<date>-<sha>.md` vs `<date>-<sha>-postupgrade.md`): minor naming complexity. Acceptance: provenance traceability outweighs.

    • What this does NOT change

  • The six §8.7 checks themselves — unchanged.
  • The audit ledger format and chain semantics — unchanged.
  • The §14 block format (Bundle I) — unchanged.
  • Existing manifest project-level overrides — preserved as-is.

    • How to use this memory

    When the user approves a protocol upgrade, expect a §8.7 pass to run automatically as the migration check. Surface findings per §8.7 severity routing. When the user wants to manually re-scan after a protocol change (e.g., they pulled an upstream release), say *"rescan BRAIN"* — equivalent to the auto-trigger but on-demand.

    When designing future schema changes, remember the auto-trigger means breaking changes will surface immediately rather than at next consolidation. Use the §0.4 propose-then-adopt loop accordingly: propose the schema change, predict which existing memories will fail new validators, prepare migration recommendations BEFORE adoption.
