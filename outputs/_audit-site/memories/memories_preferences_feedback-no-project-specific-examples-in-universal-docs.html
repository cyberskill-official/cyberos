No project-specific examples in universal protocol/spec documents

**Rule.** Documents that ship as universal protocol or spec (e.g., `CyberOS-AGENTS.md`, `CyberOS-PRD.docx`, `CyberOS-SRS.docx`) MUST NOT carry project-specific values in their canonical examples. Schema definitions, generic placeholders, and default values are fine. Concrete patterns naming a specific project's actual scopes (e.g., `module:whatsapp-miguel-dm`, `module:notion-styx-data-room`) are forbidden — they teach future readers that those patterns are universal defaults when they're project-bespoke configuration.

**Where project-specific values DO belong:**

  • Per-project `manifest.json` files (each project's actual configuration).
  • Per-project `memories/` files (project-specific decisions, refinements, drift records).
  • Per-project README or `meta/` documentation.

    • **Where they DO NOT belong:**

  • Protocol body (AGENTS.md sections).
  • PRD/SRS body — even when prose is "explanatory," the example values become reference material.
  • CHANGELOGs (use generic descriptions; the project-specific values live in their respective manifests).
  • Skill definitions, persona registries, or any document that ships across projects.

    • **Why.** Universal docs are read by every project. Project-specific examples mislead any project that's NOT the original author's. The Styx `whatsapp-*-dm > whatsapp-*-group > notion-*` tier order is correct for Styx; meaningless in CyberOS; positively misleading in some hypothetical third project. Keep universal docs structurally generic; let each project's manifest declare its own truth.

    **How to apply.**

  • Before adding any concrete value to a universal-doc example, ask: "is this a schema/structure/default, or is this a specific project's configuration?" If specific → put it in the project's manifest, not the universal doc.
  • When proposing illustrative examples, prefer angle-bracket placeholders (`<scope-glob>`, `<int ≥ 1>`) over real-looking strings (`module:whatsapp-*-dm`).
  • Always include a single literal default fallback (e.g., `{"pattern": "*", "tier": 99}`) so readers see the schema with at least one minimal valid value.
  • When reviewing universal docs, search for module/scope strings that look project-specific (whatsapp, notion, jira, slack, etc.) — those are red flags.

    • How to use this memory

    Load every session-start. Apply on every edit to AGENTS.md, PRD/SRS, or any other cross-project doc.
