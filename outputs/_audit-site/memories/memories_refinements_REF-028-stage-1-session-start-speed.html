REF-028 — Stage 1: Session-start speed

Trigger

Local-optimization plan (`docs/CyberOS-AGENTS.LOCAL-OPTIMIZATION.md`) Stage 1 surfaced as the highest-ROI improvement for daily BRAIN use: §4.7 reconciliation walks all audit rows newer than the last `consolidation_run` on every session start, which is O(N) over chain length. With ~290 audit rows already in the live store and growth ~10/day, session-start latency was creeping into multi-second territory.

Adoption

Adopted via §0.5 chat-turn approval (2026-05-09):

> approve protocol upgrade to sha256:576368647e4d17635804580ca4dded28721b1c7247f0a19666ce43f5f0eb911a

Changes (5 additive amendments)

  • **Change A — `manifest.reconciliation_checkpoint`** (§6 extension). Records the `audit_id` + `chain` + `ts` of the most recent successful `op:"session.end"` or `op:"consolidation_run"`. Updated atomically with those writes; never edited independently.
  • **Change B — §4.7 reconciliation amendment**. Walks rows newer than `manifest.reconciliation_checkpoint.audit_id` if set; falls back to full-walk on missing/stale (>30 days) checkpoint or chain-mismatch. Stale fallback emits `op:"warn" reason:"stale-checkpoint"`.
  • **Change C — `manifest.read_profile`** (§6 + §10). Declares eager vs lazy scopes. Default `eager_scopes: ["meta"]`; everything else lazy-loaded on first reference.
  • **Change D — §5.1 frontmatter compactness rule**. Write-side: omit `null`/empty optional fields except those required by classification/tombstoning. Read-side: accept both compact and verbose forms.
  • **Change E — §8.7 phase 4 stale-checkpoint check**. If checkpoint set, confirm `audit_id` resolves in ledger AND `chain` matches that row's `chain`. Mismatch → CRITICAL.

    • How to use this memory

    When the next-session-start latency regresses or the §4.7 fallback fires unexpectedly, reference this REF and the proposal at `docs/proposals/STAGE-1-PROTOCOL-UPGRADE.md`.
