Legacy memory_ids (predates §5.2 validator)

This is a **closed-set registry** of memories that were created before the §5.2 UUIDv7/ULID `memory_id` validator landed in CyberOS-AGENTS.md. Each line below documents one memory whose `memory_id` does not match the §5.2 regex but which is permitted to retain its mnemonic ID in lieu of tombstone-and-recreate.

**Format:** `<mem_id> | <originating_path> | <originally_created_at> | <reason>`

**Closed-set rule:** new entries to this registry land **only** via a §0.5 protocol upgrade. Routine BRAIN writes that produce a non-conforming `memory_id` MUST still be rejected per §5.2.

**Validator behaviour:** when reading a memory whose `memory_id` does not match the §5.2 regex, the validator looks up the ID in this registry. Found → accept and continue. Not found → reject as `op:rejected reason:invalid-memory-id`.

**Frontmatter exemption:** this file is itself a registry, not a memory. It carries no §5.1 frontmatter, by the same convention applied to `meta/tombstones.md`. §4.2 denylist regex is also skipped on this file (§4.2 rule-definition exemption set). The §4.2 injection gate still runs.

**Authority:** entries here represent a `human-edited` decision (the §0.5 chat-turn approval that introduced this registry). The four entries below were registered in the 2026-05-07 protocol upgrade that introduced this carve-out; see the `Bundle L TIER 2` entry in `docs/CyberOS-AGENTS.CHANGELOG.md` for context.
