REF-038 — Bundle N TIER 1+2: §14 omission + audit-trail suppression

Trigger

User feedback during cowork-session 2026-05-10. After running an on-demand §8.7 healthcheck (which generated a §14 block including the audit-trail head SHA), Stephen surfaced two complaints:

1. *"show Audit trail after each messages make the conversation flooded, just show in maintenance mode or when issues arise"* — the `audit/<YYYY-MM>.jsonl: <N rows appended; head=sha256:…>` line is noise on healthy turns.

2. *"can we compress 📝 .cyberos-memory updated section more? show full verbose in maintenance mode, but only show changes summary on normal (default), if no issues arise don't need to show memory changes, just show other files' changes"* — the entire §14 block should be omittable when nothing meaningful changed.

Both are §0.4 standing-rule triggers (drift between protocol verbosity and user's actual signal-needs).

Adoption

Adopted via §0.5 chat-turn approval (2026-05-10):

> approve protocol upgrade to sha256:8060fe2e188e1793e9dbc758b34a8198617ff8bf8a3320a2012595faf3012dab

Tier choice: **TIER 1 + TIER 2** (minimum viable). TIER 3 (`📁 Files changed:` block for non-BRAIN paths) deferred — Stephen did not include it in this approval and can request as a separate Bundle.

Changes (2 added; 0 deferred for this Bundle)

  • **TIER 1 — §14.0 omission rule** (sev-2). Inserted above §14.1. Skip the entire `📝 .cyberos-memory updated` block when ALL of: (a) `manifest.operational_mode == normal`, (b) no `op:rejected|revert|warn|health_check` row this turn, (c) no memory mutations (audit-row count ≤ 2 and only optional session bookends), (d) latest §8.7 self-audit reports 0 CRITICAL and 0 WARN. Verbose/debug/maintenance modes always emit §14.2 regardless.
  • **TIER 2 — §14.1.1 audit-trail suppression** (sev-2). Inserted under §14.1. When §14.1 compact IS emitted in normal mode, omit the `audit/<YYYY-MM>.jsonl: …` line unless a finding occurred this turn or latest §8.7 reports issues. Verbose/debug/maintenance always emit the audit line (it's already part of §14.2).

    • The §14 heading also changed from `(mandatory)` to `(conditional in normal mode)` to reflect the new conditionality.

    • Why this works

    The §14 block has always been a signal-to-noise compromise: detailed enough to be auditable, terse enough to not flood chat. Bundle I (2026-05-06) introduced the compact format gated by `operational_mode`; Bundle N completes the journey by recognising that on a healthy chain with nothing meaningful happening, the most signal-respectful output is no output at all. The four §14.0 conditions are narrow enough that any non-trivial event (rejection, revert, warning, mutation, finding) immediately re-enables visibility.

    How to use this memory

    The next agent reading this BRAIN must check the §14.0 conditions BEFORE writing the §14 block, and the §14.1.1 conditions BEFORE adding the audit-trail line. The conditions are cheap to evaluate (per-turn counters maintained during the session). When ambiguous, default to omission — Stephen's stated preference is "less is more". If the user asks "what changed?" and the §14 block was omitted, run the on-demand healthcheck phrase to surface state explicitly.
