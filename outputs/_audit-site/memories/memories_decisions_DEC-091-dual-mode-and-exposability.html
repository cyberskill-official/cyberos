DEC-091 — Dual-mode invocation + exposability (registry v0.2.0)

Every workflow skill is **one function body with two front doors**: standalone (chat-mode entry) and chained (envelope from upstream skill). Plus every skill declares which surfaces it ships through via `exposable_as:` (internal / agent_plugin / mcp_tool / partner_connector).

What landed in frontmatter

invocation_modes: [standalone, chained]   # workflow skills
invocation_modes: [persona_routing_only]  # persona cards (no envelope)

expects:
  optional_fields: [...]                  # NEW: distinct from required
  standalone_interview_ref: ./STANDALONE_INTERVIEW.md  # NEW: chat-mode entry script

produces:
  human_summary_ref: ./HUMAN_SUMMARY.md   # NEW: chat-mode output template

exposable_as:
  internal:           true     # CUO supervisor route (default true)
  agent_plugin:       true     # ships in plugin bundles
  mcp_tool:           true     # auto-emit MCP tool descriptor
  partner_connector:  false    # GATED — needs separate per-skill DEC

The dual-mode mechanic

Mode detection (deterministic at runtime):

  • Standalone when ALL of: no `pipeline_run_id` in call context, caller is the CHAT primitive, input envelope is empty.
  • Chained otherwise.

    • Standalone path: supervisor reads `STANDALONE_INTERVIEW.md`, asks 3-5 questions, fills `expects.required_fields` from chat answers, then invokes the function body. After completion, renders `HUMAN_SUMMARY.md` to chat. Chained path: supervisor consumes the upstream envelope, invokes the function body, emits the output envelope to the next skill. **Same function body in both paths** — this is the key property.

    The exposability mechanic

    The build pipeline reads `exposable_as:` and emits per-surface artefacts:

    | Surface | What it produces | Audience |

    | --- | --- | --- |

    | `internal` | SKILL.md the CUO supervisor routes to | CyberSkill employees |

    | `agent_plugin` | `.plugin/` bundle (skills + commands + .mcp.json) | Claude Code / Antigravity / Codex / Cursor users |

    | `mcp_tool` | `tool.json` + stdio runner stub | Any MCP-compatible agent |

    | `partner_connector` | Hosted MCP + OAuth + tenancy + billing | External partners on a marketplace |

    `partner_connector: true` requires a separate per-skill DEC before the flag flips — partner exposure has trust, billing, and SLA implications that need explicit governance.

    Trust↔exposability link

    A skill with `partner_connector: true` MUST also have `confidence_band.cite_sources: required`, `untrusted_inputs.injection_scan: required`, `self_audit.invariants_ref` populated with ≥3 invariants including INV-`scope-discipline` and INV-`fabrication-boundary`, and `human_fine_tune.review_required.on_safety_change: true`. Validator enforces these as a precondition.

    How to use this memory

    When designing or reviewing a CyberOS skill:

  • A skill that's "chained-only" (no chat surface) declares `invocation_modes: [chained]` and omits the interview/summary refs. Rare — most skills should be invokable both ways.
  • Don't conflate `agent_plugin` and `mcp_tool`. Plugins are bundles that include skills + agents + commands + MCP servers; MCP tools are a single tool descriptor surfaced to a host's tool registry. A skill can be both, neither, or one.
  • The build pipeline's transpilers (`ccsm-to-anthropic`, `ccsm-to-mcp-tool`, `ccsm-to-claude-plugin`, etc.) read `exposable_as:` to decide which targets to emit. See DEC-091's carrier specification in registry README Part 9.
