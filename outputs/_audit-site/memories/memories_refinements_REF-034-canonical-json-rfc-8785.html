REF-034 — Canonical-JSON tightening (§7.2 → RFC 8785 JCS)

Real-world trigger

The 2026-05-06 cascade's verifier (verify_v2.py) surfaced 149 pre-existing audit rows failing bit-perfect hash recompute against the new `brain_writer.py`, despite both writers nominally following §7.2's "keys sorted, compact separators, shortest IEEE-754" wording. LINK integrity remained intact, but the hash divergence revealed §7.2 as underspecified — multiple legal interpretations existed.

Surfaced as a TIER 1 §0.4 candidate at the end of the prior turn; user adopted Bundle D in the next turn alongside the rest of the post-cascade work.

Refinement adopted

TIER 1 (cite RFC 8785 JCS as authoritative; document the four common divergence points; clarify LINK-vs-recompute distinction). No TIER 2 or TIER 3 needed — the cardinal rule (additive-only) is preserved because pre-D rows are not retroactively touched.

What changed in AGENTS.md

§7.2 rewritten:

  • Now cites RFC 8785 JCS in the section title and opening.
  • Bullet list expanded with: object key ordering rule (UTF-16 code units), whitespace rule (none), separator bytes, string normalisation (NFC + non-ASCII preserved), number serialisation (ECMAScript `Number.prototype.toString` = shortest round-trip; integers without `.0`; explicit Python `1.0` → `1` warning), boolean/null lowercase rule, no-duplicate-keys rule.
  • Added "Reference implementations" paragraph naming `rfc8785` PyPI and `canonicalize` npm.
  • Added "Cross-writer-version compatibility" paragraph clarifying that LINK integrity is authoritative and hash recompute is informational.
  • The body-exclusion clarified explicitly: canonical_json receives `row_without_chain_or_prev_chain`; `prev_chain` is concatenated as raw bytes AFTER the canonical body.

    • Why this is the right fit

    The fix is specific, observed-failure-driven, and bounded. It does not require any pre-existing memory or audit row to be migrated, re-chained, or re-validated. Future writers in any language can pull a JCS library and Just Work. Implementations that hand-roll `json.dumps` get a clear test-vector requirement and four named gotchas to validate against.

    Trade-offs

  • Implementations using Python's `json.dumps(sort_keys=True, …)` will produce `1.0` for floats, which is NOT JCS-conformant. Mitigation: this REF + DEC-097 names this as the most common divergence, so implementers know to address it before chaining audit rows.
  • Pre-D rows remain hash-non-reproducible. Acceptance: LINK integrity is preserved; forcing re-chain would invalidate any external exports already pinned to those chain values.

    • How to use this memory

    When auditing a hash mismatch: first verify LINK integrity (`row[N].prev_chain == row[N-1].chain`); if intact, the mismatch is a writer-version artefact and is INFO-severity per §8.7. When adding a new writer language to the project: pull a JCS-conformant library; run JCS test vectors; only then trust the writer with audit-chain authority.
