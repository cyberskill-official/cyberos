DEC-098 — §0.5 rollback flow validated end-to-end

Decision

The §0.5 rollback flow is validated end-to-end by a controlled round-trip test on 2026-05-06. The flow steps (verify archive SHA → copy archive over loaded path → update manifest pin → append `op:"protocol_rollback"`) work correctly; chain LINK integrity is preserved across the round-trip; final state is byte-identical to the starting state.

Why

§0.5 introduced `op:"protocol_rollback"` and the `meta/protocol-history/` archive flow as the safety net for §0.5 protocol upgrades. Until validated end-to-end, rollback was an unexercised feature — exactly the kind of thing that fails silently until disaster strikes. The DEC-098 controlled test exercised the full flow on a benign HTML-comment marker in AGENTS.md so the result is verifiable and recoverable.

Test methodology

1. Saved the current post-D AGENTS.md verbatim to `meta/protocol-history/AGENTS-sha256-b4042a673bff3192705f201564665ac1e509345058ecea3c66a5e9890c2acce3.md`.

2. Appended a benign HTML comment `<!-- ROLLBACK_TEST_2026-05-06 -->` to AGENTS.md and computed the new canonical SHA.

3. Ran `op:"protocol_upgrade"` (post-D → post-test-edit).

4. Ran `op:"protocol_rollback"` (post-test-edit → post-D), restoring AGENTS.md from the archive.

5. Verified: byte-identical to original; test marker removed; manifest pin matches loaded SHA; chain LINK intact across the entire ledger.

All five verification checks passed.

Operational implications

  • Rollback IS available for any post-D protocol upgrade (Bundle B/A/C/D's pre-state archives are markers, not verbatim, so they are NOT directly rollback-able — but going forward, every protocol_upgrade flow that follows the canonical archive-before-edit sequence creates a verbatim archive that supports rollback).
  • Operators may now confidently approve protocol upgrades knowing the safety net is real.
  • The first verbatim archive in `meta/protocol-history/` is `AGENTS-sha256-b4042a673bff3192705f201564665ac1e509345058ecea3c66a5e9890c2acce3.md` (post-D state captured during this test).

    • What this does NOT do

  • Does NOT make Bundle B/A/C/D rollback-able. Their pre-state archives are markers (the prior AGENTS.md content was edited in-place during the cascade). To rollback past Bundle D would require reconstructing from git history, not from `meta/protocol-history/`.
  • Does NOT validate concurrent rollback (multi-machine). That belongs to the BRAIN module's domain (P1+).

    • How to use this memory

    When approving a future protocol upgrade, trust §0.5's rollback semantics — they are validated. When designing a future protocol-evolution refinement, remember that the canonical sequence is **archive BEFORE edit** so the resulting `meta/protocol-history/` file is verbatim and rollback-capable; in-turn cascade edits that produce markers are an exception (acceptable for the originating session but limit later rollback granularity).
