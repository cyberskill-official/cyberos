REF-016 — Always run audit-fix-audit after creating a new contract; the first draft will drift from reality

When a new contract is registered to capture a previously-undocumented convention (e.g. NATS subject names that skills already emit), the contract author has only the pre-existing reality as the source of truth. It is very easy to over-specify the contract — adding structure that sounds right in isolation but doesn't match what the code actually does. v0.2.2 caught exactly this: the first draft of `nats_subjects@1` named subjects as `<sub-persona>.<skill>.<event>` (e.g. `cuo_cpo.fr_create.fr_written`), which is a defensible naming rule. But the existing skill bodies have always emitted `<top-level-persona>.<skill>.<event>` (e.g. `cuo.fr_create.fr_written`), and the supervisor routes at that level. The contract over-specified.

How to use this memory

When you register a new contract that documents a pre-existing convention, after writing the first draft, run a focused audit pass that grep-searches the consuming skill bodies for the convention as the contract describes it. If the grep returns nothing (or returns the wrong form), the contract has drifted from reality. Fix the contract, not the skills. Reality wins for any contract whose purpose is "document what already exists." The only exception is when the convention itself is being intentionally tightened — in which case the contract bump must be MAJOR and accompanied by a coordinated update to every consuming skill.

The audit-fix-audit loop pattern (audit → fix → re-audit until clean) is the right discipline for any contract registration. v0.2.2 took 4 rounds: Round 1 surfaced the naming + anchor + schema-description drift; Round 2 caught residual `cuo_cpo` in schema.json descriptions and protocol.md prose; Round 3 caught one stale CHANGELOG line; Round 4 confirmed clean. Each round took ~1 minute. The cost of running the loop is much smaller than the cost of shipping a contract that drifts from reality.
