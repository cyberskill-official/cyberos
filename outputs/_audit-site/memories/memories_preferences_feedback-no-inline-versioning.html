memories/preferences/feedback-no-inline-versioning.md

No inline versioning in docs we maintain; use separate dated CHANGELOG.md files

**Rule.** For every design document maintained in the cyberos/styx project repos (AGENTS.md, PRD.docx, SRS.docx, future docs), the body MUST NOT carry inline version markers like "v1.0.0", "v1.1.0", "since v1.0.0", "as of v1.0.0", "added in v1.1.0", or "(this is new in v1.1.0)" — improvements land day by day and these markers compound into noise.

**The body answers "what is true now?" The CHANGELOG answers "what changed when?"**

**Where to put change history:**

  • One `<DocName>.CHANGELOG.md` per maintained document, sibling-located in the same `docs/` folder.
  • Format: dated sections, not version-numbered. Inspired by Keep a Changelog but date-stamped.
  • Each entry: ## YYYY-MM-DD — short title; ### Added / ### Changed / ### Removed sub-headings; reference back to relevant DEC entries.

    • **Exemptions (legitimate version markers):**

  • `manifest.schema_version` value in JSON examples — that's a machine field, not narrative versioning.
  • References to EXTERNAL documents that have their own versioning (e.g., "CyberSkill Global Design System v1.0.0", "Anthropic memory_20250818 tool spec", "PDPL Decree 13/2023") — keep.
  • Standards / spec versions ("MCP 2025-11-25", "Apollo Federation v2", "PostgreSQL 17") — keep.

    • **Why.** AGENTS.md §0.2 already mandates "Versioning of the protocol document is tracked via the project's git history — no inline version markers required" for AGENTS.md itself. Stephen extended this rule to ALL maintained design docs on 2026-05-04 after seeing my v1.1.0 amendment leak version commentary into PRD/SRS body text. The extension is consistent with the original §0.2 intent.

    **How to apply.**

  • Before adding any version-style commentary to a doc body, ask: "is this a change-history note?" If yes → CHANGELOG. If "what is true now?" → body.
  • When amending a doc, write the body change cleanly + add a dated CHANGELOG entry referencing the DEC.
  • When ingesting new content into an existing doc, do not preface with "v1.x.x adds…" — just write what's now true.

    • How to use this memory

    Load every session-start. Apply on every doc edit before saving.
