DEC-090 — Skills↔contracts namespace split (registry v0.2.0)

Versioned schema artefacts (artefact templates, envelope JSON schemas, wire protocols) move out of skill folders into a sibling namespace `cyberos/docs/contracts/<contract-id>/v<n>/`. Skills declare consumption via a new `depends_on_contracts:` frontmatter field. The build pipeline ships skill + declared contracts as one bundle.

The architectural distinction

A **skill** acts: takes input, runs LLM inference or deterministic work, emits output, writes an audit row. A **contract** constrains: declares the shape of an artefact, envelope, or wire protocol that one or more skills produce or consume. Conflating the two produced "skills" with empty `allowed_mcp_tools: []`, `expects: null`, `confidence_band: 1.0`, `produces.output_kind: artefact` — schemas wearing skill costumes. v0.2.0 ends that conflation.

Concrete migration

`cuo/_shared/feature-request-template/` → `cyberos/docs/contracts/feature-request/v1/`. `SKILL.md` → `CONTRACT.md` with smaller frontmatter (drops `allowed_mcp_tools`, `expects/produces`, `audit`, `confidence_band`, `untrusted_inputs`, `gated_until_phase`; gains `contract_id`, `contract_version`, `contract_kind`, `template_literal`, `steward_persona`, `escalation_on_breach`, `moved_from`). Body of `template.md` byte-identical. Old folder deleted. Every consumer updated: `cuo/cpo/fr-create` (`depends_on_contracts:` pinning v1) + `cuo/cpo/fr-audit` (same).

Three contract kinds

| `contract_kind` | Schema body | Example |

| --- | --- | --- |

| `artefact_schema` | `template.md` | `feature-request@v1` |

| `envelope_schema` | `schema.json` | future pipeline-checkpoint contracts |

| `wire_protocol` | `schema.json + protocol.md` | future genie.action_log row format |

Why this matters

When a skill ships to Antigravity / Codex / a partner connector, the build pipeline needs to package skill + its contract dependencies as one bundle. `depends_on_contracts:` makes that machine-readable. Without it, the bundler would have to grep skill bodies for path strings — fragile and silent on miss. The validator can now reject any skill whose body references a contract not declared in frontmatter.

How to use this memory

When making structural decisions on `cyberos/docs/skills/` and `cyberos/docs/contracts/`:

  • A new schema-shaped artefact (a template, a JSON Schema, a wire format) goes in `contracts/`, NOT in `skills/_shared/`.
  • A genuine cross-persona reusable **skill** (e.g., the future `draft-payslip-explanation` per DEC-061) still goes in `skills/cuo/_shared/`.
  • Test: empty `allowed_mcp_tools` + `expects: null` + `confidence_band: 1.0`

    • + `produces.output_kind: artefact|null` ⇒ it's a contract, not a skill. Move it.

  • Recipe 7 in `cyberos/docs/skills/README.md` Part 13 walks the promotion procedure end-to-end (used for the `feature-request@v1` promotion).
