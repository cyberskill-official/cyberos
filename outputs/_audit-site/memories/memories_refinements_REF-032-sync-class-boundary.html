REF-032 — Sync-class boundary (§17)

Real-world trigger

User question (Stephen, 2026-05-06): *"Currently it works as personal memory for local machine for one person. But in the future each person will contribute to CyberSkill activities (via CyberOS), so it needs to serve both personal-based memory as well as CyberOS's memory. Should we think about that now and how to implement that?"*

Diagnosis surfaced two pre-existing gaps:

1. §11.8 ↔ FACT-004 contradiction (concurrent multi-machine: unsupported vs CRDT sync). An agent reading both got contradictory signals; firing as soon as a second laptop joins.

2. Personal-vs-org boundary was implicit. Every memory written today was being classified by accident; retroactive classification at scale would have been very expensive.

A quieter Gap 3 (no identity layer for cross-actor flow) was scoped out to a future REF — the field `signed_by: ed25519` already exists in §5.1; making it mandatory is a separate refinement that earns its keep when the BRAIN module is being built.

Refinement adopted

TIER 1 (lock the boundary; resolve the contradiction). TIER 2 (signing mandate) deferred. TIER 3 (operational on-ramp) shipped via the README on-ramp doc.

What changed in AGENTS.md

  • New §17 (Personal vs shared memory boundary) with four sync classes and the per-scope defaults table.
  • §5.1 frontmatter gained 28th field: `sync_class:`.
  • §11.8 rewritten: "This protocol governs the personal layer of the BRAIN. Continuous multi-machine sync of shared scopes happens through the runtime BRAIN module (FACT-004 Layer 2)." Resolves the CRDT-sync contradiction.
  • §14 end-of-response block gained a sync-class-summary line.

    • User answers driving the design

  • Q1 *"Is CyberSkill one tenant?"* → CyberSkill is publisher, but the design stays flexible (multi-tenant SaaS at P3+ supported via per-tenant region pinning per FACT-004).
  • Q2 *"Does project/ flow to org?"* → Yes. project/ defaults to `shared`. CyberOS architecture is the company's product.
  • Q3 *"Do clients consume a slice?"* → Yes. Fourth class `client-visible` defined; default scope `client/<id>/portal-visible/`.
  • Q4 *"Offboarding semantics?"* → Absorb knowledge; discard personal fragments. Codified as the absorb-then-discard pattern.
  • Q5 *"Per-machine or per-person?"* → Per-person. Subject is the trust anchor; multiple machines mirror through the org BRAIN per-subject.

    • How to use this memory

    When proposing or evaluating future multi-person refinements, read this REF + DEC-095 together. The TIER 2 (signing mandate) follow-on refinement should land when the BRAIN module's `brain.publish` flow is being designed.
