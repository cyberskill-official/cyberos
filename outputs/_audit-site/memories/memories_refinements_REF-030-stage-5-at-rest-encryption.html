REF-030 — Stage 5: At-rest encryption + Shamir 3-of-5 escrow

Trigger

Local-optimization plan (`docs/CyberOS-AGENTS.LOCAL-OPTIMIZATION.md`) Stage 5 — make sensitive `personnel`/`client` memories safe to share via filesystem (lent laptop, contractor backup, machine handoff) without rewriting them. The §9.3 denylist already structurally excludes the highest-stakes content (comp/ESOP/secrets); encryption is the second-tier protection for content that's allowed to be stored but shouldn't be readable from a snapshot.

Adoption

Adopted via §0.5 chat-turn approval (2026-05-10):

> approve protocol upgrade to sha256:d3ce9764ac76635921f6e981a713ea8822eaec442d01200930633a805a84aaf0

Decisions baseline (per `docs/proposals/STAGE-5-OPEN-QUESTIONS.md`, "go with your recs"):

  • **Q1 (c) Sensitive-plus-private**: encrypt `member/<self>/private/` + classification: personnel | client
  • **Q2 (c) Both — HW key OR Argon2id passphrase**: ≥16 chars + zxcvbn ≥3
  • **Q3 3-of-5 Shamir, wizard-time holders, no deadman switch**
  • **Q4 (a) Body-only encryption**: frontmatter stays plaintext for index compatibility
  • **Q5 (c) User-paced migration**: `--migrate-batch 50` cadence

    • Changes (6 additive amendments)

  • **Change A — §5.6 (new) at-rest encryption envelope.** Five sub-sections: §5.6.1 per-file format (XChaCha20-Poly1305-IETF, 24-byte nonce, AAD `sha256(memory_id || last_updated_at)`); §5.6.2 key derivation (HKDF-SHA256 from HW-bound — Secure Enclave / TPM 2.0 / FIDO2 hmac-secret — OR Argon2id `t=3, m=64MiB, p=4`); §5.6.3 mandatory Shamir 3-of-5 escrow (refuses enable until 5 fragments distributed; fragments NEVER stored in BRAIN, only fingerprints); §5.6.4 indexability (frontmatter stays plaintext); §5.6.5 audit-chain compatibility (`after_hash` over plaintext).
  • **Change B — §6 manifest extensions.** New `encryption_policy` block (default `enabled: false`; scopes: `member:<self>/private`, `classification:personnel`, `classification:client`) and `shamir_fragments` block (default empty; `threshold=3, total=5`). Mutation requires wizard or §0.5 chat-turn approval.
  • **Change C — §7.1 op enum +8.** New ops: `ledger_compact`, `ledger_decompact` (Stage 6 normalisation), `encryption_policy_change`, `key_rotation`, `key_recovery_initiated`, `key_recovered`, `shamir_rotation`, `shamir_distribution_confirmed`.
  • **Change D — §4.6 encrypted-tombstone semantics.** Encrypted bodies stay base64-ciphertext after `delete`; routine reads skip them; only MAINTENANCE-mode hard-erase decrypts.
  • **Change E — §9.3 denylist clarification.** Encryption is NOT a denylist softener. Content gate runs BEFORE envelope; comp/ESOP/gov-IDs/secrets stay forbidden in ANY storage form.
  • **Change F — §17.6 cross-link refresh.** `meta/key-policy.md` now covers signing keys AND encryption master keys; rotation events audited via `op:"key_rotation"` + `op:"shamir_rotation"`.

    • What's dormant until first triggered

    Encryption is OFF on this store. The protocol primitives are landed; the actual encryption flow requires a separate `cyberos-encrypt enable` wizard run that:

    1. Detects HW key path (Secure Enclave on macOS / TPM 2.0 on Windows / TPM 2.0 or FIDO2 on Linux)

    2. Optionally accepts an Argon2id passphrase as the alternative path (≥16 chars + zxcvbn ≥3)

    3. Generates 32-byte master key via CSPRNG

    4. Shamir-splits via `vsss-rs` (Rust) / `secretsharing` (Python) — 3-of-5 threshold

    5. Renders 5 fragments as printable QR codes + base32 strings

    6. Walks holder distribution wizard (label each fragment, prompt for "press y when handed off"); appends `op:"shamir_distribution_confirmed"` per fragment

    7. Verifies master-key fingerprint pinned in `meta/key-policy.md`

    8. Atomically `str_replace`s manifest.json: flips `encryption_policy.enabled = true`

    9. Appends `op:"encryption_policy_change"` audit row

    After enable, new in-scope writes encrypt automatically. Existing 82 plaintext memories migrate via `cyberos-encrypt migrate-batch 50` at user pace.

    Implementation work this unblocks

    No further §0.5 approvals needed for any of these:

  • `runtime/tools/cyberos_encrypt.py` (~600 LOC) — enable wizard, disable, migrate-batch, rotate-shamir, recover, status
  • `cyberos_validate.py` extension — recognise `encrypted: true`; verify chain integrity using stored `after_hash` over plaintext without needing the key; `encryption-aad-mismatch` finding; `shamir-fingerprint-missing` finding
  • `cyberos_doctor.py` — new `R6-rotate-master-key` repair op (re-derive master from new HW key source after hardware change; re-encrypts all in-scope memories under MAINTENANCE mode)
  • `docs/cookbook/encryption-and-recovery.md` — operational guide: enable wizard walkthrough, holder selection guidance, recovery flow walkthrough, migration playbook, hardware-key change procedure, threat model

    • How to use this memory

    When you decide to actually enable encryption: read this REF first, then run the wizard. The Shamir 3-of-5 fragment distribution is the most important part of the operational flow — pick holders carefully (founder + spouse + business partner + lawyer + geographically-distant trusted contact is a defensible pattern), and store fingerprints in `meta/key-policy.md` so future agents can verify recovered fragments against the pinned hash.
