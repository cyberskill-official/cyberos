Scope rules

Aspect 12.2 of the Layer-1 improvement catalog. Project-specific rules

that constrain which classifications + sync-classes a memory may carry

based on its scope prefix.

Loaded by `meta/validators/check-scope-rules.py` (auto-discovered per

Aspect 12.1) and enforced on every `cyberos verify` pass.

Schema

Each rule block is a top-level `## scope: <prefix>` section. Inside:

  • `classification:` — `allow` whitelist OR `deny` blacklist (one or the other)
  • `sync_class:` — `allow` whitelist OR `deny` blacklist
  • `authority:` — minimum authority tier required

    • Unknown keys are ignored (forward-compat).

    Rules

    scope: company

  • classification:

    • deny: [personnel]

  • sync_class:

    • allow: [shared, publishable]

  • authority:

    • minimum: human-edited

    scope: persona

  • classification:

    • allow: [operational, public]

  • sync_class:

    • allow: [shared, publishable, local-only]

    scope: memories/people

  • classification:

    • allow: [personnel]

  • sync_class:

    • allow: [local-only, shared]

    scope: memories/decisions

  • sync_class:

    • deny: [client-visible]

  • authority:

    • minimum: human-confirmed

    scope: memories/refinements

  • sync_class:

    • deny: [client-visible]

  • authority:

    • minimum: human-confirmed

    scope: client

  • classification:

    • allow: [client]

  • sync_class:

    • deny: [publishable, shared]

    Notes

  • `authority:minimum` follows the tier order: human-edited > human-confirmed > llm-explicit > llm-implicit. A memory whose authority falls below the minimum is flagged WARN.
  • `client-visible` is excluded from decisions/refinements/company because those scopes contain internal protocol/strategy material that should never reach a client export.
  • The rules file is reloadable: edit it and rerun `cyberos verify`.
