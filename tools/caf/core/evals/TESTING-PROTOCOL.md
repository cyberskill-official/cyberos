# Testing protocol — output-accuracy evaluation for field runs

Promoted from the 2026-06-10 deployment-readiness review. This is the
operating procedure for evaluating the framework's accuracy on **external
repositories**; the fixture suite (`validate.py --all`) covers the validator
itself. Feedback from every tier lands in a `feedback@1` record
(`core/schemas/feedback.v1.json`, generated by `--emit-feedback`) — see
"Calibration pipeline" below.

## The four tiers

**T0 — Pre-deploy gate (automated, exists).** Full fixture suite, registry
drift, docs sync, baseline sha256, snapshot match, report-schema validation —
all green in CI before any tag is used in the field. *Hard gate.*

**T1 — Per-run mechanical validation (automated, exists).** Every field run
ends with `code-audit-validate --run . --report json` in the target's CI (the
GitHub Action). Record exit code, `violations_by_code`, waived count. First
engagement on any new repo: `MODE: gated` for at least the first two loops.
*Threshold: violations = 0, or 100% waived-with-valid-expiry, before client
handoff.*

**T2 — Human rubric scoring (manual, exists).** Every field run gets the
10-item `core/improve/RETROSPECTIVE.md` rubric (~10 minutes). The total is the
run's **labeled example** toward the ~50-example escalation bar
(`core/improve/CRITIC.md`). Items 3 and 9 (padding, change value) are the judgment
half no validator sees. *Threshold: ≥ 16/20; any item scored 0 → FAILURE_LOG
row the same day.*

**T3 — Evidence-authenticity sample (manual until the re-execution sampler
ships).** Per sampled run: re-execute **K = 3** randomly chosen MEASURED
Verify commands in the target environment; diff against the pasted output
(tolerate timing noise, not values). Any mismatch is a fabrication event →
FAILURE_LOG + immediate engagement review. *Cadence: every first-run on a new
repo, then a random 20% of subsequent runs. Metric: fabrication rate,
target 0.*

**T4 — Cross-model consistency audit (manual, once per batch).** Same repo,
same CONFIG, two different agent CLIs; compare backlogs for finding overlap,
severity agreement, and DEPTH interpretation. This doubles as the live-run
evidence gate for the two queued protocol candidates (DEPTH semantics, R2
offline clause). *Metric: severity agreement on shared findings.*

## Accuracy metrics (report per batch)

| Metric | Definition | Target |
|---|---|---|
| `mechanical_precision` | 1 − (human-adjudicated validator false positives ÷ total violations) | ≥ 0.95 |
| `known_class_recall` | violations a human spot-review found that the validator should have caught | 0 (each one becomes a B-fixture) |
| `fabrication_rate` | T3 mismatches ÷ T3 samples | 0 |
| `rubric_median` | median T2 score, trended per protocol version (`retro-summary.py`, incl. `--feedback-dir`) | ≥ 18 |
| `waiver_ratio` | waived ÷ (waived + active), per repo via `--aggregate` | alarm > 0.5 |

## Calibration pipeline (after each run / batch)

| Feedback field | Consumer | Action rule |
|---|---|---|
| `validator_false_positives` | `core/evals/fixtures` (G-*) | 1st occurrence → G-fixture; 2nd of a family → recalibrate the check (infra commit) |
| `validator_misses` | FAILURE_LOG + fixtures (B-*) | Always a FAILURE_LOG row; harness-fixable → trap fixture immediately; protocol-wording cause → Rule of Three via CRITIC |
| `denylist_gaps` | `audit-profile.yaml` per target, promoted to built-ins when seen across clients | batched per-batch commit; additive tripwires need no evidence bar |
| `retro_items` < 2 | FAILURE_LOG | 2 retros with the same low item → CRITIC cycle candidate |
| `retro_score` | `retro-summary.py --feedback-dir` | per-version field trend — "did vX.Y.Z help on real clients?" |
| `fabrication_check` | engagement governance | any mismatch: stop, review, FAILURE_LOG; recurring → build the re-execution sampler |
| `cross_model` | queued CRITIC candidates | the named evidence gate for DEPTH semantics / R2 offline |
| record count | CRITIC.md escalation path | at ~50 labeled examples + standardized project type + single metric → evaluate promptfoo → DSPy |

Cap: at most **one CRITIC cycle per batch** — attribution stays clean. Field
records live in the private field-data repo (client-coded, sanitized, R8
applies); only resulting FAILURE_LOG rows, fixtures, and pattern updates are
promoted here.
